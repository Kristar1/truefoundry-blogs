This comparison is framed around LLM security: who can call which models, where authorization is enforced, how provider credentials are isolated, and which compliance controls are native versus operator-dependent. Zero-trust remains an architectural lens here, but the primary practical question is simpler: which gateway reduces real LLM security risks in production?


§0 — TL;DR & Quick-Picks

🔐 Real-World LLM Security Risk: A Stale Contractor Credential

During a routine controls review, the platform team discovers that a contractor credential from a completed project is still active in the LLM gateway. The key was meant to be removed at offboarding, but it can still send traffic into the shared model estate. Over the next few hours, that credential starts hitting a higher-cost production model at meaningful volume, pushing unnecessary spend, increasing the risk of provider throttling, and creating an avoidable audit issue for the team that now owns the environment.

The technical problem is bigger than one forgotten key. The gateway is treating that caller as too broadly trusted. There is no clean model-level boundary to keep lower-trust identities away from production endpoints, no strong workspace isolation to keep one team’s credential from reaching the wrong control surface, and no subject-aware policy that tightens logging, guardrails, or rate controls for higher-risk callers.

That is the business question behind this comparison. LLM security is not only about prompt injection or PII redaction after the fact. It is about ensuring that the right caller can reach the right model, in the right workspace, under the right budget, audit, and policy controls—before a routine access mistake turns into a finance, compliance, or customer-data incident.

§1 — Why LLM Security Is Specifically Hard for AI Gateways

Before comparing vendors, it helps to establish a common vocabulary for where LLM security decisions are enforced. This blog uses three terms consistently:

  • PEP (Policy Enforcement Point): the component that intercepts traffic and enforces an allow/reject decision — e.g., the gateway data plane. Its location (SaaS edge vs. customer VPC vs. in-cluster pod) determines data residency and network perimeter boundaries.
  • PDP (Policy Decision Point): where the authorization logic is evaluated. If the PDP makes a remote call (to a SaaS control plane or external IdP) on every request, that call is on the hot path and introduces a latency and availability dependency.
  • Content firewall: deterministic or ML-based checks on prompt/response payloads — PII masking, injection detection, secrets scanning. This is distinct from AuthN/AuthZ: it fires after identity is established, not instead of it.

Standard API security assumes authentication, authorization, and content filtering are separable. AI gateways make all three harder:

  • The model endpoint is not a static resource. A request to /v1/chat/completions can route to different models depending on runtime policy. URL-level ACLs are insufficient as a standalone control — identity-aware routing policy must be enforced inside the PEP.
  • Provider credentials are high-value lateral movement targets. The gateway holds API keys for every LLM provider. A compromised gateway process or config store grants access to billing-enabled upstream endpoints. Vault integration and secret group isolation are not optional hardening.
  • Prompt payloads are data exfiltration vectors. Callers can craft prompts to extract training data, reveal system prompts, or probe internal tool schemas. The content firewall must compose with identity controls — not replace them.
  • Agent and MCP traffic introduces non-human identity surfaces. Autonomous agents calling tools on behalf of users require per-tool authorization checks at the PEP, not just per-request authentication.
  • PEP location determines the shared responsibility line. A SaaS-edge PEP means the vendor operates the enforcement boundary. A self-hosted K8s PEP transfers that operational responsibility — and all associated hardening — to your platform team.

This compares documented guardrail surfaces, not identical internal implementations. Each vendor's content firewall architecture reflects what is described in their public docs.

LiteLLM §2 — Python-Native Identity Stack: Virtual Keys to K8s Workload Auth

Authentication Architecture

LiteLLM's PEP is the self-hosted Python proxy container. The PDP for authentication is local: bearer token validation checks Redis or in-memory cache before falling back to the database, as documented in the "Life of a Request" page at proxy/architecture. No external identity service call is on the routing hot path for pre-issued virtual keys. Authorization context (team, budget, model allowlist) is resolved before routing — not as a post-routing check.

OIDC / JWT Auth
Validates tokens against one or more OIDC providers via JWT_PUBLIC_KEY_URL (comma-separated JWKs URIs for multiple providers). K8s ServiceAccount tokens authenticate workloads via namespace-to-team-alias mapping using team_alias_jwt_field. Tier: verify against your deployed version — documentation tier may vary.
proxy/token_auth
OAuth 2.0
Token introspection-style auth configured via environment variables. Returns user ID, role, and team fields from the token info endpoint, populating the full gateway identity context including budget and route access.
proxy/oauth2 — Enterprise only
Custom Auth Hook
Python async function returning a UserAPIKeyAuth object. Documented fields: user, team, org, budget. Supports "auto" mode: custom auth fires first, falls back to built-in key validation. Correct integration point for LDAP or proprietary identity stores.
proxy/custom_auth — Free tier
Master Key
A master_key in general_settings gates the Admin UI and management CLI. Sub-keys cannot exceed master key permissions. Database-backed deployment required for key persistence and spend state.
proxy/virtual_keys — Free tier
⚠ mTLS — Operator-Dependent (not native to LiteLLM proxy)

The LiteLLM proxy public documentation contains no mTLS configuration. Mutual TLS is operator-dependent: it must be configured at the ingress controller, external load balancer (NGINX, AWS ALB), or service mesh (PeerAuthentication in Istio) in front of the Python container. This is a standard pattern but places mTLS ownership entirely outside LiteLLM's own configuration surface.

Authorization & RBAC

The UserAPIKeyAuth object carries a team_id, and keys with a team assignment consume from the team budget rather than the personal budget (proxy/users). Model access scope is attached via a models allowlist per key. Router behavior is configurable per key or team at keys_teams_router_settings. Applied to the opening scenario: issuing a contractor key with an explicit models allowlist, a team_id with a daily budget ceiling, and no access to production model group names closes the access gap at the PEP without additional infrastructure.

Secrets & Provider Credential Isolation

Provider API keys use the os.environ/VARIABLE_NAME pattern in config.yaml (proxy/config_settings). There is no native Vault integration documented in the public LiteLLM proxy docs — credential injection is operator-dependent (K8s secret manager, Vault Agent sidecar, or CI-injected environment variables). Credential rotation requires a pod restart under this model.

📌 Shared Responsibility — LiteLLM

PEP location: self-hosted container (your infrastructure). PDP: local Redis/memory + DB for key resolution; no remote call on hot path for pre-issued keys. You own TLS termination, network isolation, secret rotation, database hardening, and Redis ACLs in their entirety. (Full placement breakdown: see Control Placement table, §6.5.)

Kong §3 — Plugin-First Auth Stack: mTLS, OIDC, and Consumer-Anchored Policy

Authentication Architecture

Kong's PEP is the data plane proxy (always customer-operated regardless of whether the Konnect control plane is SaaS-managed). Security behaviors are Gateway plugins attached to Services, Routes, Consumers, or Consumer Groups — every authentication method Kong supports for non-AI traffic is immediately available to AI traffic. The PDP for each plugin runs inside the Lua/C data plane process; no external identity call is on the hot path for key-auth or mTLS validation. The AI audit log reference confirms Consumer identity as the anchor for all AI usage attribution, with a consumer label on AI metrics from Kong 3.11 onward.

Key Authentication
The key-auth plugin validates API keys mapped to Kong Consumers. The ai-rate-limiting-advanced plugin uses Consumer identity as the primary attribution unit for token and request quotas across LLM providers.
key-auth — OSS + Enterprise
JWT / OIDC
The jwt plugin (OSS) validates JWT signatures against configured secrets — PDP is local. The openid-connect plugin (Enterprise) supports full OIDC flows including token introspection (PDP involves remote IdP call) and automatic Consumer mapping from JWT claims.
jwt: OSS · openid-connect: Enterprise
mTLS (Mutual TLS)
The mtls-auth plugin (Enterprise) validates client certificates against a CA store and maps certificate subjects to Kong Consumers. PDP is local — certificate validation happens inside the data plane process. Strongest zero-trust option for service-to-service AI traffic where PKI is already operational.
mtls-auth — Enterprise · PDP local
Vault Secret References
Kong’s Vault entity docs explicitly list AI Proxy and AI Proxy Advanced auth fields as referenceable secret fields. A value such as {vault://aws/openai-key} is resolved through Kong Vaults, so the raw provider key does not need to live in declarative config, logs, or the UI.
Vaults entity + referenceable AI Proxy / AI Proxy Advanced fields · Enterprise-capable secret backends

Consumer Groups & RBAC at the AI Layer

Consumer Groups allow the same Route to serve multiple groups with different token quotas and content firewall policies simultaneously. The ai-rate-limiting-advanced plugin's config shape — llm_providers[].name, limit (array), window_size (array) — is documented in the handbook example. The ai-prompt-guard plugin's deny_patterns field includes best-practice regex patterns for Unicode bidirectional injection, documented in the plugin's best-practices section.

Portkey §4 — SaaS-Managed PEP with Hybrid VPC Data Plane Option

Authentication Architecture

In Portkey's default (hosted edge) deployment, both the PEP and PDP are vendor-operated — Portkey's SaaS infrastructure intercepts and enforces policy on every request. Authentication is API-key based: callers pass a Portkey API key via x-portkey-api-key or the SDK apiKey constructor. A default config can be attached at key issuance so authorization policy (routing, guardrails, rate limits) is encoded in the key's associated config object, documented at ai-gateway/configs.

The hybrid deployment (enterprise/hybrid2) moves the data plane PEP into your VPC via Helm, so prompt and response payloads do not transit Portkey's infrastructure. However, per the public deployment guide, management and configuration interfaces remain in Portkey's cloud — the PDP for policy updates is vendor-operated. This is a customer-operated PEP with vendor-operated management plane, not a fully offline control plane. Teams with strict data sovereignty requirements should confirm the exact architecture boundaries with Portkey for their plan tier.

API Key Auth + Default Config
The validated Portkey handbook documents API-key authentication at the gateway and the ability to attach a default config to a key. That makes routing, retries, guardrails, and other policy surfaces key-bound by default rather than re-specified on every request.
JWT Gateway Authentication
Clients can authenticate API requests using a JWT token validated against a configured JWKS (JSON Web Key Set). JWT authentication is documented as an enterprise add-on and provides temporary, expiring tokens, fine-grained permission scopes, user identity tracking, and centralized authentication management.
SSO (OIDC & SAML 2.0) — Control Plane
Portkey Control Plane supports OIDC and SAML 2.0 for enterprise SSO. This governs who can access the Portkey management interface, not how gateway requests are authenticated. The docs also state that first-time SSO users are auto-provisioned into the organization.
Provider Credential Vaulting & Secret References
Enterprise deployments can use Secret References: the control plane stores only the reference configuration, while the data plane fetches the actual secret at runtime from AWS Secrets Manager, Azure Key Vault, or HashiCorp Vault. The docs state that the data plane caches the fetched secret value for 5 minutes.
Workspace Isolation
Workspaces are a primary policy boundary in the validated docs pack. The public admin SDK examples show workspace-level usage limits and request-per-minute ceilings, which is the most concrete isolation surface the supplied source pack validates.
Usage & Rate Policies
Enterprise policy docs add group-by dimensions and workspace-oriented governance. In practice, that gives Portkey a centralized policy plane even when the request path remains vendor-operated in the default hosted model.
Guardrail Hooks + BYO Webhooks
Security controls can be attached through raw guardrail JSON or an external webhook contract. That makes Portkey's security story less about identity federation in the validated pack and more about config-bound request enforcement.
Hybrid VPC Data Plane
Enterprise hybrid deployment moves the data-plane PEP into your VPC via Helm. The validated handbook says prompts and responses stay in your network in this mode while metrics and management continue to connect to Portkey's cloud.

ℹ️ JWT Auth vs. SSO — Two Distinct Identity Surfaces

JWT Gateway Auth authenticates individual API gateway requests using a JWKS-validated JWT. OIDC/SAML SSO governs who can log in to the Portkey control-plane dashboard. Secret References are a separate credential-custody control, where the control plane stores only a reference and the data plane fetches the secret at runtime.

📌 Shared Responsibility — Portkey

Hosted Edge: vendor-operated PEP and PDP. Hybrid VPC: enforcement occurs at the customer-operated data plane (Helm pods in your network); management and policy configuration remain vendor-operated SaaS. Prompt and response payloads are documented to stay within your VPC in the hybrid model. Confirm exact runtime PDP boundaries for your plan with Portkey. TrueFoundry §5 — K8s-Native Split-Plane: Workspace RBAC, In-Cluster PEP, Local Policy Evaluation

Authentication Architecture

TrueFoundry's PEP is the stateless Hono-based gateway plane running in your Kubernetes cluster. The PDP is local: the gateway plane architecture page documents that the gateway performs in-memory checks for authentication, authorization, rate limits, and load balancing on the hot path, with no external dependency in the routing hot path for policy evaluation. Optional caching can introduce upstream calls, but that exception is explicitly called out separately in the docs. The platform architecture page also identifies the global authentication/licensing service and analytics service as distinct from the request path.

Gateway requests authenticate with a bearer token in the Authorization header. TrueFoundry now documents that surface directly in the AI Gateway authentication page, alongside PAT vs. VAT usage, while gateway access-control and collaboration pages cover the separate runtime and platform permission layers.

Subject-Scoped Policy: Guardrails and Metadata-Based Rate Limiting

Guardrail rules support a subjects condition inside when clauses — making content firewall policies identity-aware. Combined with first-match rule evaluation (documented in guardrails configuration), a platform team can enforce different PII masking and injection-detection policies per caller group in a single YAML config with no separate physical endpoints. This is documented at guardrails-configuration.

The x-tfy-metadata request header (documented as stringified JSON of string key-value pairs in the headers page) carries caller context into the gateway policy engine. TrueFoundry's public AI Gateway Rate Limiting docs explicitly document rate_limit_applies_per as supporting metadata.* key dimensions, with metadata.project_id provided as a named example in both the migration guide and full rule examples at ai-gateway/ratelimiting. The supported values ['model'] and ['metadata.project_id'] used in the snippet below are taken verbatim from those public docs. Server-Timing response headers expose per-phase execution breakdown (authentication, guardrails, model call, logging) — documented in the headers page — enabling precise latency attribution for policy components.

§7 — Editorial Verdict

The following assessments represent editorial judgment based on documented capabilities. "Well-suited for" is the authors' characterization, not a vendor claim.

TrueFoundry
Strongest Boundary Clarity
Best fit for teams that want the request path, policy path, and data boundary to stay inside customer infrastructure. The docs are unusually explicit about what is and is not on the hot path.
Well-suited for: Regulated teams, K8s-native platforms, and agentic workloads that need clear residency and governance boundaries.
LiteLLM
Fastest Identity Wiring
Strong choice when you want virtual keys, OIDC/JWT, and workload-identity-style patterns quickly without adopting a heavier gateway stack.
Well-suited for: Teams needing fast multi-identity rollout with budget-aware isolation.
Kong AI Gateway
Broadest Native Gateway Surface
Still the most complete option for enterprises that specifically value native gateway plugins, especially documented mTLS and mature policy composition.
Well-suited for: Enterprises with Kong already in place or teams that require native gateway-layer mTLS.
Portkey
Turnkey Managed Governance
Compelling when teams want workspace controls, spend governance, and a managed gateway experience, with hybrid deployment as the step-up option.
Well-suited for: Product teams comfortable with a managed policy plane and focused on fast rollout.

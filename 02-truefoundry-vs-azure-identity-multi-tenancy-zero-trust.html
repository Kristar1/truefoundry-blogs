What this blog argues

Identity is not just who can call the AI. It is the structural property that determines what happens when a credential is stolen, when a service principal is misconfigured, or when an internal actor exceeds their intended scope. The headline difference between the two products is that TrueFoundry workspaces are physical Kubernetes namespace boundaries while Azure tenancy is a logical RBAC composition across Entra, APIM workspaces, and Azure resource groups. Different blast-radius properties under failure and breach. Section 4 of this blog walks four concrete scenarios where the difference is operationally decisive.

§0 — TL;DR & Quick-Picks

A breach that becomes an architecture question

At 03:14 on a Saturday, a security analyst at a healthcare insurer notices unusual activity in the audit log: a service principal belonging to the claims-triage AI team is making API calls outside its expected scope. The principal was assigned API Management Service Contributor at the resource-group level four months ago — at the time, the security review team flagged that the scope was wider than necessary, but the alternative (nine custom roles across nine resource groups) was operationally untenable for a small platform team.

Now, with the principal's credentials likely compromised through a leaked CI variable, the on-call team has to answer one question: what is the maximum set of changes this credential could have made before we noticed? Walking the Azure RBAC graph: the principal can edit policies on every API in every workspace in that resource group. It can rotate Key Vault secrets that APIM references through managed identity. It can attach new backends, including ones pointing outside the tenant. It can enable diagnostic-setting changes. And because RBAC is additive and permissions take up to an hour to propagate, even after the credential is revoked the team still has to wait through the propagation window before they can be confident the revocation has reached every cached token issuer.

The fix takes 36 hours of cleanup. The architecture lesson lands later: had each AI team's runtime been isolated at a Kubernetes namespace boundary instead of being a shared resource group with overlapping APIM service-level RBAC, the same compromise would have been bounded to one team's namespace. That is the comparison this blog is about.

TL;DR
  • Azure identity is a multi-service composition. Entra issues tokens; APIM validates them via validate-azure-ad-token; managed identity authenticates to backends; Key Vault holds secrets referenced by managed identity; RBAC at service and workspace scopes is enforced additively across Entra, ARM, and the APIM data plane. Each piece is well-engineered. The seams are documented but real.
  • TrueFoundry identity is one model with physical isolation. Workspaces are Kubernetes namespaces. The gateway pod caches the SSO issuer's public keys; the user-to-model authorization map is held in-memory and refreshed via NATS. Workspace-scoped secret groups and virtual accounts handle service-to-service auth.
  • The blast-radius question is the difference that matters. A logical RBAC composition is harder to bound than a physical namespace boundary when the credential or scope is wrong. Both architectures can be operated safely; the operational discipline cost is not the same.
Quick-Pick
Azure
The right call when the security organization has Entra centrality as a non-negotiable, when managed identity to Azure AI services is core to the audit story, and when the platform team is large enough to operate the dual-scope RBAC composition (service + workspace) with discipline.
Quick-Pick
TrueFoundry
The right call when blast-radius bounding is decisive — regulated data, multi-team AI platform, customer-controlled clusters — and the security review prefers physical boundaries to logical RBAC composition. Especially strong when workspace = K8s namespace simplifies the breach response runbook.
Editorial frame
Why blast-radius is the lens
Identity comparisons that compare auth mechanisms feature-by-feature miss the operational point. The right question is what the worst-case incident looks like under each model. Section 4 walks four breach scenarios in concrete detail.

§1 — Why identity is Movement I's second axis

Identity sits in Movement I (Foundations) for the same reason architecture does: it determines what every later blog is allowed to assume. Routing decisions in Blog 4 assume an authentication model. FinOps attribution in Blog 6 assumes a tenancy model. MCP tool governance in Blog 11 assumes a service-identity model. If the identity layer is uncertain, the rest of the comparison reads as feature lists rather than as a coherent platform.

The most common framing of an identity comparison is to walk auth mechanisms — JWT validation, mTLS, API keys, managed identity — and tally checkmarks. That framing produces an answer like "both products support most modern auth standards," which is true but misses the operationally consequential point. The question that survives a security review is not which mechanisms are supported. It is what is the maximum damage a compromised credential can do, and how easily can the response be bounded?

That question has a structural answer for both products, and the answers are different. Azure's answer is a logical composition: Entra controls who can request a token, APIM controls what that token can do at the gateway tier, ARM controls what the token's bearer can do as a control-plane operator, and Key Vault controls what backend secrets the runtime can access. Each layer is well-engineered, and Microsoft has invested heavily in making them compose cleanly through documented patterns. The cost of the composition is that the maximum-damage answer requires walking the graph at incident time — across Entra logs, ARM activity logs, APIM diagnostic logs, Key Vault access logs, and any custom application telemetry.

TrueFoundry's answer is a physical boundary. Workspaces are Kubernetes namespaces. A compromised credential at the workspace level cannot, by construction, modify resources in another namespace. A compromised platform-admin credential is a different and serious problem (and addressed in §3), but the routine breach scenarios that drive most security incident response runbooks have a structural upper bound at the namespace edge.

The honest boundary on this argument

Physical isolation does not eliminate breach risk. A K8s namespace boundary protects against most credential-scope mistakes but is not a substitute for cluster-level security hygiene, image-scanning policies, secret-rotation discipline, or network segmentation. The argument is that workspace-level breaches are easier to bound and reason about in a namespace-based model than in a multi-service RBAC composition — not that the namespace model is inherently more secure. Both products can be operated to a high security bar; the runbook complexity is what differs.

The rest of the blog does three things. §2 walks the Azure identity composition in working detail, including the dual-scope RBAC requirement and the four documented service- and workspace-level role types, with a working APIM XML example. §3 walks the TrueFoundry model — the in-memory authorization map, namespace-as-workspace, virtual accounts, and secret groups — with a working YAML example. §4 is the heart of the blog: four concrete blast-radius scenarios shown side by side. §5 ends with the editorial verdict.

Microsoft§2 — Azure: a multi-service identity composition

Azure identity for AI workloads composes several services. The composition is documented well and operates predictably when set up carefully — but it is a composition, and the structural fact of the composition is what produces the seam tax that this blog measures.

The four layers of Azure identity for AI

For an AI request flowing through APIM into Azure OpenAI (or any Azure AI service), four identity layers are relevant:

  1. Caller identity (Entra ID). The application or user obtains an Entra-issued JWT. APIM validates it through the validate-azure-ad-token policy with tenant-id, client-application-ids, audiences, and optional required-claims.
  2. Backend identity (managed identity). APIM authenticates outbound to Azure OpenAI or Azure AI services using the authentication-managed-identity policy. The managed identity is either system-assigned (the APIM service's own identity) or user-assigned (a separately created identity that may be reused across services).
  3. Secret identity (Key Vault). Named values, certificates, and any other gateway-managed secrets are stored in Azure Key Vault and referenced by APIM through that same managed identity. This is a clean pattern — secrets never appear in policy XML — but it means the managed identity must be granted access to Key Vault separately, and that grant lives in Key Vault's RBAC, not APIM's.
  4. Operator identity (ARM RBAC). Platform engineers who edit APIM configurations, register backends, or rotate policies do so through Azure RBAC at service, workspace, or sub-resource scopes. Microsoft's RBAC overview for APIM documents three built-in service-level roles plus four workspace-level roles.

The dual-scope rule

The dimension that matters most for blast-radius reasoning is the role-composition rule. From Microsoft Learn: "A workspace collaborator must be assigned both a workspace-scoped role and a service-scoped role." The three service-level roles are:

  • API Management Service Contributor — full management of the APIM service.
  • API Management Service Reader — read access to APIM resources, including subscription keys.
  • API Management Service Operator — restart and similar operational actions.

The workspace-level roles include:

  • API Management Workspace Contributor — manages the workspace; can view but not modify members.
  • API Management Workspace Reader — read-only at workspace scope.
  • API Management Service Workspace API Developer — read access plus read/write on API-editing entities at workspace scope.
  • API Management Service Workspace API Product Manager — read access plus read/write on API-publishing entities.

And custom roles can be created when none of the built-ins fit. Each role lists ARM-level Microsoft.ApiManagement/... actions that are permitted; the additive evaluation model means a user with both Reader at one scope and Contributor at a wider scope effectively has Contributor.

Why the dual-scope rule produces operational complexity

If you want a developer to edit APIs in only the AI workspace, you cannot just assign Workspace API Developer. You must also assign a service-level role — typically API Management Service Reader — at the APIM service scope, which gives that developer read access to all APIs in the entire APIM instance, including ones outside their workspace. That is not a bug; it is a documented requirement. But it means the smallest blast radius an AI workspace developer can structurally have is "their workspace's writes plus all of APIM's reads." For platform teams that share an APIM instance across multiple programs, this is the seam to watch.

The Azure identity composition diagram

Azure identity composes Entra (caller), managed identity (backend), Key Vault (secret), and ARM RBAC (operator) across five services. Each layer is well-engineered; the composition is what the customer maintains.

The full APIM inbound chain a serious AI deployment writes

The XML below is the canonical shape for the inbound chain on an AI API in APIM. It composes validate-azure-ad-token (Layer 1), set-backend-service with managed identity (Layers 2–3), and the operator-side concerns (audit, diagnostics) that the platform team writes once and reuses:

XML
<!-- Illustrative APIM inbound chain showing all four identity layers -->
<policies>
  <inbound>
    <base />
    <!-- LAYER 1: caller identity from Entra -->
    <validate-azure-ad-token tenant-id="{{aad-tenant-id}}"
                             header-name="Authorization"
                             failed-validation-httpcode="401">
      <client-application-ids>
        <application-id>{{ai-coding-assistant-app-id}}</application-id>
        <application-id>{{ai-claims-triage-app-id}}</application-id>
      </client-application-ids>
      <audiences>
        <audience>api://contoso-ai-gateway</audience>
      </audiences>
      <required-claims>
        <!-- Caller must have a role claim including ai-user -->
        <claim name="roles" match="any">
          <value>ai-user</value>
          <value>ai-admin</value>
        </claim>
      </required-claims>
    </validate-azure-ad-token>

    <!-- Capture the calling app id for downstream attribution -->
    <set-variable name="callerAppId"
                  value="@(((Jwt)context.Variables[&quot;jwt&quot;]).Claims[&quot;appid&quot;].FirstOrDefault())" />

    <!-- LAYER 2-3: backend selection + managed identity to AOAI -->
    <set-backend-service backend-id="aoai-claims-primary" />
    <authentication-managed-identity resource="https://cognitiveservices.azure.com/"
                                     output-token-variable-name="managed-identity-token" />
  </inbound>
  <backend><forward-request /></backend>
  <outbound>
    <base />
    <!-- LAYER 4: operator-visible audit dimension -->
    <llm-emit-token-metric namespace="Contoso/AI">
      <dimension name="caller-app" value="@((string)context.Variables[&quot;callerAppId&quot;])" />
      <dimension name="workspace" value="claims-triage" />
    </llm-emit-token-metric>
  </outbound>
</policies>
Illustrative composition based on Microsoft Learn for validate-azure-ad-token, authentication-managed-identity, and llm-emit-token-metric. The required-claims + set-variable pattern is the standard idiom for downstream attribution. Workspace gateways do not currently support authentication-managed-identity per the March 2025 breaking changes announcement.

Read this XML as a series of identity transitions. The inbound JWT proves Layer 1. The set-backend-service + authentication-managed-identity pair pins Layer 2 to the gateway's managed identity. The named values {{aad-tenant-id}}, {{ai-coding-assistant-app-id}}, and {{ai-claims-triage-app-id}} resolve through Layer 3 (Key Vault). And the policy itself is editable only by holders of the appropriate Layer 4 ARM RBAC roles. The four-layer composition is concrete here, not abstract.

A specific operational caveat

Microsoft's RBAC overview documents that role-assignment changes can take "up to one hour" to propagate to cached tokens across the stack. In a routine breach response — where a credential is revoked and the team needs to be confident the revocation has reached every cached identity issuer — that propagation window is part of the runbook. It is not a security flaw, but it is a fact about the architecture that incident-response time-to-clean is bounded below by token-cache TTLs across multiple services.

TrueFoundry§3 — TrueFoundry: namespace-physical tenancy with one identity model

TrueFoundry's identity model is consolidated. The platform key-concepts page and the gateway-plane-architecture page together describe a system where every identity question — caller, backend, secret, operator — resolves through one control plane and lands at one physical boundary: the workspace, which is a Kubernetes namespace.

Workspace = Kubernetes namespace

This is the most consequential single fact in the comparison. A TrueFoundry workspace is not a logical scope composed across Entra and ARM RBAC and APIM workspace roles. It is a Kubernetes namespace. Every deployment, every async-service, every secret group, every model-serving pod, every gateway-routed request that names that workspace lands inside that namespace. The K8s API server enforces the namespace boundary at every operation. There is no "service-scoped role" that incidentally grants read access across all workspaces; namespace boundaries are enforced by the cluster.

This produces a clean breach-response runbook. If a credential scoped to a workspace is compromised, the maximum damage is bounded by what that namespace contains. There is no walk across nine logs to determine the maximum-damage answer; the answer is the contents of one namespace.

The hot-path identity flow inside the gateway pod

Inside the gateway plane, identity is in-memory. The relevant facts from the gateway-plane-architecture documentation:

  1. The gateway pod fetches the SSO issuer's public keys when it learns the issuer URL from the control plane, and caches them in-memory.
  2. JWT verification happens against those cached keys with no external network call on the request path.
  3. The user-to-model authorization map (which users can call which virtual models, with which budget caps and rate limits) is held in-memory inside each gateway pod.
  4. Authorization-map updates are propagated via NATS — the control plane publishes a delta, every gateway pod consumes it asynchronously and refreshes its in-memory view.
  5. If NATS is unreachable, gateway pods continue serving with the last-known-good authorization map. The HTTP fallback to the control plane backend resyncs when NATS recovers.

For the operator, this is operationally identical to "fast and consistent JWT validation with deterministic latency." For the security reviewer, the consequential property is that the gateway has no external identity-service dependency on the request path. There is no Entra discovery cache to age out, no managed-identity token to refresh, no Key Vault round-trip if the secret rotated overnight.

Service-to-service identity: virtual accounts and secret groups

The two primitives that handle non-user identity are virtual accounts and secret groups:

  • Virtual accounts are TrueFoundry-issued credentials that represent a service or external system. They are scoped to one or more workspaces, can be limited to specific virtual models, and are revocable from the control plane. The pattern is comparable to an Entra service principal in scope and usage, but the credential lives in the TrueFoundry control plane and the scope it grants is workspace-scoped at the K8s namespace level.
  • Secret groups are workspace-scoped collections of secrets (provider keys, third-party credentials) that can be referenced by the services and gateway routes inside that workspace. Secret groups can be backed by external secret stores — Azure Key Vault, AWS Secrets Manager, GCP Secret Manager — through the platform's secret-integration documentation, but the consumption surface is one referenceable identifier inside one workspace.

The TrueFoundry identity diagram

TrueFoundry identity: SSO and authorization map flow from one control plane through NATS to in-memory gateway state. Workspaces are Kubernetes namespaces, with the boundary enforced by the K8s API server.

What the operator writes

Workspace + virtual account + secret group declarations are the unit. The shape below is illustrative and consistent with the TrueFoundry access-control documentation:

YAML
# Illustrative workspace + virtual account + secret group bundle
type: workspace
name: claims-triage
namespace: tfy-claims      # the K8s namespace this workspace owns
description: Claims-triage AI team workspace

# Membership: maps SSO identities (Entra, Okta, Auth0…) to roles
members:
  - subject: group:claims-ai-team@contoso.com
    role: workspace-admin
  - subject: user:alice.stewart@contoso.com
    role: workspace-developer
  - subject: user:bob.lee@contoso.com
    role: workspace-viewer

# Virtual account: revocable credential for a CI service or external caller
virtual_accounts:
  - name: claims-ci-deployer
    role: workspace-developer
    allowed_models: [claims-triage-prod, claims-triage-staging]
    rate_limit: { requests_per_minute: 100 }

# Secret group: workspace-scoped credentials, optionally backed by an external store
secret_groups:
  - name: provider-keys
    secrets:
      - name: openai-api-key
        source: { kind: azure-key-vault, vault: prod-vault, key: openai-key }
      - name: anthropic-api-key
        source: { kind: literal, value_ref: env:ANTHROPIC_API_KEY }
Illustrative shape aligned to TrueFoundry docs for key concepts (workspaces and namespaces), access control, and secret management. The single declaration produces (1) a K8s namespace, (2) RBAC bindings inside that namespace, (3) a virtual-account credential redeemable at the gateway, and (4) a secret group whose secrets are referenceable from any deployment in the workspace.

Read this YAML as one unit: it produces the namespace, the RBAC inside it, the credentials that can be used from outside the platform, and the secrets that flow into the runtime — all from one declaration applied through one IaC entry point. There is no separate Entra app registration, no separate Key Vault grant on a managed identity, no separate APIM workspace-scoped role assignment. The composition is the platform's responsibility, not the customer's.

§4 — Blast-radius under breach: four concrete scenarios

The most useful way to compare these identity models is to walk concrete breach scenarios and identify the worst-case damage, the response runbook, and the time-to-clean. The four scenarios below are realistic patterns drawn from common incident-response postmortems. Each is shown side-by-side under both products. None of the scenarios is biased toward either product by construction; the goal is to show what each architecture's worst case actually is.

SCENARIO 1Compromised app-team service principal / virtual account

A service principal (Azure) or virtual account (TrueFoundry) belonging to the AI-coding-assistant team is compromised through a leaked CI variable. The attacker has whatever scope the principal was granted, plus whatever propagation window is required to revoke it.

Azure — worst case

If the principal had API Management Service Contributor at resource-group scope (a common pattern when fine-grained role design is operationally untenable), the attacker can edit policies on every API in every workspace in that RG, attach new backends, modify Key Vault references, and change diagnostic settings. The dual-scope rule means that even a workspace-only role couples to a service-level role granting at-minimum read across the entire APIM instance. Time-to-clean: revoke principal credentials in Entra (immediate), wait for ARM token cache propagation (up to ~1 hour per Microsoft RBAC docs), audit policy changes across multiple workspaces.

TrueFoundry — worst case

If the virtual account had workspace-developer in the claims-triage workspace, the attacker can deploy services, modify routes, and consume budgets only inside the tfy-claims Kubernetes namespace. The K8s API server enforces the boundary; cross-workspace operations fail at the API server, not at a logical RBAC check. Time-to-clean: revoke virtual account in the control plane (effective at next NATS sync to gateway pods, seconds to ~1 minute); audit changes in one namespace.

SCENARIO 2Stolen developer JWT in transit

A developer's JWT is stolen through a network capture (the developer was on an unsecured network and triggered a request from a desktop tool). The token has the developer's normal scope and a TTL of one hour.

Azure — worst case

The token is valid for its remaining TTL across every Azure resource the developer can access — APIM data plane, ARM control plane, Key Vault, Azure Monitor. Time-to-clean: invalidate the developer's session in Entra (continuous-access-evaluation can speed this up if configured), revoke the access token. Without CAE, Azure RBAC's documented propagation window means the token may continue to authenticate for up to its TTL on cached endpoints. The attacker can use the token across the full breadth of the developer's role assignments while it remains valid.

TrueFoundry — worst case

The token authenticates against gateway pods that hold the SSO issuer's public keys in-memory. Revoking the developer in the control plane invalidates the user-to-model authorization map at next NATS sync; gateway pods deny new requests for that user immediately. The attacker can use the token for the remaining TTL only against the gateway data path — control-plane operations require the control-plane authorization layer, which honors the revocation immediately. Time-to-clean: revoke in control plane, wait one NATS sync (seconds), confirm via gateway logs.

SCENARIO 3Authorized but rogue insider with workspace-level write

A platform-engineering team member is leaving the organization on bad terms. They have legitimate write access to one workspace (Azure: APIM workspace + service-level read; TrueFoundry: workspace-admin role). They want to maximize damage during their notice period.

Azure — worst case

Within their workspace, they can edit any policy, attach any backend, modify any product, and rotate any policy fragment. The dual-scope rule's service-level read means they can also enumerate all APIs and configurations across the entire APIM instance, which they can save and exfiltrate. They cannot directly modify other workspaces but they have substantial reconnaissance value across the whole instance. Mitigation: rely on Azure Monitor alerts on policy changes, JIT access patterns, and approval workflows.

TrueFoundry — worst case

Within their workspace (one K8s namespace), they have admin rights — they can modify deployments, change routes, edit secrets in the workspace's secret group, exfiltrate workspace-local data. They cannot enumerate or read configurations from other workspaces; the namespace boundary prevents it. The blast radius is structurally bounded to one namespace. Mitigation: deployment-validation policies (Blog 12) plus the standard K8s-level controls (audit logs, OPA/Cedar policies on apply-time changes).

SCENARIO 4Failed identity rotation: provider key leaked

A provider API key (OpenAI, Anthropic) is accidentally committed to a public repository and detected hours later. The key needs to be rotated immediately, and every running service that uses it needs to pick up the new key with minimal downtime and zero exposure window of the old key continuing to work.

Azure — worst case

Update the named value in Key Vault. APIM's named-value cache must refresh — the documented refresh interval is several minutes. During the cache window, requests through APIM continue using the old (compromised) key. Time-to-clean: provider-side key revocation (immediate at provider), APIM named-value refresh (minutes), workspace gateway impact: workspace gateways do not currently support Key Vault named values via managed identity, so AI APIs in workspaces cannot use the same secret-rotation pattern and rely on a different mechanism with its own propagation timing.

TrueFoundry — worst case

Update the secret in the workspace's secret group (or in the upstream Key Vault if backed by one). The control plane publishes the change via NATS; gateway pods refresh in-memory secrets at next sync. The provider-side revocation is the bottleneck for clean exposure-window closure, not the platform's propagation model. Time-to-clean: provider-side revocation + control-plane secret update + one NATS sync (seconds).

§5 — Editorial verdict

Best fit
Azure
The composition is well-engineered, mature, and deeply integrated with the rest of the Microsoft ecosystem. validate-azure-ad-token with cached discovery is a credible JWT path; managed identity to Azure AI services is best-in-class for removing provider keys from the runtime; Key Vault-backed named values are an institutional secret-management pattern. The seam tax is real but the right organization can amortize it well.
Choose Azure if: Entra centrality is non-negotiable, your security org thinks in Azure RBAC + ARM scopes natively, your platform team has the operational discipline to maintain the dual-scope role composition, and your AI workloads benefit from the managed-identity-to-Azure-AI-services audit story.
Best fit
TrueFoundry
The physical-tenancy model dramatically simplifies the breach-response runbook and the security-review conversation. Workspace = K8s namespace is a property the security team can reason about with one mental model. NATS-driven revocation propagation eliminates the multi-service propagation window that drives time-to-clean in Azure incidents. Virtual accounts and secret groups produce one-declaration identity scopes.
Choose TrueFoundry if: blast-radius bounding is decisive (regulated data, customer-controlled clusters, multi-team AI platform), your security review prefers physical boundaries to logical RBAC composition, or the operational simplicity of one identity model — rather than four cooperating ones — is itself the value.

The most honest framing is that the two architectures optimize for different security postures. Azure optimizes for a security organization that already operates well across Entra, ARM RBAC, and Azure-native audit; the AI workload inherits that discipline. TrueFoundry optimizes for a security organization that wants one model with a physical boundary and is willing to accept Kubernetes as the substrate that enforces it. Neither model is universally better; the question is which one matches your incident-response runbook.

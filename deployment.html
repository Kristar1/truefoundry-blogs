§0 — TL;DR & Quick-Picks

🏦 Real-World Scenario: The Data-Sovereign AI Platform Mandate

A pan-European financial institution receives a formal supervisory finding under DORA's ICT third-party risk provisions. The finding is unambiguous: their AI inference pipeline, which routes customer-facing prompts through a US-domiciled SaaS control plane, is non-compliant with cross-border data transfer restrictions and ICT concentration risk rules. The CISO issues a mandate — every component in the LLM governance path must run within the institution's own infrastructure. No SaaS control plane. No outbound licensing calls on the inference hot path. Full audit log ownership. The platform team has a 90-day window to re-architect.

The team evaluates four candidates. Portkey's standard hybrid deployment moves the data plane into the VPC, but the control dashboard and higher-level configuration features remain in Portkey's cloud — the CISO rejects that default posture immediately unless a fully private enterprise deployment is explicitly scoped. LiteLLM is fully self-hostable in Python but the team discovers that at their target 1,500 RPS peak, Redis and careful database architecture become mandatory, and the Python GIL creates a horizontal-scaling story that adds operational surface area. Kong AI Gateway in db-less self-managed mode satisfies data residency, but the AI-specific features they need — AI Proxy Advanced load balancing, AI Rate Limiting Advanced — require an enterprise license whose validation path the security team scrutinizes carefully. TrueFoundry is the only platform with a formally documented split-plane model where the control plane itself can be self-hosted on enterprise plans; the gateway data plane is stateless and evaluates routing, budgets, and auth entirely in-memory with no external calls on the inference hot path. The public docs are mixed, however: architecture pages describe global licensing/authentication and analytics services outside customer infrastructure, while installation docs also describe an air-gapped deployment flow and an external-SSO option that removes dependence on the managed auth server. For strict regulatory reviews, that nuance should be validated directly against the latest enterprise deployment guidance.

This blog dissects exactly why deployment architecture is not a Day-2 concern for AI gateways. It is often the decision that determines whether a product can be adopted at all.

§1 — Why Deployment Architecture Is a First-Class Concern

Routing strategy, fallback chains, and observability pipelines get most of the attention in AI gateway evaluations. Deployment architecture is typically deferred to a later phase — until it isn't. The moment a platform team encounters a security review, a data residency requirement, an air-gap mandate, or a large-scale production traffic target, deployment architecture becomes the dominant selection criterion.

For AI gateways specifically, the deployment question is structurally harder than it is for conventional API gateways. Several forces converge:

  • The inference hot path carries regulated content. Prompt and completion payloads frequently contain PII, PHI, customer data, or proprietary business logic. Which components sit in that path, and where those components run, is a compliance question, not just an infrastructure one.
  • Control planes know too much. A gateway's control plane manages keys, budgets, routing policy, and audit logs. If the control plane is cloud-hosted, that knowledge graph lives outside your perimeter. This matters for both regulatory compliance and competitive confidentiality.
  • Stateless vs. stateful data planes have wildly different scaling contracts. A stateful data plane — one that accumulates key-lookup state, rate-limit counters, or routing decisions locally — becomes operationally rigid: you need sticky sessions, coordinated Redis clusters, or distributed locks to scale horizontally. A stateless data plane, where policy is pushed in from outside rather than accumulated locally, enables clean K8s HPA-based scaling without those constraints.
  • Provider credentials must go somewhere. Whether stored in a local database, a vault, an environment variable, or a SaaS secret store, the location of provider API keys shapes the threat model for every request that crosses the gateway.

A single synthesis positioning map appears in §7 after the vendor-by-vendor analysis, once the deployment evidence has been laid out.

LiteLLM §2 — Python Proxy: Flexible Deployment, Runtime-Bounded Scale

Deployment Architecture

LiteLLM's gateway is a Python process — the same runtime used for the SDK — and it can be deployed in any environment that can run a Python container. The public docs document four primary paths: direct Docker container, Docker with a database backend for persistent key/spend state, Kubernetes via Helm, and Terraform-based ECS for AWS-native environments.

The deployment model creates a clear three-tier architecture. The lightest tier — a single containerized proxy with a config.yaml mounted in — handles routing, model aliasing, and provider normalization without external dependencies. The second tier adds a PostgreSQL database and master key, which unlocks persistent virtual keys, team budgets, the Admin UI, and spend tracking. The third tier layers Redis on top for state coordination at scale.

The key inflection point documented explicitly by the LiteLLM deployment guide is 1,000 requests per second: above this threshold, Redis is described as required to prevent database connection exhaustion and deadlocks. This is not a soft recommendation — it is a hard architectural constraint on the Python runtime's ability to manage concurrent state across multiple proxy replicas. The practical consequence for platform teams is that a highly available, horizontally scaled LiteLLM deployment is a three-component system (proxy + Redis + Postgres), and all three components must be designed for failure independently.

# Database-backed LiteLLM container — required for persistent key management,
# team budgets, and the Admin UI. Redis must be added separately at >1k RPS.

Kubernetes Scaling

LiteLLM supports K8s HPA, but horizontal scaling requires careful design. Because multiple proxy replicas must share the same Redis and Postgres instances to make consistent decisions on rate limits and key lookup, adding pods without coordinating that shared state layer creates inconsistent behavior. Redis becomes the coordination bus — its availability and latency directly affect whether HPA-triggered scale-out actually improves throughput or simply amplifies the pressure on the shared state layer. For lightweight deployments with a single replica, no Redis dependency exists; the complication arises specifically as the team moves toward high-availability multi-replica setups.

⚠ Limitation — Shared State Required for HA Multi-Replica Deployments

The Python GIL and the requirement for shared Redis/Postgres state at scale mean that high-availability HPA-based scale-out for LiteLLM carries operational coupling. Each new replica must connect to the shared state layer. For teams that need low-latency, cost-efficient horizontal autoscaling with no coordination bus, this is the primary constraint to plan around. Single-replica or low-traffic deployments are unaffected.

Kong §3 — Multi-Topology C/Lua Runtime: Three Ways to Deploy

Deployment Architecture

Kong AI Gateway's deployment architecture is documented at developer.konghq.com/ai-gateway through the lens of its underlying gateway primitives. The core insight is that AI features are a plugin layer on top of Kong Gateway's mature network runtime — and that runtime already offers three battle-tested topologies: traditional (DB-backed, admin API-driven), hybrid (centralized control plane with distributed data planes), and db-less (declarative config, no database dependency, YAML-driven). This topology choice is the central deployment decision for Kong AI Gateway.

The Konnect SaaS offering provides a managed control plane with a supported path for hybrid or cloud-gateway data planes. The alternative is fully self-managed Kong: all control plane and data plane components within your own infrastructure. The quickstart path uses Konnect, which the docs explicitly frame as "a quickstart for exploration, not the endpoint of a production deployment journey." For regulated environments, self-managed Kong with db-less topology is the relevant deployment model because it has no external control-plane dependency whatsoever — config is consumed from a declarative YAML file pushed via decK or the Admin API from within the perimeter.

Kubernetes & Scaling

Kong's C/Lua non-blocking runtime handles high raw concurrency per pod. In db-less mode, data plane pods are stateless with respect to routing configuration — declarative config is loaded at startup or via decK push — making K8s HPA straightforward. In traditional topology, state lives in the database, so scale-out is clean for the data plane itself. The AI Rate Limiting Advanced plugin's state coordination approach depends on the underlying topology and should be reviewed against the plugin compatibility reference before sizing decisions are made.

⚠ Limitation — AI Feature Availability Is Version- and License-Gated

DB-less topology with Kong Gateway OSS supports AI Proxy (from 3.6). AI Proxy Advanced, AI Rate Limiting Advanced, AI Semantic Cache, and AI RAG Injector all require enterprise license and higher minimum versions (3.7–3.10). The plugin compatibility reference is the authoritative source and must be consulted per topology before committing to a deployment model.

Portkey §4 — SaaS-First with a VPC Data-Plane Escape Hatch

Deployment Architecture

Portkey is documented at portkey.ai/docs/product/ai-gateway as "a fully managed SaaS Anycast edge network." This is the default and primary deployment model: clients connect to Portkey's globally distributed edge, routing, retry, fallback, and config evaluation all execute in Portkey's cloud, and requests are forwarded to upstream providers. Setup time for this path is measured in minutes, and the absence of infrastructure to manage is a genuine productivity advantage for teams without dedicated platform engineering resources.

The same Portkey docs also expose a quick local open-source gateway runtime via npx @portkey-ai/gateway. That matters for evaluation because it gives teams a concrete artifact to run before adopting the managed service or the hybrid Helm path, even though the deeper control-plane and dashboard experience remains cloud-centered in Portkey's published product story.

For organizations that cannot send prompt or completion payloads through a third-party cloud, Portkey documents a hybrid deployment path at portkey.ai/docs/enterprise/hybrid2. The data plane runs inside the customer VPC as a containerized service deployed via Helm. The declared contract is explicit: prompts and responses stay inside the network while only metrics leave. The Portkey control dashboard, config management surface, and higher-level features remain in Portkey's cloud — routing behavior is configured through that cloud control plane regardless of where the data plane runs. This is a meaningful improvement over pure SaaS for data path residency, but it is not equivalent to full self-hosting.

# Portkey VPC data-plane hybrid deployment via Helm
# Prompts and responses remain in your network — only metrics leave to Portkey cloud

Scaling

In SaaS mode, scaling is automatic and transparent — Portkey's managed Anycast edge handles traffic growth without customer intervention. In hybrid mode, the data plane pods deployed via Helm can be scaled with K8s HPA. Portkey's hybrid deployment documentation does not explicitly characterize the stateless or stateful nature of these pods, so statelessness assumptions should be validated against the latest Helm chart and deployment guidance before designing an HPA strategy around them. Routing configuration changes in hybrid mode still flow through the cloud control surface.

⚠ Caveat — Hybrid vs. Air-Gapped Deployments

In Portkey's standard hybrid deployment, the control plane (configs, dashboards, analytics) remains in the cloud while the data plane runs in your VPC. Portkey's public docs also describe air-gapped enterprise deployments where the control plane runs entirely within customer infrastructure, and where the Gateway and Backend operate without access to the hosted control plane. Teams with strict sovereignty requirements should explicitly scope and validate that fully private tier rather than assuming the standard hybrid posture.

TrueFoundry §5 — K8s-Native Split-Plane: Six Deployment Scenarios & Stateless Gateway Pods

The Split-Plane Architecture

TrueFoundry's deployment story is organized around a formally documented split-plane model: control plane (orchestrates config, auth, policy, integrations, and deployment state), compute plane (runs model workloads, jobs, and services), gateway plane (proxies LLM and MCP traffic with routing, budgets, and observability), and data plane (stores models and artifacts). The gateway plane is the operationally critical component for AI inference governance, and its architecture is documented explicitly at truefoundry.com/docs/platform/gateway-plane-architecture.

The gateway plane is built on the Hono framework and is designed with an explicit hot-path contract: no external calls during request handling (unless cache is involved). Rate limit evaluation, load balancing, authentication, and authorization all execute in-memory. Logs and metrics are queued asynchronously, and the docs state that request handling continues even if the external queue is down. This is a materially different architecture from LiteLLM's Python proxy (which needs Redis to coordinate state across replicas at scale) and Portkey's hybrid mode (where routing behavior is configured through a cloud control plane).

The gateway plane is stateless, subscribing to configuration updates via a queue with HTTP fallback. This enables cross-region multi-gateway deployments while configuration is managed centrally — and it is the architectural property that makes clean K8s HPA scaling possible.

Deployment Scenarios

The deployment-overview page enumerates six deployment scenarios — an unusually specific breakdown that forces architectural tradeoffs into the open rather than hiding them behind generic capability language:

  • Gateway-only SaaS — Managed gateway plane; minimum infrastructure overhead; fastest path to model governance.
  • Gateway-plane-only — Customer-hosted gateway pods; hosted control plane. Inference path stays within customer network; policy management in TrueFoundry's cloud.
  • Full control plane + gateway — Both control plane and gateway plane within customer infrastructure. Enterprise plan.
  • Compute plane + managed gateway — Self-hosted model serving plus managed inference proxy.
  • Compute-only — Model serving without gateway.
  • Fully self-hosted combination — All planes within customer infrastructure. Maximum control; maximum operational responsibility. Enterprise plan.
✅ Why TrueFoundry Stands Out on Deployment Architecture
  • SaaS Deployment — gateway-only SaaS for the lowest operational overhead.
  • VPC / On-Prem Deployment — gateway-plane-only and fuller self-hosted deployment paths are both documented.
  • Air-gapped Deployment — strong sovereign posture, but the public docs are mixed on whether auth/licensing and analytics are fully severed; validate the exact enterprise posture before making a hard air-gap claim.
  • Multiple Gateway Planes — docs explicitly support multiple gateways deployed cross-region with central configuration management.
  • GitOps (Infrastructure as Code) — installation model is OpenTofu / Terraform plus Helm.
ℹ️ Important Nuance — Public Docs Are Mixed on Air-Gapped Dependencies

The public docs are not fully consistent here. Architecture pages still describe a global licensing/authentication server and analytics server outside customer infrastructure, and the self-hosted control-plane install guide lists egress access to auth.truefoundry.com, login.truefoundry.com, catalogue.truefoundry.com, and analytics.truefoundry.com. At the same time, installation docs also describe an air-gapped / restricted-network deployment flow, and enterprise docs describe an external SSO option that deploys the control plane without the TrueFoundry Auth Server. For strict air-gap reviews, validate the exact enterprise deployment posture against the latest product guidance rather than relying on a single doc page.

Kubernetes HPA — How Stateless Pods Enable Workload-Specific Scaling

TrueFoundry's stateless gateway architecture has a direct and practical consequence for Kubernetes autoscaling. Because each gateway pod evaluates routing policy, rate limits, and auth entirely in-memory at request time — rather than consulting a shared Redis cluster or a remote state store — pods have no coupling to one another. This means the K8s Horizontal Pod Autoscaler can scale the gateway layer based on resource metrics or custom metrics (for example CPU, request rate, or external queue depth) without any concern for state consistency across replicas.

Different application workloads within the same platform benefit from this in distinct ways:

  • Real-time voice agents require low TTFT and can spike rapidly. Stateless pods can be pre-warmed and aggressively scaled using CPU-based HPA. When a new pod starts, it subscribes to the current config snapshot over the queue — there is no cache warm-up period for routing or policy data.
  • Batch summarization jobs can tolerate higher latency and run at predictable request rates. These can use separate Kubernetes deployments targeting lower replica counts, scaling based on queue depth via an external scaler or scheduled capacity windows. They share the same centrally managed routing and guardrail configs while operating on independent autoscaling targets.
  • Interactive API services with variable intraday traffic patterns benefit from the tight HPA loop — since pod state is exclusively in-memory and ephemeral, scale-down is clean. There is no "draining" of per-pod caches to handle.
TrueFoundry's stateless gateway pods enable workload-specific autoscaling profiles without a shared Redis coordination bus. Each pod class can scale independently while all pods consume the same centrally pushed routing and guardrail config. CPU-based HPA is straightforward; request-rate and queue-depth examples here imply custom metrics or an event-driven scaler. Illustrative parameters only.
⚠ Limitation — K8s Operational Prerequisite

The gateway-plane deployment guide requires a running Kubernetes cluster, a domain, a TLS certificate, image-pull credentials, and a control-plane API key before a single request is handled. The platform ships as a combination of OpenTofu / Terraform and Helm charts. For organizations without a dedicated K8s platform engineering function, this is a real adoption barrier. The gateway-only SaaS deployment scenario avoids this entirely.

§7 — Editorial Verdict

Editorial assessments as of March 10, 2026.

LiteLLM
The Self-Hosted Pragmatist
Lowest barrier to a fully self-hosted deployment. Docker and Helm paths are well-documented. A single-container deployment has no external state dependencies; the Redis/Postgres coupling arises specifically at HA multi-replica scale. For teams below 1k RPS or with single-region, single-replica footprints, those constraints are manageable.
Ideal for: Fast-moving platform teams, internal tooling, teams that want full self-hosting without K8s expertise and can accept Python-runtime scaling constraints at high concurrency.
Kong AI Gateway
The Network Operator's Choice
Best deployment-model breadth for teams already on Kong. DB-less topology delivers a genuinely stateless, portable data plane with no external dependencies. Multi-topology flexibility is real, but AI feature availability is tightly coupled to plugin version and license tier.
Ideal for: Existing Kong users, teams wanting a battle-hardened C/Lua runtime, organizations comfortable managing Kong's plugin-version/license matrix.
Portkey
The SaaS-Speed Operator
Unmatched time-to-production. The hybrid VPC path addresses data residency for inference, while Portkey's air-gapped enterprise deployment path removes the hosted control-plane dependency for stricter regulatory environments.
Ideal for: Product teams wanting SaaS velocity, or enterprises procuring the fully private / air-gapped tier for strict residency requirements.
TrueFoundry
The Platform Engineer's Architecture
The most architecturally coherent choice for K8s-native platform teams with data-sovereignty needs. Stateless Hono pods, a formally documented split-plane model, SaaS and VPC / on-prem deployment options, multiple gateway planes, and GitOps-friendly installation make it the strongest deployment story here. The main caveat is not the hot path — which is documented as fully in-cluster — but that the public docs are currently mixed on how auth/licensing and analytics behave in the strictest air-gapped posture.
Ideal for: Regulated or enterprise platform teams that want SaaS-to-self-hosted choice, clean K8s scale-out, multi-region gateway operations, and IaC-driven deployment workflows.

What this blog argues

Most AI-platform discussions about caching pick one cache. They compare semantic-cache implementations, or they compare exact-match key-value caches, or they discuss provider prompt caching. But a production AI request can hit all three on the same hop, and reasoning about cost, latency, and correctness requires a model that includes all of them. This blog argues that the right comparison frames are (1) three caches in one request path — gateway exact, gateway semantic, provider prompt — with different cost-impact and correctness profiles; (2) the threshold convention difference between APIM (cosine distance, lower = stricter) and TrueFoundry (cosine similarity, higher = stricter); and (3) the configuration locus — APIM puts cache strategy in policy XML, TrueFoundry puts it in a per-request header — which determines who owns threshold tuning and how visible cache decisions are at debug time. Both products are well-engineered; the operating-model fit is what differs.

§0 — TL;DR & Quick-Picks

A cache hit that becomes a correctness question

An e-commerce search team enables semantic caching on their product-recommendation endpoint to reduce token spend on a long-context RAG workload. They've read the docs, picked a similarity threshold that sounded reasonable, set a TTL of 10 minutes, and shipped the change on a Thursday. Cache-hit rates land at 31% within an hour — better than the team's hoped-for 25% — and the finance lead celebrates the projected $14K/month token savings on the team's Slack.

Three days later, a customer-success ticket arrives that the team almost dismisses. A user has filed a screenshot: they searched "noise-cancelling headphones for sleeping" and the recommendation card returned three pairs of noise-cancelling headphones for travel. The product titles are correct; the visual styling is correct; the recommendations are simply for the wrong use case. The user's request is reasonable — they want sleeping-optimized recommendations, not generic noise-cancelling — and what they got back is a previous recommendation generated for a different user who searched something similar. The cache returned a hit; the hit was on a prompt with high embedding similarity but materially different intent.

The platform engineer on call walks the diagnostic. The two prompts — "noise-cancelling headphones for sleeping" and "noise-cancelling headphones for travel" — have a cosine similarity of around 0.93 on the embeddings backend the team is using. The configured threshold, set to what the docs called a "balanced default," would have surfaced the cache hit. The architectural lesson lands hard at the standup the next morning: semantic similarity in the embedding space is not the same as task equivalence. Two prompts that look semantically close can produce very different correct answers when the use case differs subtly. The fix takes ninety minutes — tighten the threshold and add a workspace dimension to the cache namespace so that different product categories don't share entries. The lesson takes longer.

This blog is not about whether semantic caching is good or bad. It's a useful tool when calibrated for the workload. The architectural point is that the threshold tuning, the namespace partitioning, and the observability of why a cache hit happened are the three operational properties that determine whether semantic caching improves the system or introduces correctness regressions. Microsoft's own documentation acknowledges this directly: "semantic caching returns responses based on similarity (not exact match), it can surface responses that are incorrect, outdated, or unsafe for the current request. Evaluate this feature carefully for your workload and include safeguards." That warning sets the bar for any comparison framework: caching is not a free win — it is a calibrated trade-off, and the platform's job is to make the calibration tractable.

TL;DR
  • Three caches, not one. An AI request can hit (1) the gateway's exact-match cache, (2) the gateway's semantic cache, and (3) the provider's own prompt cache (OpenAI implicit, Anthropic explicit via cache_control). All three exist whether you reason about them or not. For high-volume long-prompt workloads, provider caching shapes the cost curve more than either gateway cache.
  • Azure: two policy pairs with explicit infrastructure. azure-openai-semantic-cache-lookup/store for AOAI; llm-semantic-cache-lookup/store for other LLM APIs. Both require an embeddings-backend (typically AOAI text-embedding) and an external cache (Azure Managed Redis). The threshold is a distance ceiling — lower = stricter; Microsoft's docs warn "above 0.2 may lead to cache mismatch." Configuration locus is policy XML, per API.

§1 — Three caches in one request path

Most AI-platform discussions about caching pick one cache and compare implementations of it. The framing is wrong because a production AI request can hit three caches on the same hop, and reasoning about cost, latency, and correctness requires a model that includes all of them. The three caches are different kinds of objects with different control surfaces, and only when all three are accounted for does the cost picture make sense.

The three caches

  1. Gateway exact-match cache. A byte-for-byte (or normalized-byte-for-normalized-byte) hash of the request producing a cache key. If the same prompt is sent twice, the second one returns the first one's response. Both products implement this; it is operationally trivial and correctness-safe — the only correctness risk is staleness, which the TTL bounds.
  2. Gateway semantic cache. An embedding of the prompt is computed; a similarity search finds the nearest neighbor in cached entries; if the similarity exceeds (or, in APIM's distance model, falls below) the configured threshold, the cached response is returned. Both products implement this; the configuration mechanics differ; the correctness risk is real and threshold-dependent.
  3. Provider prompt cache. The model provider's own caching layer, sitting underneath whatever the gateway does. OpenAI's is implicit (long shared prefixes are cached automatically when prompts exceed 1024 tokens); Anthropic's is explicit (the application opts in by adding cache_control: {"type": "ephemeral"} to message blocks). Neither gateway can change how provider caching works; both pass through the headers and request structure that providers need to recognize cache opportunities.

Why all three matter for the cost discussion

For a workload that sends short, varied prompts (an interactive search assistant, say), the gateway caches do most of the work. Exact-match catches verbatim repeats; semantic caches catches reformulated repeats; provider caching does little because prompts are too short to benefit. For a workload that sends long, structured prompts with shared prefixes (a RAG application that injects 4K of retrieved context plus a 1K system prompt into every call), the picture inverts: provider prompt caching shapes the cost curve more than either gateway cache. A RAG workload paying ~80% of its input-token bill on the prefix can knock that down to single-digit percent through OpenAI's automatic prefix caching or Anthropic's explicit cache_control — and gateway caching only helps for the small fraction of prompts that are sufficiently similar to existing entries.

The implication for the comparison framework: "which gateway has better caching" is the wrong question to ask in isolation. The right questions are: (a) does the gateway's semantic cache work well for the workload's prompt-similarity distribution? (b) does the gateway preserve the structure providers need to recognize their own cache opportunities? (c) does the platform make threshold tuning and cache-decision debugging tractable enough for the team to operate semantic caching safely? Both products answer (a) and (b) competently; the difference shows up in (c), which is what the rest of this blog walks.

A safety warning that is not Claude's, but Microsoft's

Microsoft's azure-openai-semantic-cache-lookup policy reference includes this warning verbatim: "Because semantic caching returns responses based on similarity (not exact match), it can surface responses that are incorrect, outdated, or unsafe for the current request. Evaluate this feature carefully for your workload and include safeguards." The e-commerce scenario at the top of this blog is exactly the kind of incident this warning anticipates. Both products carry the same fundamental risk because the risk is intrinsic to semantic caching itself, not to either implementation. The platform's job is not to eliminate the risk but to make it tractable: tunable thresholds, partitioned namespaces, and per-decision observability are the three operational properties that determine whether the team can manage the trade-off.

The rest of the blog walks both architectures in working detail. §2 walks the Azure semantic-cache surface — two policy pairs (AOAI-specific and LLM-general), the embeddings-backend pattern with managed identity, the Azure Managed Redis dependency, the threshold-as-distance convention — with a working policy composition. §3 walks TrueFoundry's per-request x-tfy-cache-config header model and the response-header observability surface — with a working request and a debugging walkthrough. §4 zooms out to the third cache: provider prompt caching from OpenAI and Anthropic, what each gateway does to preserve it, and what neither can change. §5 walks three concrete caching scenarios side by side. §6 ends with the editorial verdict.

Microsoft§2 — Azure: llm-semantic-cache-lookup with embeddings backend + Azure Managed Redis

APIM's semantic-cache surface is, like its routing surface, a composition of well-engineered primitives. The architectural choice that shapes the operating model is that cache strategy lives in policy XML and depends on first-class Azure infrastructure: a configured embeddings-backend resource and an Azure Managed Redis instance. Both are real Azure resources that the customer provisions and operates; both are referenced from policy. Setting up semantic caching is not a single-flag toggle — it is a multi-resource composition that, once configured, is mature and well-integrated.

Layer 1 — Two policy pairs for two API surfaces

APIM exposes two separate semantic-cache policy pairs for two different API surfaces:

  • azure-openai-semantic-cache-lookup + azure-openai-semantic-cache-store — the AOAI-specific pair, used when the API in APIM is an Azure OpenAI in Foundry models endpoint. Optimized for AOAI Chat Completion APIs and AOAI's specific request/response shapes.
  • llm-semantic-cache-lookup + llm-semantic-cache-store — the general pair, used for LLM APIs that come in through the Azure AI Model Inference API or for OpenAI-compatible models served through third-party inference providers.

The two pairs share the same configuration shape — score-threshold, embeddings-backend-id, embeddings-backend-auth, ignore-system-messages, max-message-count, vary-by — and the same fundamental semantics. The split exists because the AOAI pair is tuned for AOAI's specific request structure while the LLM pair is more general. Operationally this means a workload that mixes AOAI and non-AOAI providers writes two policy compositions; the team has to pick the right pair per API.

Layer 2 — The embeddings-backend pattern with managed identity

Semantic caching requires computing an embedding for each incoming prompt. APIM does not embed; it delegates to a configured backend. The "Enable semantic caching for LLM APIs" tutorial documents the pattern: create a backend resource of type Custom URL pointing at an embeddings model deployment (typically AOAI text-embedding-3-small or similar), configure managed-identity auth on that backend, then reference the backend by ID from the cache-lookup policy.

The recommended auth pattern is system-assigned managed identity with the Cognitive Services OpenAI User role on the embeddings AOAI resource. The policy's embeddings-backend-auth="system-assigned" attribute hooks into APIM's managed-identity flow (per the Layer 2 identity model from Blog 2). This is one of the cleaner identity compositions in the Azure stack — when it works, it works well.

Layer 3 — Azure Managed Redis as the cache backend

The cache itself is external. Microsoft's Azure Managed Redis tutorial documents the canonical pattern: provision an Azure Managed Redis instance with the RediSearch module enabled, configure APIM to use it as an external cache, ensure the connection auth (typically access key) is in place. The semantic-cache policies use Redis with vector similarity search to find nearest-neighbor cached entries. This means semantic caching adds two infrastructure dependencies the team must operate: the embeddings backend and the Redis cluster.

The threshold convention — a footgun, documented

The score-threshold attribute is cosine distance, not similarity. Lower values are stricter (closer to identical); higher values are looser. Microsoft's docs explicitly say "above 0.2 may lead to cache mismatch" and recommend starting at 0.05 and tuning from there. This is the inverse of TrueFoundry's convention (similarity floor — higher = stricter), and it is the inverse of how most engineers intuitively think about similarity scores. A threshold of 0.5 in APIM is loose; a threshold of 0.5 in TrueFoundry is loose-to-moderate; the numbers are not directly comparable. Teams switching between the two products, or evaluating both, need to keep this conversion in mind explicitly.

The Azure semantic-caching topology diagram

Azure semantic caching composes three first-class Azure resources — APIM policy, AOAI embeddings backend with managed identity, and Azure Managed Redis with RediSearch — coordinated through policy XML. Cache decisions are observable through Azure Monitor.

What the operator writes — semantic caching for an AOAI Chat Completion API

The XML below is the canonical AOAI semantic-cache shape, with the rate-limit guard Microsoft's documentation specifically recommends adding immediately after the cache lookup. The vary-by partitions cache entries by subscription so cross-tenant cache poisoning is structurally prevented:

XML
<policies>
  <inbound>
    <base />

    <!-- Lookup with subscription-partitioned cache entries -->
    <azure-openai-semantic-cache-lookup
        score-threshold="0.05"
        embeddings-backend-id="embeddings-backend"
        embeddings-backend-auth="system-assigned"
        ignore-system-messages="true"
        max-message-count="10">
      <vary-by>@(context.Subscription.Id)</vary-by>
      <vary-by>@(context.Request.Headers.GetValueOrDefault(&quot;X-Tenant&quot;,&quot;&quot;))</vary-by>
    </azure-openai-semantic-cache-lookup>

    <!-- Microsoft's recommended pattern: rate-limit immediately after the
         cache lookup, to prevent overload if the cache is unavailable -->
    <rate-limit calls="10" renewal-period="60" />

    <!-- Standard backend selection + managed identity to AOAI -->
    <set-backend-service backend-id="aoai-primary" />
    <authentication-managed-identity resource="https://cognitiveservices.azure.com/" />
  </inbound>

  <outbound>
    <!-- Store the response keyed by the prompt embedding for future hits -->
    <azure-openai-semantic-cache-store duration="600" />
    <base />
  </outbound>
</policies>
Composition based on Microsoft Learn for azure-openai-semantic-cache-lookup, azure-openai-semantic-cache-store, and the enable-semantic-caching tutorial. The ignore-system-messages="true" attribute excludes system messages from the embedding computation — useful when system messages are templated and should not affect the cache key. max-message-count="10" bounds how many recent messages contribute to the embedding.

What APIM is genuinely strong at — and what the operating model costs

What APIM's semantic-cache surface does well is real and worth naming explicitly:

  • Mature managed-identity integration for the embeddings backend. The system-assigned auth flow is the same one APIM uses everywhere else — institutional muscle for any team operating APIM seriously.
  • The ignore-system-messages attribute is a thoughtful piece of engineering. Most production AI workloads include long system prompts that are nominally identical across requests but shouldn't dominate the embedding similarity; excluding them is exactly right.
  • max-message-count bounds the embedding computation cost on long conversation histories — sensible for chat workloads where the conversation grows over time.
  • Composability with the rest of the policy stack. Cache lookup, rate-limit, backend routing, managed-identity auth, and emit-token-metric all compose in one policy XML document. For teams already fluent in APIM, this is excellent leverage.
  • Azure Monitor integration for cache-hit metrics. Observability flows through standard APIM diagnostic settings.
The honest caveats

(1) Two infrastructure dependencies the team operates: the embeddings AOAI deployment (which has its own capacity, quota, and cost) and Azure Managed Redis (which has its own SKU, scaling, and operational burden). Both are first-class Azure resources, which is good, but neither is part of APIM itself. (2) The threshold convention is unintuitive — distance not similarity, and the docs themselves warn that 0.2 is too loose. Teams new to semantic caching typically over-permit hits on first deployment. (3) No first-party response-header observability. The similarity score that triggered a hit is not surfaced in a response header; debugging false-positive hits requires correlation through Application Insights traces. The e-commerce scenario at the top of this blog would have been faster to diagnose with a similarity-score response header.

TrueFoundry§3 — TrueFoundry: per-request cache control with response-header observability

TrueFoundry's cache surface is shaped by a different operating-model question. Where APIM puts cache strategy in policy XML — owned by the platform team, set per API, applied uniformly to all callers of that API — TrueFoundry puts it in a per-request header sent by the application. The platform-team-versus-application-team boundary is drawn differently, and the consequences flow through to threshold tuning, namespace partitioning, and cache-decision observability.

Layer 1 — The x-tfy-cache-config request header

TrueFoundry's caching documentation defines the cache contract as a request header rather than a platform-side configuration. The application sends x-tfy-cache-config on each call, with a JSON value specifying the cache strategy for that call. The header has four primary fields:

  • type — exact-match or semantic. Determines which of the two gateway-side caches participates.
  • similarity_threshold — for semantic caching, the cosine similarity floor. Higher = stricter, the inverse of APIM's distance-ceiling convention. Typical values are 0.92–0.97 for production usage.
  • ttl — time-to-live in seconds. Bounds how long a cache entry remains valid.
  • namespace — partition key. Different namespaces have isolated cache pools; the right pattern is to name the namespace after the workload (e.g., product-search, customer-support-classifier) so unrelated calls don't share entries.

The architectural implication is that the application opts in per call. Different endpoints can use different cache strategies — a high-stakes credit-decisioning endpoint might disable caching entirely, while a low-stakes product-recommendation endpoint enables semantic caching with a tight threshold and a workload-specific namespace. The platform team isn't choosing the strategy on behalf of every caller; the application that knows the use case picks the strategy that fits.

Layer 2 — Response-header observability

This is the part of the design that produces operational value disproportionate to its size. The gateway returns three observability headers on every cache-touching response:

  • x-tfy-cache-status — hit, miss, or bypass. Tells the caller (and the logs) whether the cache participated in this response and what it decided.
  • x-tfy-cached-trace-id — when the status is hit, this is the trace ID of the original request whose response is being replayed. Critical for debugging: "why does this response include X?" becomes answerable by following the trace ID to the original generation context.
  • x-tfy-cache-similarity-score — when the status is hit and the type is semantic, the actual similarity score that triggered the hit. Makes threshold tuning data-driven rather than guess-and-check.

The e-commerce scenario at the top of this blog would have been diagnosed in minutes rather than hours with this surface. The CS ticket includes a request ID; the platform engineer pulls the request from logs; the response carries x-tfy-cache-status: hit, x-tfy-cached-trace-id: trace-7b2f9a, x-tfy-cache-similarity-score: 0.93; the engineer pulls trace-7b2f9a, sees the original prompt was about travel, immediately understands the false-positive hit. The threshold-tuning conversation that follows is data-driven from the start.

Layer 3 — Cache as the documented exception to the no-external-calls rule

The gateway-plane-architecture page states this explicitly: in the gateway pod's hot path, rate-limit checks, auth verification, and load-balance decisions are all in-memory; cache is the one operation that legitimately leaves the pod. This is by design — semantic caching requires an embedding lookup and a similarity search over potentially millions of entries, neither of which fits in a pod's memory footprint at production scale. The platform manages the cache backend; the customer doesn't provision a separate Redis cluster.

The architectural implication is operational simplicity: one fewer infrastructure dependency for the customer to operate. The trade-off is less direct control — the cache backend is a platform-managed component rather than a customer-provisioned Redis instance the team can tune. For most workloads this is the right trade; for workloads that need specific Redis tuning, the trade-off goes the other way.

The TrueFoundry semantic-caching topology diagram

TrueFoundry semantic caching — application opts in via x-tfy-cache-config; gateway computes embedding, queries platform-managed cache, and returns observability headers on every response. The similarity score is exposed for data-driven threshold tuning.

What the operator writes — request, response, and debugging walkthrough

The shape below shows a request with semantic caching enabled, the response headers the gateway returns, and how the response surface answers the operational question that the e-commerce scenario raised:

Bash
# Application sends request with cache config opting in to semantic caching
curl -X POST https://gateway.contoso.com/api/llm/api/inference/openai/chat/completions \
  -H "Authorization: Bearer ${TFY_TOKEN}" \
  -H "Content-Type: application/json" \
  -H 'x-tfy-cache-config: {"type":"semantic","similarity_threshold":0.95,"ttl":600,"namespace":"product-search"}' \
  -d '{
    "model": "product-recommendation-prod",
    "messages": [
      {"role":"system","content":"You are a product recommendation assistant..."},
      {"role":"user","content":"noise-cancelling headphones for sleeping"}
    ]
  }'

# Response headers when the cache returns a hit:
#
#   HTTP/1.1 200 OK
#   x-tfy-cache-status: hit
#   x-tfy-cached-trace-id: trace-7b2f9a4e-8c11-4b3d
#   x-tfy-cache-similarity-score: 0.93
#   ...
#
# At debug time:
#   1. Customer reports "wrong recommendations" — gives request ID
#   2. Engineer pulls request from logs, sees x-tfy-cache-status: hit + score 0.93
#   3. Engineer pulls trace-7b2f9a4e (the original request whose response was cached)
#   4. Original prompt was "noise-cancelling headphones for travel"
#   5. Diagnosis: false-positive hit; threshold 0.95 was too loose for this workload
#   6. Fix: tighten threshold to 0.97, add use-case dimension to namespace
#
# Total debug time: ~10 minutes from CS ticket to identified root cause.
Illustrative request/response pair consistent with TrueFoundry caching docs. The response-header surface makes false-positive hits diagnosable from logs alone — no separate observability correlation required. The namespace field is the partitioning primitive that prevents cross-workload cache poisoning.

Compare this to the equivalent debugging on Azure: the platform engineer pulls the request from Application Insights, queries Log Analytics for the cache-hit metric, correlates the request ID to find the cache decision, then has to look elsewhere to understand which prior request the cached response came from — because the similarity score and the trace-back are not in the response itself. Both products can produce the same diagnostic answer; the operational distance is what differs.

What this earns operationally

(1) Per-call control. Different endpoints can use different cache strategies without a policy redeploy on each. The credit-decisioning endpoint disables caching; the product-recommendation endpoint enables it with a tight threshold; the FAQ endpoint enables it with a loose threshold and long TTL. (2) Response-header observability. The x-tfy-cache-similarity-score on every hit makes threshold tuning data-driven; the x-tfy-cached-trace-id makes false-positive hits diagnosable from logs alone. (3) One fewer infrastructure dependency. No customer-provisioned Redis, no customer-provisioned embeddings backend; the platform manages both. (4) Namespace partitioning is structural. The namespace field in the cache config is a first-class primitive — workloads with different correctness profiles get isolated cache pools by construction.

§4 — The third cache: provider-side prompt caching

Both OpenAI and Anthropic operate prompt caches at the provider level that sit underneath whatever the gateway does. These caches are invisible to most platform-comparison discussions, but for high-volume long-prompt workloads they shape the cost picture more than either gateway-side cache. A correct comparison framework has to account for them — and the framework has to be honest that neither gateway can change how provider caching works. What both can do is preserve the request structure providers need to recognize their own cache opportunities.

OpenAI prompt caching — implicit, prefix-based

OpenAI's prompt caching is automatic. Requests with prompts of 1024 tokens or more that share a long common prefix (system prompts, few-shot examples, retrieved RAG context that's the same across calls) get cache-hit pricing on the prefix automatically. Cached input tokens are billed at a fraction of the standard input rate. No application configuration is required; the prefix matching happens server-side based on the actual content of the request.

The cost implication for a typical RAG workload is dramatic. Consider a customer-support agent that:

  • Includes a 2,000-token system prompt with persona, constraints, and tool descriptions.
  • Includes 1,500 tokens of retrieved knowledge-base context relevant to the user's question.
  • Includes 500 tokens of conversation history.
  • Adds a 200-token user message.

That's 4,200 input tokens per call. The system prompt and many of the few-shot examples are identical across calls; the retrieved context varies but often shares prefixes when users ask similar questions. With OpenAI's prompt caching active, a meaningful fraction of those 4,200 tokens hits cache pricing — knocking 60-80% off the input-token cost for that workload at no operational effort. Neither gateway adds anything to make this happen; OpenAI does it server-side based on content matching.

Anthropic prompt caching — explicit, with cache_control

Anthropic's prompt caching is explicit. The application opts in by adding cache_control: {"type": "ephemeral"} to specific message blocks (typically system messages, retrieved context, or tool definitions). Cached blocks are billed at approximately 10% of the standard input rate on cache hits and approximately 125% on the first write. TTL is 5 minutes by default with optional 1-hour extended cache.

The application-side request structure determines whether caching kicks in, which means: both gateways must preserve the cache_control field on their way through. Stripping it (in a misguided "normalization" step) would silently disable Anthropic prompt caching. Both products document that they pass through provider-specific request fields unchanged on the documented hot path.

What the gateways do — and what they cannot

For the cost discussion, the operational facts are:

  • Both gateways pass through provider-side cache headers and structure unchanged. OpenAI's prefix caching works regardless of whether the request flowed through APIM, TrueFoundry, or directly. Anthropic's cache_control field is preserved by both.
  • Neither gateway can make provider caching work better. The cache is the provider's; the gateway has no influence on its hit rate beyond preserving the structure that determines hits.
  • Gateway exact-match caching can short-circuit provider caching. When the gateway exact-cache returns a hit, the request never reaches the provider, so provider caching does not contribute to the cost. For workloads where the gateway exact-match hit rate is very high, this is fine; for workloads where it's low, the provider cache is what carries the cost story.
  • Gateway semantic caching changes the prompt. A semantic cache hit returns a cached response without forwarding to the provider. So semantic caching, when it hits, is "free" of provider costs — but the provider cache stops contributing on those calls. The cost calculus combines both.
The composite cost picture

For a workload that:

  1. Has long shared prefixes (RAG, agent system prompts, few-shot example sets) — provider caching is the dominant cost reducer; gateway caching is incremental.
  2. Has short, varied prompts (interactive search, classification) — gateway caching does most of the work; provider caching contributes little.
  3. Has a mix — provider caching reduces input-token cost on the prefix; gateway caching reduces total request count for repeats.

A correct cost model accounts for all three caches because their effects compose differently for different workload shapes. "Which gateway has better caching" is the wrong unit of comparison; "which caching strategy fits this workload, and how do the platforms support it" is the right one.

§5 — Three caching scenarios compared

The three scenarios below walk realistic patterns where caching strategy decides the cost or correctness outcome. Each is shown side by side under both products.

SCENARIO 1Product-recommendation false-positive — debugging from a CS ticket

The e-commerce scenario from the opening. A customer reports wrong recommendations from a semantic-cached endpoint. The platform team needs to identify that the cause is a false-positive cache hit, find the original prompt that the cached response was generated for, and tune the threshold appropriately.

Azure path

Pull the request from Application Insights with the customer's request ID. Query Log Analytics for cache-hit telemetry on that request. Correlate the cache-hit event to find which entry was returned. Look up the original request that produced that entry — typically through additional KQL on the request that originally populated the cache (no direct response-header link). Identify that the original prompt was different. Tune the threshold by adjusting the policy XML's score-threshold value, deploy the policy update. Time-to-diagnose: 1-3 hours depending on telemetry pipeline maturity. Time-to-fix: minutes once diagnosed; deploy is policy XML update.

TrueFoundry path

Pull the request from logs. Read x-tfy-cache-status: hit, x-tfy-cached-trace-id: trace-7b2f9a, x-tfy-cache-similarity-score: 0.93 directly from the response. Pull trace-7b2f9a from logs to see the original prompt. Identify the false-positive immediately. Tighten threshold to 0.97 in the application's cache-config header (which can be deployed independently per endpoint). Time-to-diagnose: ~10 minutes. Time-to-fix: minutes; deploy can be application-side per endpoint.

SCENARIO 2RAG workload with shared system prompt — provider caching dominates

A customer-support agent uses a 2K-token system prompt + 1.5K retrieved KB context per request, calling Anthropic Claude. The team wants to maximize cost savings without changing the user-visible behavior.

Azure path

The big lever is Anthropic prompt caching via cache_control: {"type": "ephemeral"} on the system message and the retrieved-context blocks. APIM passes this through unchanged on the request path; the AI gateway's role here is to not interfere. Adding gateway exact-match caching catches verbatim repeats but for an agent with varied user inputs the hit rate is low. Adding gateway semantic caching can help for clusters of similar questions but requires careful threshold tuning to avoid false positives. Outcome: provider caching does most of the work; gateway caches are incremental.

TrueFoundry path

Same fundamental answer. Application uses Anthropic cache_control; the gateway preserves it on the request path. The application can additionally enable gateway exact-match caching with a long TTL via x-tfy-cache-config: {"type": "exact-match", "ttl": 3600, "namespace": "support-agent"} for verbatim repeats. Semantic caching is optional and tunable per endpoint. The response-header observability surface helps tune any gateway-cache decision. Outcome: equivalent provider caching, slightly easier per-endpoint gateway-cache tuning.

SCENARIO 3Multi-tenant SaaS — different customers must not share cache entries

A B2B SaaS company runs an AI feature across hundreds of customer tenants. The team wants semantic caching for cost savings but absolutely cannot have one tenant's cached response served to another tenant — both for correctness reasons and for data-isolation compliance.

Azure path

The vary-by element in the cache-lookup policy is the partitioning primitive. <vary-by>@(context.Request.Headers.GetValueOrDefault("X-Tenant",""))</vary-by> isolates entries per tenant ID. The implementation is correct as long as every request carries the tenant header and the policy correctly extracts it. Verification at audit time requires walking the policy XML and confirming the vary-by elements are correct on every cache-using API. Effective; verification is policy-by-policy.

TrueFoundry path

The namespace field in x-tfy-cache-config is the partitioning primitive. The application can include the tenant ID in the namespace name (e.g., "namespace": "support-agent:tenant-a-12345") to isolate entries per tenant. Combined with workspace-scoped tenancy from Blog 2 (workspaces are K8s namespaces with physical isolation), the data-isolation story is structurally clean. Effective; verification is checking that the application includes the tenant ID in the namespace, which is application-side.

§6 — Editorial verdict

Best fit
Azure
A mature semantic-cache implementation with clean managed-identity integration to AOAI text-embedding deployments, the institutional Azure Monitor observability story, and thoughtful attributes like ignore-system-messages and max-message-count that show genuine engineering attention to AI workloads. The two-policy-pair split (AOAI-specific + general LLM) gives precise tuning for each surface. The vary-by through policy expressions is the standard pattern teams already operating APIM know how to apply. For organizations whose cache discipline already includes Azure Managed Redis as institutional infrastructure, the composition is excellent.
Choose Azure if: the team already operates Azure Managed Redis, the policy-XML configuration locus matches the team's caching-discipline practice, Azure Monitor is the observability target, and the threshold-as-distance convention is part of how the team thinks about similarity scores.
Best fit
TrueFoundry
Per-request cache control through the x-tfy-cache-config header lets different endpoints use different strategies without policy redeploys — a credit-decisioning endpoint can disable caching while a product-recommendation endpoint enables it tightly, all from application-side decisions. The response-header observability surface (x-tfy-cache-status, x-tfy-cached-trace-id, x-tfy-cache-similarity-score) makes false-positive hits debuggable from logs alone. The platform-managed cache backend removes two customer-operated infrastructure dependencies. For AI engineering teams that need fast iteration on cache strategy, the cumulative reduction in operational distance is the value the design delivers.
Choose TrueFoundry if: per-call cache control matters (different endpoints with different strategies), response-header observability is valuable for debugging cache decisions, or the operational simplicity of one platform-managed cache backend is preferred over composing Redis + embeddings backend manually.

The most honest framing is that both products implement the same fundamental technique — semantic caching with cosine-similarity matching against an embedding store — and the architectural difference is where the configuration sits and how visible the cache decision is at debug time. Azure puts strategy in policy XML owned by the platform team and integrates with the broader Azure observability stack; TrueFoundry puts strategy in a per-request header owned by the application and surfaces decisions in response headers. Both approaches are coherent; both produce equivalent cost outcomes on workloads where the strategy fits.

The deciding question is whether the team thinks of caching as a platform-team decision applied uniformly per API, or as an application-team decision varied per endpoint. The former matches APIM's model; the latter matches TrueFoundry's. For AI workloads where different endpoints have very different correctness profiles — credit decisioning vs FAQ vs product search vs internal-tool classification — the per-endpoint flexibility TrueFoundry's design provides is a meaningful operational simplification. For workloads where caching is uniform per service surface, APIM's model is equally fit.

And underneath both: provider-side prompt caching shapes the cost picture more than either gateway-side cache for high-volume long-prompt workloads. Both gateways pass through the structure providers need; neither can change how provider caching itself works. A correct cost model accounts for all three caches because their effects compose differently for different workload shapes.

The next blog (Token Governance & FinOps Attribution) walks the third hot-path dimension. Token enforcement and per-team attribution determine whether cost-spike response is hours or minutes — and the architectural choice that makes attribution fast is what decides which platform fits which FinOps shape.

LLM Budget Enforcement at the Gateway Layer:
LiteLLM vs. Kong vs. Portkey vs. TrueFoundry

⏱ 22 min read 🗓 March 2026 👤 Platform Engineering · FinOps
⚠ EDITORIAL DISCLAIMER — Assessments reflect the authors' opinions as of March 2026. Feature claims are cross-referenced against official public documentation. Verify before production decisions.

§0 — TL;DR & Quick-Picks

Incident Pattern: Policy Exists, but the Business Still Feels the Failure

A platform engineering team rolled out an internal coding agent through a shared gateway, with team-scoped credentials, budget alerts, and a monthly governance review already in place.

After a deployment change introduced an unintended retry loop, the agent began generating a large burst of long-context requests over an extended window. By the time on-call triaged the incident, spend had materially exceeded the intended cap. The gateway had logged usage accurately; it had not stopped the flood early enough.

The post-incident finding was architectural rather than administrative: spend was reconciled after responses completed, so concurrent in-flight requests continued to launch against stale budget state. That distinction between synchronous pre-request blocking and post-response accounting is the core technical question in this comparison.

Quick-Pick

Three-level virtual-key budget hierarchy with cache-first pre-routing checks. TPM/RPM limits are synchronous; spend / usage persistence is async after the response. On cache miss, key validation can hit DB. Custom pricing lives under model_info.
Quick-Pick
Kong AI Gateway
The ai-rate-limiting-advanced plugin (Kong 3.7+, AI License) evaluates limits in the Access phase before the upstream call. Cost-based limiting can use model.options.input_cost / output_cost; the caveat is that cost reflection is documented on the next request.
Quick-Pick

Quick-Pick

Synchronous data-plane enforcement plus the FinOps trio business teams usually need: Budget Limiting at user/team/workload scope, Cost Attribution by team/user/model/app via metrics and metadata, and Custom Pricing Models (Public Cost / Private Cost) for internal chargebacks.

§1 — Why Synchronous Enforcement Is a Different Problem

When a platform engineer thinks "budget limit," the mental model is a cap stored in a database. An LLM gateway breaks every assumption about how that cap gets checked:

  • Concurrent agentic requests overwhelm async counters. A single agent framework can dispatch 50 parallel completions. If your counter is updated post-response, all 50 are in flight before the first decrement fires. The cap is real; the enforcement is not.
  • Cost is only known after the response. Dollar cost requires knowing actual output token counts — available only when the LLM responds. This creates tension between precision and synchronicity. Token-per-minute limits, by contrast, can be checked pre-dispatch.
  • Pricing is not uniform across providers. GPT-4o, Claude Sonnet 4.6, and a self-hosted Llama 3.3 70B on your GPU cluster all have different effective per-token costs. A gateway that cannot express custom rates cannot produce meaningful chargebacks for internal infrastructure.
  • Rate limits and budget limits serve different functions. TPM/RPM limits protect against burst abuse and can be enforced pre-request. Dollar budgets enforce financial ceilings and require either a forward estimate or a reserve-and-reconcile pattern to be truly synchronous.
Async enforcement exposes a concurrency window where multiple expensive requests can launch against stale budget state. In practice, that means finance sees an overrun only after engineering and business users have already experienced failures, retries, or degraded service. Synchronous in-memory checking narrows that window to the request that actually crosses the threshold.
📌 Cost Prediction vs. Cost Accounting

Most gateways know actual dollar cost only after the LLM responds (the usage object in the response). True synchronous dollar-budget blocking therefore requires either (a) blocking on a worst-case input-token estimate pre-dispatch, or (b) using token-rate limits (TPM) as the synchronous hard ceiling and treating dollar budgets as secondary thresholds updated post-response. Both are valid; understanding which your gateway implements tells you exactly where your exposure window is.

LiteLLM §2 — Three-Level Budget Hierarchy with Redis TPM Enforcement

LiteLLM Proxy structures budget enforcement around three concentric levels: a proxy-global budget (litellm_settings.max_budget), team budgets, and virtual-key budgets (max_budget + budget_duration per key). The current request-flow docs say the bearer token is checked up front to ensure it is valid and under budget, first in Redis or in-memory cache and, on cache miss, in the database.

Rate limits at the token-per-minute (tpm_limit) and request-per-minute (rpm_limit) level are enforced pre-routing. After the response is returned, LiteLLM performs asynchronous post-request work, including updating spend / usage in the database. Budget routing (provider_budget_config) is also supported, and current docs place custom pricing under model_info, with support that goes beyond simple per-token overrides.

LiteLLM checks virtual-key validity and budget before routing using cache first, with DB lookup on cache miss; spend / usage writes occur asynchronously after the response. Any concurrency overshoot for dollar budgets is an architectural inference from this flow, not a vendor-stated defect. Custom pricing lives in model_info. Based on LiteLLM request-flow and config docs.
# proxy_config.yaml — three-level budgets + custom pricing for a self-hosted model
model_list:
  - model_name: gpt-4o
    litellm_params:
      model: openai/gpt-4o
      api_key: os.environ/OPENAI_API_KEY

  - model_name: llama-3.3-70b-gpu
    litellm_params:
      model: openai/llama-3.3-70b
      api_base: http://vllm-cluster:8000/v1
      api_key: not-needed
    model_info:
      input_cost_per_token: 0.0000004
      output_cost_per_token: 0.0000012
      # Optional in current docs: input_cost_per_second, output_cost_per_second

litellm_settings:
  max_budget: 5000.0
  budget_duration: "30d"
  success_callback: ["langfuse"]
  callbacks: ["otel"]

general_settings:
  master_key: os.environ/LITELLM_MASTER_KEY
  database_url: os.environ/DATABASE_URL

router_settings:
  provider_budget_config:
    openai:
      budget_limit: 200.0
      time_period: 1d

redis_host: redis
redis_port: 6379

⚠ Limitation — Dollar Budget Concurrency Window

The tpm_limit and rpm_limit checks are clearly pre-routing in the current request-flow docs. Spend / usage persistence is asynchronous after the response. Because eligibility is checked before routing and spend is written after the response, concurrent traffic can plausibly overshoot a dollar budget between updates; that is an architectural inference from the documented flow, not an explicit LiteLLM bug statement. Use TPM/RPM as the hard synchronous ceiling and treat max_budget as the financial guardrail.

Kong §3 — Plugin-Chain Token Throttling at the Data Plane

Kong Gateway enforces AI cost controls through the ai-rate-limiting-advanced plugin (minimum Kong 3.7, AI License). This plugin operates in Kong's Access phase — before any upstream call is dispatched. It supports sliding-window and fixed-window strategies backed by Redis or in-memory store and can limit by prompt tokens, completion tokens, total tokens, or cost (USD). Limits can be scoped by identifier such as consumer, credential, IP, service, header, path, or consumer-group, depending on plugin placement and configuration.

Limit evaluation happens in the Access phase, and a breach returns a 429 before the upstream call is made. For cost-based limiting, current Kong docs explicitly say the plugin uses input_cost and output_cost defined in AI Proxy or AI Proxy Advanced model.options, in whatever unit you choose. The caveat is equally explicit: Kong documents that cost is only reflected on the next request, so cost-based blocking is not fully synchronous for the request that crosses the threshold.

Kong's ai-rate-limiting-advanced plugin fires in the Access phase — before any upstream call. Token and request limits can be enforced synchronously, and cost-based limiting can use model.options.input_cost / output_cost. Kong's documented caveat is that cost is reflected on the next request. Based on Kong AI Rate Limiting Advanced and AI Proxy docs.
# kong.yaml (deck) — cost-based limiting with custom model costs
# Cost units below are arbitrary but must match the ai-rate-limiting threshold unit.
_format_version: "3.0"

plugins:
  - name: ai-proxy
    config:
      route_type: llm/v1/chat
      model:
        provider: llama2
        name: llama2
        options:
          llama2_format: ollama
          upstream_url: http://llama2-server.local:11434/api/chat
          input_cost: 0.40
          output_cost: 1.20

  - name: ai-rate-limiting-advanced
    config:
      identifier: consumer
      strategy: redis
      sync_rate: 0
      tokens_count_strategy: cost
      llm_providers:
        - name: llama2
          limit:
            - 50
          window_size:
            - 3600

# Kong docs caveat: cost is reflected on the next request, so the request that crosses
# the threshold may still complete before the following request is blocked.
Source: Kong — AI Rate Limiting Advanced · Kong — Monitor AI LLM metrics · Kong — AI Proxy Llama2 route example
📌 Kong Custom Pricing Note

Kong's current docs explicitly support cost-based limiting from model.options.input_cost and model.options.output_cost on AI Proxy / AI Proxy Advanced. That makes custom pricing possible for self-hosted targets as well. The real limitation is timing rather than expressiveness: Kong documents that cost is only reflected on the next request, so cost-based enforcement lags the request that actually crosses the threshold.

Portkey §4 — Workspace Budgets, Model Catalog Limits, and Alert Thresholds

Portkey's current governance surface is centered on the Model Catalog, which replaces the older Virtual Keys terminology with managed Providers and Models. Workspace-level usage limits and rate limits are documented via the admin SDK, while credential-level controls are now expressed through integrations, providers, and models rather than the legacy Virtual Key UX.

Rate limits and usage limits are both documented, but they behave differently. Current usage-limit policy docs say budget exhaustion returns HTTP 412 Precondition Failed, while rate-limit exhaustion returns 429 Too Many Requests. Current docs also document private models, custom models, and pricing overrides in Model Catalog. The caveat is operational rather than categorical: unsupported models can still show $0.00 until pricing support or an explicit pricing configuration is in place, and some custom-pricing controls are enterprise-specific.

Portkey's current governance model centers on workspaces plus Model Catalog providers/models. Workspace-level credit_limit and alert_threshold are set via the admin SDK or dashboard. Usage-limit exhaustion is documented as HTTP 412, while rate-limit exhaustion is 429. Based on Portkey administration, usage-policy, and Model Catalog docs.
# Python SDK — workspace-level budget + rate limit (admin surface)
# Source: portkey.ai/docs/product/administration/enforce-workspace-budget-limts-and-rate-limits
from portkey_ai import Portkey

portkey = Portkey(api_key="PORTKEY_API_KEY")

workspace = portkey.admin.workspaces.update(
    workspace_id="WORKSPACE_ID",
    usage_limits=[
        {
            "type": "cost",            # or "tokens"
            "credit_limit": 500,        # dollars if type == "cost"
            "alert_threshold": 400,
            "periodic_reset": "monthly"
        }
    ],
    rate_limits=[
        {
            "type": "requests",       # or "tokens"
            "unit": "rpm",            # "rpm" | "rph" | "rpd"
            "value": 120
        }
    ],
)

print(workspace)
Source: Portkey — Enforce Workspace Budget and Rate Limits · Portkey — Usage & Rate Limit Policies · Portkey — Model Catalog
# Python SDK — config-level retries + gateway features
# Source: portkey.ai/docs/product/ai-gateway/configs
from portkey_ai import Portkey

client = Portkey(
    api_key="PORTKEY_API_KEY",
    config={
        "retry": {
            "attempts": 3,
            "on_status_codes": [429, 500, 503]
        },
        "strategy": {"mode": "fallback"},
        "targets": [
            {
                "provider": "@openai-prod",
                "override_params": {"model": "gpt-4o"}
            },
            {
                "provider": "@anthropic-prod",
                "override_params": {"model": "claude-3-5-sonnet"}
            }
        ]
    }
)

# If a usage-limit policy is exhausted, Portkey returns HTTP 412.
# Rate-limit exhaustion remains HTTP 429.

⚠ Limitation — Self-Hosted Model Pricing & Reactive Enforcement

Portkey now documents private models, custom models, and pricing overrides in Model Catalog, so it is no longer accurate to say custom pricing is entirely unavailable. The practical caveat is that accurate budget enforcement depends on pricing support or an explicit pricing configuration. Unsupported models can still show $0.00 until pricing is configured, and cost budgets remain reactive relative to already-completed spend.

TrueFoundry §5 — Synchronous Data-Plane Budget Blocking & Cost-Per-Entity Controls

TrueFoundry's AI Gateway is documented as a low-latency, high-throughput proxy built on the Hono framework. The gateway plane architecture page explicitly states its core design principle: no external calls in the hot path — rate limits, load balancing, authentication, and authorization are all evaluated in-memory, with logs and metrics queued asynchronously. This is the architectural basis for synchronous enforcement: budget and rate-limit checks happen within the process before the upstream LLM call is dispatched.

Budget policy is expressed as typed YAML of type gateway-budget-config. The documented unit is cost_per_day (USD), meaning TrueFoundry tracks token spend as a cost figure natively. Budgets can be applied per-user, per-model, per-virtual-account, or per metadata dimension (e.g., metadata.project_id) using the budget_applies_per field. Rate limiting uses the companion gateway-rate-limiting-config type with dimensions including tokens_per_day and tokens_per_hour. Both config types use first-match rule semantics — rule order is part of correctness.

TrueFoundry's gateway plane performs in-memory budget and rate-limit checks before dispatching the upstream LLM call — no external DB hop on the hot path. Business-facing controls sit on top of that fast path: user/team budgets, workload-level attribution by model or app metadata, and Public Cost / Private Cost pricing for internal chargebacks across both managed APIs and self-hosted GPU endpoints.
# gateway-rate-limiting-config — token velocity controls
# Source: truefoundry.com/docs/ai-gateway/ratelimiting
name: platform-rate-limits
type: gateway-rate-limiting-config

rules:
  # First-match wins — order is part of correctness
  - id: "model-hourly-tokens"
    when: {}
    limit_to: 1000000
    unit: tokens_per_hour
    rate_limit_applies_per: ["model"]

  - id: "project-daily-tokens"
    when:
      metadata:
        environment: "production"
    limit_to: 5000000
    unit: tokens_per_day
    rate_limit_applies_per: ["metadata.project_id"]
Source: truefoundry.com — Rate Limiting · first-match semantics confirmed in handbook.
# gateway-budget-config — cumulative cost controls (USD)
# Source: truefoundry.com/docs/ai-gateway/budgetlimiting
name: platform-budgets
type: gateway-budget-config

rules:
  # Per-user daily budget
  - id: "user-daily-budget"
    when: {}
    limit_to: 50
    unit: cost_per_day
    budget_applies_per: ["user"]

  # Team monthly budget
  - id: "team-monthly-budget"
    when:
      subjects: ["team:engineering"]
    limit_to: 2000
    unit: cost_per_month

  # Per-virtual-account weekly budget
  - id: "va-weekly-budget"
    when: {}
    limit_to: 1000
    unit: cost_per_week
    budget_applies_per: ["virtualaccount"]

  # Per-project daily budget
  - id: "project-daily-budget"
    when:
      metadata:
        environment: "production"
    limit_to: 500
    unit: cost_per_day
    budget_applies_per: ["metadata.project_id"]
Source: truefoundry.com — Budget Limiting · truefoundry.com — Gateway Plane Architecture · truefoundry.com — Cost Tracking
✓ TrueFoundry — Budget Limits, Cost Attribution, and Custom Pricing Fit Together

TrueFoundry exposes budget_applies_per for per-user, per-model, per-virtual-account, and metadata-scoped budgets, and it also documents team-scoped budgets through subjects: ['team:…']. On the observability side, its metrics UI and APIs can group cost and usage by team, user, model, virtual account, and metadata — which is the practical basis for per-application or per-customer chargeback. On pricing, Public Cost and Private Cost let operators align tracked spend with provider list prices, negotiated contracts, or self-hosted GPU economics without changing the request path.

§6 — Custom Pricing for Self-Hosted GPU Models: Cross-Platform Summary

One of the most underappreciated FinOps challenges is accurately costing self-hosted workloads. When you run Llama 3.3 70B or Qwen2.5 72B on an internal H100 cluster, there is no provider invoice. The real cost is amortised hardware, electricity, networking, and engineering overhead — and it varies by team. Without custom pricing at the gateway level, none of this cost can be tracked, budgeted, or charged back accurately.

LiteLLM

  • Per-model pricing via model_info.input_cost_per_token / output_cost_per_token in proxy_config.yaml — confirmed in current config docs
  • Also supports broader custom pricing options including cost-per-second and zero-cost models
  • Pricing is still model-scoped rather than per-team or per-application on the same model
  • Custom rates flow into virtual-key spend tracking and dashboard analytics

Kong AI Gateway

  • ai-rate-limiting-advanced supports cost-based limiting from model.options.input_cost / output_cost
  • This applies to self-hosted targets too, provided the model costs are defined in AI Proxy / AI Proxy Advanced
  • The caveat is timing: Kong documents that cost is reflected on the next request
  • Good for metering and guardrails, but not a perfectly synchronous hard dollar stop

Portkey

  • Tracks spend for supported providers/models and documents Model Catalog pricing management
  • Custom models, private providers, and pricing overrides are now documented in current Portkey docs
  • Unsupported models can still show $0.00 until pricing is configured
  • Some custom-pricing and enterprise governance controls are plan-dependent

TrueFoundry

  • Budget Limiting is documented for users, teams, virtual accounts, models, and metadata-scoped workloads.
  • Cost Attribution is documented through dashboard/API grouping by team, user, model, virtual account, and custom metadata such as application or customer.
  • Custom Pricing Models are handled through Public Cost and Private Cost, which is useful for negotiated contracts and internal chargebacks on self-hosted models.
  • The same cost tracking surface applies to both managed API providers and self-hosted GPU endpoints in the fleet.

§8 — Editorial Verdict

LiteLLM
Best OSS Flexibility, With a Concurrency Asterisk
LiteLLM remains a strong OSS option for teams that want virtual keys, provider budget routing, and flexible pricing controls. The current docs support pre-routing key checks with cache first and async spend writes after the response. Treat TPM/RPM as the hard synchronous ceiling, and describe any dollar-budget concurrency window as an architectural inference rather than a documented LiteLLM flaw.
Ideal for: Cost-aware OSS deployments needing fast setup with multi-provider budget governance and provider-level spend routing.
Kong AI Gateway
Infrastructure-Grade Token Throttling
Kong's ai-rate-limiting-advanced plugin provides very strong synchronous token/request enforcement in the Access phase. The docs also support custom cost units through model.options.input_cost / output_cost, including self-hosted targets. The caveat is that Kong documents cost reflection on the next request, so cost-based blocking is slightly lagged even when token limits are precise.
Ideal for: Teams already operating Kong infrastructure who need rock-solid token throughput controls and can handle USD analytics externally.
Portkey
Developer-Friendly Budgets, Managed-Provider Focus
Portkey delivers a polished governance surface for teams that want workspace controls and Model Catalog-based model management with minimal operational overhead. Current docs now cover custom models, private providers, and pricing overrides. The limitation is enforcement shape rather than pure feature absence: usage budgets are still reactive relative to already-observed spend, and unsupported models need explicit pricing configuration to avoid $0.00 tracking.
Ideal for: Teams using managed providers exclusively who want low-ops budget governance without running gateway infrastructure.
TrueFoundry
Strictest Hot-Path Enforcement + Cost-Per-Entity Granularity
TrueFoundry is the clearest fit when you need strict hot-path enforcement and business-readable FinOps controls in the same platform. The docs now support a stronger story than simple budget blocking: user/team budget limiting, cost attribution by team/user/model/app, and Public Cost / Private Cost pricing for internal chargebacks across both hosted and self-hosted model fleets.
Ideal for: K8s-native platform teams, regulated environments, and enterprises that need one gateway surface for policy enforcement, chargeback-ready attribution, and custom pricing.

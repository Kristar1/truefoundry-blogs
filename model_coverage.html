§0 — TL;DR & Quick-Picks

🚨 Real-World Scenario: The 3 AM Vendor Lock-In Crisis

It's 3:17 AM. Your on-call engineer is paged: your enterprise OpenAI account has been throttled to 20% of normal rate limits due to a billing dispute. Your document-intelligence pipeline—processing 40,000 requests per hour—is returning 429s.

For non-engineers, a 429 Too Many Requests response means the upstream model provider is refusing to serve more traffic right now. In business terms, that means user workflows stall, queued jobs back up, customer-facing features become intermittent, support tickets spike, and every minute of delay pushes you closer to missed SLAs, contractual penalties, and executive escalation.

The engineering lead pulls up the codebase. The OpenAI base URL is hardcoded in 14 microservices across 3 languages. Every prompt has been tuned for GPT-4o's specific parameter behavior. A full migration would take a sprint—but you have 15 minutes before dashboards turn red, internal stakeholders start asking for ETAs, and customers notice their documents are no longer being processed on time.

With an AI Gateway providing true API unification: the fix is a config edit. Your applications still call production-chat as a stable logical model name. The gateway silently re-routes that alias to claude-opus-4-6 or a self-hosted Llama-3.3-70B-Instruct instance already running in your VPC. No application code changes. No redeployment. No emergency vendor-specific patching across 14 services. This is what API unification means in production: the business sees continuity while the platform team swaps the underlying provider behind the scenes.

§1 — Why API Unification Is the First Problem to Solve

Every major LLM provider has diverged on the operational details that matter: authentication header patterns, streaming event formats, function-calling schemas, error codes, and endpoint path conventions. OpenAI's /v1/chat/completions, Azure OpenAI's deployment-scoped paths like /openai/deployments/{deployment_name}/chat/completions, Anthropic's x-api-key header, and Google Vertex's OAuth2 token flow are all meaningfully different at the wire level.

Without a unification layer, every application team becomes its own LLM control plane—implementing provider-specific retry logic, handling divergent error codes, managing credential rotation per vendor, and maintaining forks of inference code for each provider. The blast radius of any vendor incident is proportional to how deeply that provider's API shape has leaked into application code.

There are three structural layers to solve:

  • Layer 1 — Wire format normalization: Translating request/response schemas between an OpenAI-compatible client and non-OpenAI upstream endpoints without application changes.
  • Layer 2 — Credential abstraction: Centralizing provider API keys, IAM roles, and tokens so that applications present a single gateway credential, not per-vendor secrets.
  • Layer 3 — Model namespace stability: Exposing logical model names (e.g., production-chat) that decouple application intent from the physical provider and deployment decision.
API unification solves three stacked problems. All four gateways address Layer 1 and 2; the depth of Layer 3 (logical namespace stability) is where they diverge. Editorial illustration.
LiteLLM §2 — Python-Native Universal Adapter

Architecture

LiteLLM's unification model is built around the model_list in config.yaml. Each entry exposes a model_name—the logical alias your application calls—mapped to a litellm_params block containing the actual upstream model identifier, api_key, and api_base. The docs are explicit: "model_name is the name received by the proxy, while litellm_params.model is the identifier actually routed on the backend."

Critically, multiple entries can share the same model_name. The proxy treats these as an implicit load-balanced group under the active routing strategy (usage-based-routing, latency-based-routing, simple-shuffle, least-busy). This means multi-provider fallback is a zero-application-code-change operation. The request pipeline places virtual-key validation and budget checks before the router, documented at docs.litellm.ai/docs/proxy/architecture—so routing decisions are informed by identity and spend state before provider health is evaluated.

# config.yaml — multi-provider model_list with self-hosted endpoint
# Source: docs.litellm.ai/docs/proxy/configs · docs.litellm.ai/docs/proxy/reliability

model_list:

  # Logical name "production-chat" → OpenAI GPT-4o (primary)
  - model_name: production-chat
    litellm_params:
      model:     openai/gpt-4o
      api_key:  os.environ/OPENAI_API_KEY
      tpm: 200000
      rpm: 500

  # Same logical name → Anthropic Claude (load-balanced with above)
  - model_name: production-chat
    litellm_params:
      model:     anthropic/claude-sonnet-4-6
      api_key:  os.environ/ANTHROPIC_API_KEY
      tpm: 150000
      rpm: 400

  # Self-hosted vLLM — same interface, private cluster endpoint
  # api_base points at any OpenAI-compatible server
  - model_name: private-llm
    litellm_params:
      model:     openai/meta-llama/Llama-3.3-70B-Instruct
      api_key:  os.environ/INTERNAL_VLLM_KEY
      api_base: http://vllm-svc.ml-infra.svc.cluster.local:8000/v1

router_settings:
  routing_strategy: usage-based-routing
  num_retries: 2
  fallbacks:
    - production-chat: [private-llm]

general_settings:
  master_key: os.environ/LITELLM_MASTER_KEY

litellm_settings:
  callbacks: ["otel"]
  success_callback: ["langfuse"]
  service_callbacks: ["prometheus"]
  turn_off_message_logging: false
Source: docs.litellm.ai/docs/proxy/configs · docs.litellm.ai/docs/proxy/reliability · config_settings

Self-Hosted & Custom Provider Support

The api_base override is LiteLLM's most powerful self-hosted primitive. Any OpenAI-compatible endpoint—a vLLM server, a TGI deployment, an Ollama instance, or any custom inference API—is treated identically to a cloud provider once its api_base is set. The full governance stack applies: virtual keys, budget tracking, rate limits, and callback logging. Per the official docs, LiteLLM also supports pass-through endpoints for routing arbitrary external API traffic (rerankers, OCR, image APIs) through the same master key, spend tracking, and budget system.

⚠ Scale Constraint — Redis Required at Production Traffic

The official deployment guide states explicitly: Redis is required at 1,000+ requests per second to prevent database connection exhaustion and deadlocks. At this scale, Redis underpins rate-limit state, auth lookup cache, and load-balancing coordination simultaneously—not only response caching. This is an architectural planning dependency, not an optional optimization.

Kong §3 — Plugin-Chain Normalization on a Mature Gateway Substrate

Architecture

Kong AI Gateway is documented as an AI feature layer on top of Kong Gateway—not a separate broker process. The core architecture remains Service, Route, Upstream, Consumer, and plugin chains. AI behavior is composed via plugins attached to that substrate. The key distinction from the other three platforms is that LLM traffic inherits the same operational grammar as the rest of your API estate: the same decK declarative config, the same Konnect control plane, the same Consumer identity model.

The AI Proxy plugin (min. Kong 3.6) is the normalization engine. The official docs describe it as: "transforming and proxying requests to AI providers and models, accepting standardized OpenAI formats and translating them to the configured target format." The configuration surface is route_type—which maps capability intent to provider-specific upstream paths. AI Proxy Advanced (min. Kong 3.8, enterprise) extends this to multi-provider, multi-target routing with algorithm choice (round-robin, lowest-latency, lowest-usage, priority, semantic routing).

Kong's provider reference pages document the upstream-path translation explicitly per provider—making the normalization contract auditable rather than a black box:

Illustrative route_type to upstream-path mapping (not runnable config)

This mapping is documented in Kong's provider references, but it should be presented as an explanatory table rather than as copyable code.

Provider route_type Upstream path
OpenAI llm/v1/chat /v1/chat/completions
OpenAI llm/v1/completions /v1/completions
OpenAI llm/v1/embeddings /v1/embeddings
Azure OpenAI llm/v1/chat /openai/deployments/{deployment_name}/chat/completions
Azure OpenAI llm/v1/embeddings /openai/deployments/{deployment_name}/embeddings

Clients still send a normalized request to the Kong Route; AI Proxy resolves the configured provider plus route_type to the correct upstream path.

# Kong — AI Proxy plugin config (decK YAML format)
# Source: developer.konghq.com/plugins/ai-proxy/
#         developer.konghq.com/ai-gateway/get-started/

_format_version: "3.0"

services:
  - name: llm-chat-service
    url: https://api.openai.com   # illustrative upstream service URL; exact host/path behavior depends on AI Proxy provider config
    routes:
      - name: chat-route
        paths: ["/llm/v1/chat"]

plugins:
  - name: ai-proxy
    service: llm-chat-service
    config:
      route_type: llm/v1/chat         # capability → upstream path mapping
      auth:
        header_name:  Authorization
        header_value: "Bearer ${{ env \"DECK_OPENAI_API_KEY\" }}"
      model:
        provider: openai
        name:     gpt-4o

# For multi-target routing, use AI Proxy Advanced (enterprise, min. Kong 3.8)
# and move targets to the Upstream entity with balancer configuration
Source: developer.konghq.com/plugins/ai-proxy/ · AI Gateway Get Started
⚠ Enterprise Licensing Gates Key Unification Features

The most critical API unification features require explicit licensing. AI Proxy Advanced (multi-provider, multi-target routing, min. 3.8) is enterprise-only. AI Semantic Cache (min. 3.8) is enterprise-only. AI RAG Injector (min. 3.10) is enterprise-only. AI Rate Limiting Advanced (min. 3.7) requires an AI license. Always consult the plugin compatibility matrix for your specific version and license tier before planning architecture.

Portkey §4 — Config-First, Provider-Slug Addressing

Architecture

Portkey's unification design is built on two interlocking abstractions. The Model Catalog lets you define provider aliases as @provider-slug names—logical identifiers that hold the actual credentials and endpoint config. The Config object is a JSON policy carrier attached to API keys or individual requests to specify routing, fallback, retry, and caching behavior.

The result is an addressing pattern like @openai-prod/gpt-4o where @openai-prod is a stable logical name remappable to any credential, region, or provider without touching application code. Configs can be attached via SDK, via x-portkey-config header (making them compatible with any OpenAI-compatible client), or set as the default on an API key for organization-wide policy.

# Portkey - provider-slug addressing via Python SDK
# Source: portkey.ai/docs/guides/getting-started/getting-started-with-ai-gateway
#         portkey.ai/docs/product/ai-gateway/configs

from portkey_ai import Portkey

# The @provider-slug decouples application code from credentials entirely
portkey = Portkey(api_key="PORTKEY_API_KEY")

response = portkey.chat.completions.create(
    model="@openai-prod/gpt-4o",        # provider-slug + model reference
    messages=[{"role": "user", "content": "Summarise Q4 results"}]
)

# Swap to Anthropic without changing application code:
# Remap @openai-prod in the Model Catalog, OR pass a config at request time
response = portkey.with_options(
    config={
        "strategy": {"mode": "fallback"},
        "targets": [
            {"override_params": {"model": "@openai-prod/gpt-4o"}},
            {"override_params": {"model": "@anthropic-prod/claude-opus-4-6"}}
        ]
    }
).chat.completions.create(
    messages=[{"role": "user", "content": "Summarise Q4 results"}]
)
Source: portkey.ai/docs/product/ai-gateway/configs · Getting Started with AI Gateway
# Portkey — attach Config via HTTP header (any OpenAI-compatible client)
# Source: portkey.ai/docs/product/ai-gateway/configs

curl https://api.portkey.ai/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "x-portkey-api-key: $PORTKEY_API_KEY" \
  -H "x-portkey-provider: openai" \
  -H "x-portkey-config: $CONFIG_ID" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'

Hybrid VPC Deployment

For environments where prompts and responses cannot leave the customer network, Portkey provides a Helm-based data-plane deployment inside your VPC. The official deployment guide states explicitly: "deployment of the data plane inside your VPC, prompts and responses stay inside the network, and only metrics leave." The control plane and observability dashboards remain in Portkey's SaaS environment.

# Portkey — Helm-based VPC data-plane deployment
# Source: portkey.ai/docs/enterprise/hybrid2

helm repo add portkey https://portkey-ai.github.io/helm
helm repo update
helm upgrade --install portkey-gateway portkey/gateway \
  -n portkey \
  -f values.yaml          # your org-specific values override

# Health check the locally running data plane
kubectl -n portkey port-forward svc/portkey-gateway 8787:8787
curl http://localhost:8787/v1/health
# → {"status":"healthy"}

# First call through private gateway — prompts never leave your VPC
curl https://gateway.internal.mycompany.com/v1/chat/completions \
  -H "x-portkey-api-key: <your-portkey-api-key>" \
  -H "x-portkey-provider: @openai-prod" \
  -d '{"model":"gpt-4o","messages":[{"role":"user","content":"Hello"}]}'
Source: portkey.ai/docs/enterprise/hybrid2 — Deploy Portkey in Your Infrastructure
ℹ Budget & Rate Limit Policy Availability

Portkey's admin-level budget and rate-limit policy surfaces (Usage & Rate Limit Policies, workspace credit ceilings, per-dimension group-by rules) are explicitly documented as Enterprise-only or Select-Pro features. The OSS npx @portkey-ai/gateway runtime and the hosted SaaS tier have a subset of these controls. The distinction matters when evaluating Portkey as an internal platform versus a single-team SDK convenience layer.

TrueFoundry §5 — Kubernetes-Native Split-Plane Architecture

Architecture

TrueFoundry's API unification story is inseparable from its split-plane Kubernetes architecture: a control plane for orchestration and policy, a stateless gateway plane for request handling, a compute plane for model workloads, and a data plane for artifacts. The gateway plane is documented as built on the Hono framework, with an explicit hot-path design principle: "no external calls in the hot path unless cache is involved; in-memory checks for rate limits, load balancing, authentication, and authorization; async queueing for logs and metrics."

The gateway plane is stateless, subscribing to configuration updates via a queue with HTTP fallback—enabling cross-region multi-gateway deployments while configuration is managed centrally. This means gateway scaling is CPU-bound proxy scaling, not stateful coordinator scaling.

Provider Accounts as the Unification Primitive

TrueFoundry uses Provider Accounts as first-class objects: one per upstream credential domain, each exposing multiple models. The docs explicitly call out the multi-account pattern: multiple accounts from the same provider "for environment separation, quota isolation, or customer segmentation." This enables openai-prod, openai-dev, and openai-customer-A as distinct Provider Accounts with independent rate limits and budgets—without any application code changes.

# TrueFoundry — gateway request headers
# Source: truefoundry.com/docs/ai-gateway/headers

# Applications present a single gateway credential
Authorization: Bearer your-api-key

# x-tfy-metadata: stringified JSON; string keys AND string values only
# Used as segmentation primitive for rate limits and budget rules
x-tfy-metadata: {"project_id":"order-intelligence","environment":"prod","team":"ml-ops"}

# Optional per-request retry policy
x-tfy-retry-config: {"attempts": 3, "onStatusCodes": [429, 500, 503]}

# Response headers returned by the gateway (documented in official headers page):
# x-tfy-resolved-model     → physical model/provider that handled the request
# x-tfy-applied-configurations → which configs, rate limits, guardrails fired
# server-timing            → per-stage latency breakdown
#   Example: Auth 0.9ms | Input guardrails 0.7ms | Model 1350ms | Logging 1.1ms
Source: truefoundry.com/docs/ai-gateway/headers — Request and Response Headers
# TrueFoundry — rate limiting config (typed YAML, first-match-wins)
# Source: truefoundry.com/docs/ai-gateway/ratelimiting

name: ratelimiting-config
type: gateway-rate-limiting-config
rules:
  # First matching rule applies — rule ORDER is part of correctness
  - id: "model-daily-limit"
    when: {}          # matches all requests
    limit_to: 1000000
    unit: tokens_per_day
    rate_limit_applies_per: ['model']

  - id: "project-hourly-limit"
    when: {}
    limit_to: 50000
    unit: tokens_per_hour
    # metadata.project_id from x-tfy-metadata header — must be string value
    rate_limit_applies_per: ['metadata.project_id']
Source: truefoundry.com/docs/ai-gateway/ratelimiting

Self-Hosted LLM Deployment → OpenAI-Compatible Endpoint

TrueFoundry's most distinctive capability is managing both the deployment of self-hosted LLMs and their exposure through the governed gateway endpoint. Models deployed via the platform's LLM catalogue and registered in the Model Registry—with metadata including pipeline_tag, library_name, and base-model information—become deployable onto Kubernetes and are exposed as OpenAI-compatible endpoints. Gated Hugging Face model access is mediated through the platform's Secret Management system, not hardcoded into manifests. The public docs also show OpenAI-client usage with an HTTP Basic Authorization header; because the uploaded handbook excerpt appears to contain an extraction artifact in the credential-construction line, the code block below is presented as illustrative pseudocode rather than a verbatim doc transplant.

# TrueFoundry — illustrative pseudocode for OpenAI SDK + Basic Auth
# Source concept: truefoundry.com/docs/deploying-an-llm-model-from-the-model-catalogue

from openai import OpenAI

# Verify the exact credential-string construction against the current TrueFoundry docs.
# The important interoperability point is that the OpenAI SDK can attach a Basic Authorization header.
encoded_creds = "<base64-encoded-basic-auth-credentials>"

client = OpenAI(
    base_url="<YOUR_TRUEFOUNDRY_ENDPOINT>/v1",
    api_key="TEST",
    default_headers={"Authorization": f"Basic {encoded_creds}"}
)

response = client.chat.completions.create(
    model="meta-llama/Llama-3.3-70B-Instruct",
    messages=[{"role": "user", "content": "Explain MACD in one paragraph"}]
)
Source: truefoundry.com/docs/deploying-an-llm-model-from-the-model-catalogue — Deploying LLMs
⚠ Operational Prerequisite — Kubernetes Cluster + Control Plane

The deploy-gateway-plane guide requires a running Kubernetes cluster, domain, TLS certificate, image-pull credentials, and a control-plane API key (stored as a truefoundry-creds Kubernetes Secret) before any request is handled. Teams without a dedicated K8s platform engineering function face meaningful adoption overhead compared to the other three platforms.

§8 — Editorial Verdicts

LiteLLM
The Broadest-Coverage Pragmatist
The model_list + api_base pattern is one of the simplest paths from zero to a governed, multi-provider endpoint. Python's native provider matrix is among the broadest explicitly documented in this comparison. Adding an OpenAI-compatible self-hosted endpoint is a very small YAML change. The Redis dependency at scale and Python runtime constraints are real operational costs.
Ideal for: Fast-moving teams that need maximum provider coverage, GitOps config management, and don't want to manage a gateway cluster.
Kong AI Gateway
The Most Auditable Normalization Contract
The explicit route_type → upstream path documentation per provider makes Kong's normalization contract especially transparent and auditable. AI traffic inherits the full Kong governance model. The enterprise licensing gate for multi-target routing is a real adoption barrier for teams not already on Kong.
Ideal for: Existing Kong Enterprise users and teams needing AI traffic inside an established API governance posture.
Portkey
The Cleanest Config-Object Design
The @provider-slug addressing and Config-as-policy model is a notably developer-friendly unification design. Attaching routing, fallback, retry, and caching to a single Config ID referenced in one header is genuinely elegant. The split control/data plane in hybrid VPC mode is a real trade-off to understand before committing.
Ideal for: Product teams optimizing for developer experience, fast iteration, and minimal infrastructure ownership.
TrueFoundry
The Deepest Self-Hosted Integration
In this comparison, TrueFoundry is the platform that most explicitly combines self-hosted LLM deployment and governed routing in one system. The in-memory hot path, stateless Hono proxy, and x-tfy-resolved-model header for per-request auditability are operationally principled. Kubernetes as a hard prerequisite is the clearest adoption filter.
Ideal for: ML platform teams managing the full lifecycle from K8s LLM deployment to governed inference endpoint in their own VPC.

§9 — Positioning: Operational Complexity vs. Self-Hosted Integration Depth

X-axis = operational complexity to deploy and operate the gateway (not provider count). Y-axis = depth of self-hosted LLM integration. All four platforms offer broad provider coverage, but compatibility and feature depth vary by provider and configuration. Editorial positioning, not vendor-endorsed.

ℹ Bottom Line

If you're optimizing for provider breadth with minimum friction, LiteLLM is one of the strongest fits in this comparison. If you need AI traffic inside an existing Kong Enterprise gateway estate, Kong is the natural extension. If developer experience and config elegance matter most and you want a SaaS control plane, Portkey is especially compelling. If you need to manage the entire lifecycle from K8s LLM deployment to governed endpoint inside your own cluster, TrueFoundry is the platform in this comparison that most explicitly spans both sides.

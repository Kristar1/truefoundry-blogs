LiteLLM vs. Kong vs. Portkey vs. TrueFoundry

§0 — TL;DR & Quick-Picks

🔥 Real-World Scenario: The Black Friday LLM Meltdown

It's 11:47 PM on Black Friday. Your AI-powered customer service bot is handling 40,000 concurrent sessions against gpt-5.4. The provider starts returning 503 Service Unavailable errors — meaning it is temporarily unable to handle requests — and latency climbs past 9 seconds. To customers, that doesn't look like an HTTP issue. It looks like chats hanging, answers arriving too late to help, and some sessions failing altogether. Within 90 seconds, retries pile up, your queue fills, and delays spill into the rest of the customer journey. Support containment drops, carts get abandoned, and checkout conversion starts to fall.

This is not a freak event. Major model providers do occasionally report elevated errors and degraded performance on public status pages such as status.openai.com and status.claude.com, so provider-side disruption is something you should architect around, not treat as impossible. The architectural mistake is static routing with no fallback, no health awareness, and no weight redistribution. With GPT-5.4 at $2.50/1M input tokens and Claude Sonnet 4.6 at $3/1M, you now have two tier-1 frontier models to balance across — static single-provider routing is hard to justify for production customer workflows.

The problem with LLMs specifically is that routing them isn't like routing HTTP microservices. Wall time is non-deterministic, token budgets are asymmetric across providers, Time-to-First-Token (TTFT) variance can swing dramatically under load, and rate limits are enforced per-model, per-org, per-minute — not per-connection. Any gateway solving this correctly must be LLM-aware, not just network-aware.

§1 — Why LLM Routing Is a Different Problem

When a platform engineer says "load balancing," the mental model is round-robin across identical replicas behind a VIP. LLM traffic breaks every assumption:

  • TTFT variance is non-deterministic and model-specific. In practice, TTFT variance can swing dramatically under load. gemini-3.1-pro-preview and claude-opus-4-6 show similarly heavy long tails under queue pressure. Static routing has no mechanism to respond.
  • Rate limits are multi-dimensional. GPT-5.4 enforces limits per-minute on both tokens and requests, per-org, and per-deployment. Distributing load without understanding these dimensions causes 429 storms that are worse than the original bottleneck.
  • SLA requirements differ wildly by application. A real-time voice agent needs TTFT under 300ms and should shed aggressively to gemini-3.1-flash-lite or claude-haiku-4-5. A batch summarization job can tolerate 30-second completions and should optimize cost-per-token on gpt-5-mini. No single policy fits both.
  • Model fallbacks are semantically meaningful. Failing from gpt-5.4 to claude-sonnet-4-6 isn't transparent — format and quality can diverge. Routing logic must encode which fallback is acceptable for which workload class.
  • Spend is a first-class routing signal. With frontier models ranging from $0.25/1M (Gemini 3.1 Flash-Lite) to $5/1M (Claude Opus 4.6 input) to $2.50/1M (GPT-5.4 input), daily provider budget exhaustion is a legitimate routing trigger — not an error condition.
HTTP load balancing assumes homogeneous replicas. LLM routing must account for latency variance, token economics, budget state, and model-specific semantics.

LiteLLM §2 — Python-Native Budget-Aware Router

Architecture

LiteLLM's routing layer is built around a Python Router class initialized with a model_list array. Each entry maps a logical model_name (the name the proxy exposes) to a litellm_params block containing the actual upstream model identifier, api_key, api_base, and per-model rate caps. The request pipeline, documented at docs.litellm.ai/docs/proxy/architecture, places virtual-key validation and budget checks before the router — meaning routing decisions are informed by identity and spend state, not just provider health.

The routing engine supports: simple-shuffle, least-busy, latency-based-routing (EWMA of response times), and usage-based-routing (TPM/RPM headroom). Budget routing (provider_budget_routing) lets you attach daily spend ceilings to providers or models — the router excludes exhausted targets from selection, making economics a first-class routing signal alongside availability. Per-key and per-team router overrides are documented at keys_teams_router_settings, making the routing fabric genuinely multi-tenant.

# Adapted LiteLLM pattern based on current routing / fallback / budget docs
model_list:
  - model_name: flagship-primary
    litellm_params:
      model: openai/gpt-5.4
      api_key: os.environ/OPENAI_API_KEY
      weight: 60
      tpm: 200000
      rpm: 500

  - model_name: flagship-secondary
    litellm_params:
      model: anthropic/claude-sonnet-4-6
      api_key: os.environ/ANTHROPIC_API_KEY
      weight: 40
      tpm: 150000
      rpm: 400

  - model_name: cost-fallback
    litellm_params:
      model: google/gemini-3-flash
      api_key: os.environ/GOOGLE_API_KEY

router_settings:
  routing_strategy: usage-based-routing
  num_retries: 2
  fallbacks:
    - flagship-primary: [flagship-secondary, cost-fallback]
  provider_budget_config:
    openai:
      budget_limit: 200.0
      time_period: 1d
    anthropic:
      budget_limit: 100.0
      time_period: 1d
    google:
      budget_limit: 50.0
      time_period: 1d
Source: docs.litellm.ai/docs/proxy/reliability · provider_budget_routing · routing-load-balancing
How to run this pattern

Save the YAML as config.yaml, export the provider API keys referenced in the file, and start LiteLLM Proxy against that config. If you turn on shared routing state or budget-aware routing in production, plan on Redis as part of the deployment rather than treating it as optional scale garnish. Replace the example logical aliases and model IDs with the exact ones you expose in your environment.

LiteLLM places identity and spend checks before the router — economics govern routing decisions before target health is even evaluated. Editorial reconstruction based on LiteLLM architecture docs.
⚠ Limitation — Python Runtime & Scale Considerations

LiteLLM's Python runtime can become an operational scaling constraint under heavy concurrency; Redis and deployment design matter in production. For higher-throughput LiteLLM deployments, Redis and careful database/connection planning become part of the intended architecture.

Kong §3 — Gateway-First AI Plugin Stack with Semantic Features

Architecture

Kong AI Gateway is documented at developer.konghq.com/ai-gateway as an AI feature layer on top of Kong Gateway — not a separate broker process. The core architecture is still Services, Routes, Upstreams, Consumers, and plugin chains; AI behavior is a composition of plugins attached to that substrate.

The AI Proxy plugin (min. Kong 3.6) normalizes provider request/response formats. AI Proxy Advanced (min. Kong 3.8, enterprise) extends that to multi-target routing with: round-robin, least-connections, lowest-latency, lowest-usage, priority, consistent-hashing, and semantic routing. The AI load balancer also supports retries, fallback behavior, and balancer failure-threshold controls that function as circuit-breaking behavior.

# Validated Kong pattern — documented ai-proxy-advanced least-connections shape + provider rate limiting
_format_version: "3.0"

services:
  - name: llm-service
    url: http://localhost
    routes:
      - name: llm-route
        paths: [/v1/chat/completions]

plugins:
  - name: ai-proxy-advanced
    service: llm-service
    config:
      balancer:
        algorithm: least-connections
        retries: 3
        failover_criteria: [error, timeout, http_429, non_idempotent]
      targets:
        - model:
            name: gpt-4o
            provider: openai
            options:
              max_tokens: 1024
              temperature: 1.0
          route_type: llm/v1/chat
          weight: 2
          auth:
            header_name: Authorization
            header_value: Bearer ${{ env "DECK_OPENAI_API_KEY" }}
          logging:
            log_statistics: true
            log_payloads: true
        - model:
            name: gpt-4o-mini
            provider: openai
            options:
              max_tokens: 1024
              temperature: 1.0
          route_type: llm/v1/chat
          weight: 2
          auth:
            header_name: Authorization
            header_value: Bearer ${{ env "DECK_OPENAI_API_KEY" }}
          logging:
            log_statistics: true
            log_payloads: true

  - name: ai-rate-limiting-advanced
    service: llm-service
    config:
      llm_providers:
        - name: openai
          limit: [100, 1000]
          window_size: [60, 3600]

How to run this pattern

This block now uses a documented ai-proxy-advanced shape instead of hand-invented balancer keys. Create the Service and Route, attach ai-proxy-advanced and ai-rate-limiting-advanced, then apply with decK or the Admin / Konnect API. Replace the example models and auth values with the exact provider credentials and model IDs enabled in your environment. If you want lowest-latency instead of least-connections, switch only to fields that exist in the current Kong examples / reference — do not add undocumented keys like latency_strategy.

Kong's AI features are C/Lua plugins composed in a non-blocking data plane — not a separate broker process. Editorial reconstruction based on Kong AI Gateway docs; not an official vendor architecture diagram.
⚠ Limitation — Feature Availability Still Depends on Plugin / Version

Kong documents lowest-latency routing and balancer failover behavior in AI Proxy Advanced, but this post now shows a least-connections example because its config shape is easier to validate line by line against the public docs. The real limitation is operational complexity: advanced routing features are distributed across specific enterprise plugins and Kong versions. Always consult the plugin compatibility reference before treating an example like a production-ready manifest.

Portkey §4 — Config-First Composable Control Plane

Architecture

Portkey's architecture is a fully managed SaaS Anycast edge network, but the deeper product is the config object — a JSON document that encapsulates routing strategy, fallback chains, retry logic, caching, guardrails, and provider credentials. As documented at portkey.ai/docs/product/ai-gateway/configs, a config can be attached via the SDK, via x-portkey-config header, or set as a default on an API key.

The routing strategies are genuinely composable as a nested graph: loadbalance (weighted), fallback (prioritized sequential), and conditional (MongoDB-style query operators on metadata, request params, or URL path). Circuit breaker, canary testing, request timeouts, and automatic retries are all first-party features that compose into the same config object.

{
  "strategy": {
    "mode": "fallback",
    "on_status_codes": [429, 500, 502, 503, 504]
  },
  "targets": [
    {
      "strategy": { "mode": "loadbalance" },
      "targets": [
        { "override_params": { "model": "@openai/gpt-4o" }, "weight": 1 },
        { "override_params": { "model": "@azure/gpt-4o" }, "weight": 1 }
      ]
    },
    { "override_params": { "model": "@anthropic/claude-sonnet-4-5-20250514" } }
  ],
  "retry": {
    "attempts": 3,
    "on_status_codes": [429, 500, 502, 503, 504]
  }
}
Source: Portkey — Combining Routing Strategies · Fallbacks · Automatic Retries · Gateway Config Object
How to run this pattern

Create the config in Portkey UI or through the Portkey API, save it to get a config ID, and attach that config either to an API key or per request with the SDK / x-portkey-config header. This block is now strict JSON: no // comments, no trailing notes inside the object. If you need annotations, keep them outside the JSON. An extra field like "_comment"

might work only if the Portkey schema ignores unknown keys, so do not rely on that unless you have tested it in your own workspace. Replace the example model aliases with the exact provider/model slugs available in your environment.

Portkey's config object composes routing strategies as a nested graph. Conditional routing, load balancing, and fallback chains are siblings, not separate systems. Editorial reconstruction based on Portkey docs.
⚠ Limitation — Cloud Control Dependency & Network Hop

All routing logic traverses Portkey's cloud edge. Portkey does offer a hybrid deployment path (portkey.ai/docs/enterprise/hybrid2) where the data plane runs inside your VPC via Helm — prompt and response payloads stay in your network while only metrics leave. However, the control dashboard and higher-level features remain in Portkey's cloud.

TrueFoundry §5 — K8s-Native Hono Data Plane with Typed YAML Routing Policy

Architecture

TrueFoundry's AI Gateway is a Kubernetes-native data plane built on the Hono framework. The gateway plane architecture page documents its hot-path design explicitly: no external calls during request handling (unless cache is involved), in-memory evaluation for rate limits, load balancing, authentication, and authorization, with async queueing for logs and metrics. The gateway plane is stateless, subscribing to configuration updates via a queue with HTTP fallback — enabling cross-region multi-gateway deployments while configuration is managed centrally.

Routing policy is expressed through typed YAML configs. The load-balancing-overview and load-balancing-configuration docs describe three routing strategies: weight-based, latency-based (using time per output token / inter-token latency in the data plane), and priority-based (with adaptive SLA cutoff — unhealthy targets recover when performance improves, not after a fixed cooldown).

TrueFoundry's gateway plane is stateless, Kubernetes-native, and evaluates routing + guardrails entirely in-memory on the hot path. Logs and metrics are queued asynchronously. Editorial reconstruction based on TrueFoundry architecture docs.
TrueFoundry's adaptive SLA cutoff: unhealthy targets are not blocked for a fixed cooldown. The gateway continues passive sampling and restores traffic when performance recovers — aligning with rapidly fluctuating LLM serving systems. Based on TrueFoundry load-balancing-overview docs; illustrative timing, not measured data.
# Adapted TrueFoundry pattern — routing + rate limiting with scoped when clauses
name: production-llm-routing
type: gateway-load-balancing-config
rules:
  - id: "realtime-voice-prod"
    type: "latency-based-routing"
    when:
      subjects:
        - "team:voice-platform"
      models:
        - "realtime-assistant"
      metadata:
        environment: "production"
        workload: "voice"
    load_balance_targets:
      - target: "openai/gpt-5.4"
        retry_config:
          attempts: 1
      - target: "anthropic/claude-sonnet-4-6"
        retry_config:
          attempts: 1

  - id: "support-chat-canary"
    type: "weight-based-routing"
    when:
      subjects:
        - "team:ai-platform"
      models:
        - "support-chat"
      metadata:
        environment: "staging"
        release_channel: "canary"
    load_balance_targets:
      - target: "openai/gpt-5.4"
        weight: 90
      - target: "openai/gpt-5-mini"
        weight: 10
---
name: ratelimiting-config
type: gateway-rate-limiting-config
rules:
  - id: "voice-prod-team-daily"
    when:
      metadata:
        environment: "production"
        workload: "voice"
    limit_to: 2000000
    unit: tokens_per_day
    rate_limit_applies_per: ['metadata.team_id']
  - id: "support-chat-global-hourly"
    when:
      models:
        - "support-chat"
    limit_to: 500000
    unit: tokens_per_hour
    rate_limit_applies_per: ['model']
Source: load-balancing-overview · load-balancing-configuration · ratelimiting · budgetlimiting · virtual-model
How to run this pattern

Save the YAML as a gateway policy file and apply it through your normal TrueFoundry configuration workflow, such as the typed YAML / tfy apply path used in your environment. The example assumes your workspace already has matching logical models, subject selectors, and metadata keys, so replace those with the exact names your gateway recognizes before rollout. Validate the result through resolved-model headers, applied-configuration headers, and traces.

TrueFoundry's four-hook guardrail system (LLM input, LLM output, MCP pre-tool, MCP post-tool) is documented at guardrails-overview. OTEL-compliant trace export is documented at export-opentelemetry-data. Response headers including x-tfy-resolved-model and x-tfy-applied-configurations expose exactly which routing decisions and control surfaces fired per request.

⚠ Limitation — K8s Operational Prerequisite

The deploy-gateway-plane guide at deploy-gateway-plane requires a running K8s cluster, domain, TLS certificate, image-pull credentials, and control-plane API key before a single request is handled. For teams without a dedicated K8s platform engineering function, this is a real adoption barrier.

§7 — Editorial Verdict

Editorial assessments as of March 10, 2026.

LiteLLM
The Budget-Aware Pragmatist
Wins on iteration velocity and economic intelligence. The model_list + router_settings + provider_budget_config pattern expresses multi-provider routing, fallback chains, and spend governance in a single YAML file.
Ideal for: Fast-moving platform teams, internal tooling, budget-aware multi-provider routing without heavy gateway infrastructure.
Kong AI Gateway
The Network Workhorse with Semantic Depth
The right choice when you're already on Kong, need extreme concurrency, and want semantic routing, RAG injection, or gateway-level PII sanitization.
Ideal for: Existing Kong users, very high-throughput workloads, teams needing semantic cache / RAG injection / MCP governance.
Portkey
The Composable Control Plane
The config-first model is genuinely sophisticated: conditional routing with MongoDB-style operators, circuit breakers, canary testing, semantic cache, guardrails, and OTel tracing all compose through one JSON object.
Ideal for: Product teams wanting composable routing + observability without K8s or Lua expertise; SaaS applications.
TrueFoundry
The K8s-Native Platform Contender
Most architecturally coherent for K8s-native platform teams. Native latency-based routing using inter-token latency / TPOT, adaptive priority with SLA cutoff, four-hook guardrails, typed YAML policies, and OTEL export.
Ideal for: K8s-native platform teams, regulated industries, agentic/MCP workloads, large-scale AI platform builds.

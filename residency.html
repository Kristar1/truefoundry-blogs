§0 — TL;DR & Quick-Picks

⚠ Real-World Scenario: The GDPR Audit That Started with a Prompt Log

A German fintech running AI-assisted KYC document analysis — ingesting passport scans, IBANs, and tax identification numbers — receives a formal inquiry from the German DPA (Datenschutzbehörde) under GDPR Article 46. The investigation trigger: their AI gateway's default log-shipping configuration was silently synchronizing full prompt payloads, including the structured JSON containing customer PII, to a SaaS observability dashboard hosted in the United States. No standard contractual clauses were in place. No pseudonymisation. No consent.

The engineering team had understood "prompt logging" as a debug convenience, not a data-transfer mechanism. But from a regulatory standpoint, each request log was a cross-border transfer of personal data to a third-country sub-processor. The gateway's payload logging configuration had quietly created a compliance liability that cost months of remediation, mandatory DPA notification, and significant legal overhead.

This scenario is not hypothetical edge-case paranoia. As AI platforms grow, every gateway that logs prompt and response content becomes a de-facto data processing system — one that must satisfy the same residency, sub-processor, and retention obligations as any other enterprise data store. The choice of gateway architecture determines whether your compliance posture is strong by design, achievable through configuration, or fundamentally constrained by a SaaS control dependency you cannot fully govern.

§1 — Why AI Gateway Data Sovereignty Is a Different Problem

Most data residency discussions in enterprise architecture focus on databases, blob storage, and message queues. AI gateways introduce a qualitatively different challenge: the request payload itself is the sensitive artifact. A prompt sent to claude-sonnet-4-6 for legal contract analysis is not metadata — it is the contract. A chat message routed through gpt-5.4 for customer support may contain medical history, financial details, or government identifiers. The gateway sees every word of it.

This creates five distinct data surfaces that each AI gateway handles differently, and which compliance teams must independently evaluate:

  • Prompt and response payload logging. The full request/response body. This surface is commonly logged unless teams explicitly disable, redact, or privacy-scope logging. It is the highest-risk surface for PII exposure.
  • Request metadata and usage analytics. Model name, token counts, latency, cost, consumer identity. Lower sensitivity, but still subject to data minimization requirements when linked to users.
  • Distributed traces. Span trees capturing the gateway's internal decision path — which routing rules fired, which fallback was used, which guardrails triggered. Often contains derived context from the payload.
  • Audit logs. Security and compliance events: who called which model, with which key, at what time. Must satisfy retention and access-control obligations independent of payload logging.
  • Control plane state. API key metadata, team membership, budget state, routing policy. Sensitive organizational data, even if not user PII.

No two gateways in this comparison handle all five surfaces the same way. The architectural commitments each vendor has made — SaaS-first, self-hosted, K8s-native, plugin-composable — fundamentally determine what is possible without forking the product. Understanding those commitments is the right starting point for any residency evaluation.

Five AI gateway data surfaces. Surface ① — full prompt/response payloads — is the primary GDPR/HIPAA/CCPA liability. Gateway architecture determines which surfaces are operator-controlled vs. vendor-managed.
LiteLLM §2 — Callback-Driven Data Locality with Budget-Aware Identity

Residency Architecture

LiteLLM's data residency story is built on a single foundational commitment: the proxy is fully self-hostable, and all data flows are operator-configured. There is no cloud control plane and no mandatory SaaS telemetry endpoint in a self-deployed proxy setup. The docs are clear that the proxy can run on Docker, Kubernetes, or ECS without any external dependency beyond the database you provide.

Log destinations are expressed entirely through the callback system documented at docs.litellm.ai/docs/proxy/config_settings. The callbacks, success_callback, failure_callback, and service_callbacks fields each accept a list of named integrations. Every integration is an outbound call — and every outbound call is a potential data residency decision. Teams in regulated environments should treat each callback destination as a data sub-processor requiring an explicit data processing agreement.

The most operationally important residency toggle is turn_off_message_logging: true. This single setting, documented in the full settings reference at docs.litellm.ai/docs/proxy/config_settings, disables logging of prompt and response content while retaining metadata (model, tokens, cost, latency). This is the correct default posture for any deployment touching user PII: retain operational metrics, suppress raw payload content.

The x-litellm-call-id header, documented in the logging page at docs.litellm.ai/docs/proxy/logging, provides the correlation handle for cross-component tracing without requiring payload transmission. Operators can use this ID to correlate gateway logs with application logs while keeping the prompt content entirely within the originating application tier. Separately, LiteLLM's current UI logs docs state that request and response content is not stored by default unless teams explicitly enable prompt storage.

Payload Masking Before Storage

For environments where logging cannot be disabled entirely — audited systems, model quality reviews, root-cause analysis pipelines — LiteLLM supports PII redaction via the Presidio guardrail integration. This approach executes PII entity detection as a pre_call hook, substituting entity values with mask tokens before the request proceeds to the router. The guardrail is configured in config.yaml under the documented guardrails key at docs.litellm.ai/docs/proxy/guardrails/quick_start.

# LiteLLM — payload masking with Presidio + selective log suppression
# Source: docs.litellm.ai/docs/proxy/config_settings · guardrails/quick_start
model_list:
  - model_name: flagship-primary
    litellm_params:
      model: anthropic/claude-sonnet-4-6
      api_key: os.environ/ANTHROPIC_API_KEY

litellm_settings:
  turn_off_message_logging: true      # suppress raw prompt/response from logs
  callbacks: ["otel"]                  # route traces to operator-controlled OTel backend
  success_callback: ["langfuse"]       # application-level events — operator-hosted Langfuse
  failure_callback: ["sentry"]         # error events — no payload in error metadata
  service_callbacks: ["prometheus"]    # service callback for Redis/Postgres failures

guardrails:
  - guardrail_name: presidio-pii-redaction
    litellm_params:
      guardrail: presidio
      mode: pre_call               # redact before model call AND before logging
      presidio_language: en
      pii_entities_config:
        CREDIT_CARD: MASK
        EMAIL_ADDRESS: MASK
        PHONE_NUMBER: MASK
        PERSON: MASK
        IBAN_CODE: MASK
      default_on: true
Source: config_settings · guardrails/quick_start · logging
LiteLLM routes logs and metrics exclusively via operator-configured callbacks. With turn_off_message_logging: true, no payload content reaches any backend. All callback destinations are under operator control — including self-hosted Langfuse, on-prem OTel collectors, and internal Prometheus. Current LiteLLM docs also note that UI request/response content storage is opt-in rather than default-on.
⚠ Limitation — PII Redaction Requires External Presidio Service

LiteLLM's PII masking is not a built-in in-process capability. It delegates to the Presidio guardrail service, documented at docs.litellm.ai/docs/proxy/guardrails/quick_start. This means PII redaction introduces an external service dependency on the hot path: if Presidio is unavailable, the guardrail fails. For regulated environments, this dependency must itself be scoped to the same data-residency zone. The approach is architecturally sound, but operationally it shifts one sub-processor concern to another.

Kong §3 — Plugin-Stack Data Governance with Gateway-Native PII Sanitization

Residency Architecture

Kong AI Gateway's data residency posture inherits from the broader Kong Gateway philosophy: every behavior is a plugin, and the plugin chain is composable, version-controlled, and deployable entirely on customer infrastructure. The AI audit log reference at developer.konghq.com/ai-gateway/ai-audit-log-reference is explicit about the mechanism: AI Gateway emits structured analytics logs for AI plugins through the standard Kong Gateway logging infrastructure, and those logs can be shipped to external systems via standard logging plugins. There is no mandatory Konnect SaaS dependency in self-managed deployments.

The observability data model in Kong is layered across three distinct mechanisms. First, structured AI audit logs flow through whatever Kong logging plugin you attach — http-log, file-log, tcp-log, or any other compatible plugin. Second, the Prometheus plugin, when config.ai_metrics: true is enabled, exposes AI-specific metrics including AI request counts, AI cost, AI token totals, provider latency, and cache fetch latency — all without payload content. Third, the Gen AI OpenTelemetry attributes reference at developer.konghq.com/ai-gateway/llm-open-telemetry documents a rich set of span attributes (gen_ai.operation.name, gen_ai.provider.name, gen_ai.request.model, etc.) emitted via OTLP over HTTP with protobuf encoding to any compatible backend. All three mechanisms stay within the Kong data plane. One nuance matters for residency reviews: Kong's Gen AI payload attributes and AI audit logs can include full request and response content when payload logging is enabled, so privacy posture depends on which logging and tracing paths you turn on.

AI PII Sanitizer — Gateway-Native Payload Scrubbing

Kong's distinctive data governance capability is the AI PII Sanitizer plugin, documented at developer.konghq.com/plugins/ai-sanitizer. Unlike LiteLLM's approach of delegating PII detection to an external guardrail service called before the model, Kong's PII Sanitizer can operate on both request bodies (before reaching the upstream LLM) and response bodies (before returning to the client). It uses an external AI PII Anonymizer Service, and critically, includes an optional restoration mode: the gateway can re-inject original values back into the response after the model call, so downstream consumers receive unredacted answers while the model itself never processes the raw PII.

This bidirectional sanitization-and-restoration model is significant for regulated workloads where the user experience must not be degraded by anonymization: the patient-facing chatbot responds with the correct name, but the model called gpt-5.4 or claude-sonnet-4-6 only ever saw a placeholder token. Kong documents this as a sequence diagram on the AI PII Sanitizer page, showing the request path through the gateway, through the anonymization service, to the model, and back.

# Kong AI Gateway — AI audit log routing + Prometheus AI metrics

#         developer.konghq.com/ai-gateway/monitor-ai-llm-metrics

services:
  - name: llm-service
    url: http://localhost
    routes:
      - name: llm-route
        paths: [/v1/chat/completions]

Kong's AI data governance composes via the plugin chain. The AI PII Sanitizer (enterprise) intercepts and anonymizes payloads before model calls. Logs, metrics, and traces each route through separate, operator-configured plugin paths — no mandatory outbound to Konnect in self-managed deployments.
⚠ Limitation — Advanced Data Governance is Version and License Gated

Kong's AI PII Sanitizer (min. Kong 3.10, enterprise license), AI AWS Guardrails (min. Kong 3.11, enterprise), and AI Azure Content Safety (enterprise) are not available in open-source Kong Gateway. Self-managed enterprise deployments do not require Konnect connectivity and maintain full data residency. However, the full PII sanitization surface requires an enterprise contract and careful version planning against the plugin compatibility reference. Teams on Kong OSS must implement payload masking through custom Lua plugins or upstream application logic.

Portkey §4 — Hybrid VPC Data Plane with Cloud Control Dependency

Residency Architecture

Portkey's default deployment is a managed SaaS Anycast edge network. In this mode, prompt and response payloads transit Portkey's infrastructure. For many enterprise compliance frameworks, this is a non-starter: the SaaS endpoint is a data processor operating in vendor-controlled infrastructure, and the data processing agreement scope must encompass your full prompt content.

Portkey addresses this through a hybrid deployment path documented at docs.portkey.ai/docs/self-hosting/hybrid-deployments/architecture. In hybrid mode, the Helm-based data plane runs gateway pods inside your Kubernetes VPC while the control plane remains hosted by Portkey. Current docs describe the split as follows: the data plane keeps sensitive LLM traffic in your environment, while the Portkey-hosted control plane handles administration, configs, and analytics. Treat the “no data leaves your network” marketing summary as applying to the LLM traffic hot path, not to all control-plane synchronization.

This split creates a well-defined but incomplete residency posture. The hot path — prompt content, model responses, provider credentials — stays in your VPC. The control plane, including the config dashboard, guardrail configuration UI, analytics aggregation, and higher-level observability surfaces, operates in Portkey's infrastructure in hybrid mode. Current Portkey docs also expose configurable request-logging modes such as Full Logging, Metrics Only, and Do Not Track behaviors for sensitive traffic. For audit and compliance purposes, this means prompt payload residency is achievable, but control-plane sovereignty is only partial in hybrid deployments. Organizations subject to stricter frameworks should map this boundary explicitly before adopting the hybrid mode.

Payload Masking and Guardrails

Portkey's PII Redaction feature, documented at portkey.ai/docs/product/guardrails/pii-redaction, can detect and redact phones, emails, locations, IP addresses, SSNs, names, and credit card information. Redaction is attached to configs via before_request_hooks and after_request_hooks. This means PII masking runs on the gateway data plane — within your VPC in hybrid mode — before any content is logged or transmitted.

For teams with existing compliance tooling, the Bring Your Own Guardrails webhook documented at portkey.ai/docs/integrations/guardrails/bring-your-own-guardrails is architecturally significant. The webhook contract accepts the full OpenAI-formatted request body along with request metadata, and the response determines whether the request proceeds or is blocked. In hybrid mode, this webhook can point to an internal endpoint in your VPC, keeping the payload entirely within your network boundary even during policy evaluation.

Enterprise OTel Analytics Export
While control-plane configuration remains SaaS-managed in hybrid deployments, Portkey Enterprise recently introduced automatic log export to OTLP-compatible HTTP endpoints. This allows organizations to push all telemetry—including GenAI semantic attributes, tool definitions, and token usage—directly into their own internal observability stack (e.g., Datadog, Splunk, or an in-VPC OTel Collector), strengthening the data localization posture for audit logs.

Portkey hybrid mode keeps prompt and response payloads on the customer side of the split for the LLM hot path, while the control dashboard, analytics, and configuration plane remain Portkey-hosted. This is a well-defined but partial residency posture in hybrid mode — suitable for many data-localization requirements, but not for frameworks requiring operator-controlled control-plane residency.
⛔ In Portkey Hybrid Deployments, Control Plane Is Portkey-Hosted

In hybrid VPC deployments, Portkey's docs confirm the control dashboard and higher-level features remain in Portkey's cloud. For organizations subject to FedRAMP, financial sector sovereignty rules (e.g., DORA in the EU), or frameworks requiring that all system components processing or controlling AI inference be under operator jurisdiction, the hybrid mode does not satisfy the complete requirement. The data plane residency is genuine; the control plane residency is vendor-managed. Treat this boundary explicitly in your DPA and sub-processor disclosure. Consult current Portkey enterprise docs for any fully air-gapped or on-premises control-plane options that may be available under separate enterprise agreements.

TrueFoundry §5 — K8s-Native In-Cluster Data Plane with Four-Hook Guardrails and Configurable Data Routing

Residency Architecture

TrueFoundry's AI Gateway data residency story is architecturally explicit about the synchronous hot path. The gateway plane architecture page at truefoundry.com/docs/platform/gateway-plane-architecture documents the request-handling principle directly: no external calls during request handling (unless cache is involved). Rate limits, load balancing, authentication, authorization, and built-in guardrails are evaluated in-memory, while logs and metrics are queued asynchronously. Request handling continues even if the external queue is down.

This is best understood as a residency-first hot-path design, not an unconditional guarantee. Prompt and response payloads are not sent to a vendor logging endpoint, remote policy engine, or SaaS analytics collector on the synchronous inference path. The payload exists in memory within the Kubernetes pod, transits to the LLM provider, and the response returns to the caller within the request lifecycle. Separately, current deployment docs describe request-response data export to an ingestion server on the TrueFoundry control plane in some deployment modes, after which the data may be stored in TrueFoundry-managed or customer-managed storage depending on configuration.

The OTEL export page at truefoundry.com/docs/ai-gateway/export-opentelemetry-data confirms that the gateway is OTEL compliant for traces and can export those traces to compatible observability platforms without disabling TrueFoundry-managed trace storage. That is valuable for customer-controlled observability, but it should not be conflated with universal direct log export. Current data-routing docs separately state that all metrics always go to the default destination, while custom destinations are traces-only. Kubernetes-native IAM patterns such as IRSA or Workload Identity remain relevant when the chosen customer-managed sink uses those identities.

TrueFoundry's current data-routing docs make the storage picture more specific than the changelog summary: you can configure the default destination for request-response data and metrics, but all metrics always go to the default destination, while custom destinations are traces-only. In parallel, deployment docs describe a mode where request-response data is exported to an ingestion server on the TrueFoundry control plane before being stored in either TrueFoundry-managed or customer-managed blob storage. The combined picture is strong but nuanced: customer-managed storage is supported, trace export to customer-controlled OTEL backends is explicitly documented, and the exact residency boundary depends on the deployment and routing mode you choose.

One caveat the docs explicitly acknowledge: two globally hosted services exist outside customer infrastructure — a licensing and authentication endpoint, and an analytics endpoint. The docs characterize these as "narrower in scope," but compliance teams should evaluate whether these endpoints constitute data processing under their applicable regulations. The gateway hot path itself (authentication, rate limits, routing) operates in-memory within the cluster; the global services are involved in licensing validation and aggregate analytics, not per-request prompt processing.

Four-Hook Guardrail System for Per-Application Payload Masking

TrueFoundry's guardrail system, documented at truefoundry.com/docs/ai-gateway/guardrails-overview, offers the most granular payload masking architecture of the four platforms. Four distinct hooks — LLM input, LLM output, MCP pre-tool, MCP post-tool — can each carry independent guardrail configurations. This is significant for data residency because different applications processing different data classes can be assigned different masking policies without a single global rule that over-masks some workloads and under-protects others.

The built-in PII/PHI detection at truefoundry.com/docs/ai-gateway/tfy-pii recommends explicit hook-placement mapping: redact PII from user prompts at LLM input, from model answers at LLM output, from tool parameters at MCP pre-tool, and from tool results at MCP post-tool. The built-in secrets detection at truefoundry.com/docs/ai-gateway/secrets-detection operates in either validate mode (block the request) or mutate mode (redact the credential in-place), with zero external dependencies — meaning the detection logic runs in-process within the gateway pod, not via an outbound API call.

The guardrail configuration at truefoundry.com/docs/ai-gateway/guardrails-configuration evaluates all matching rules and applies the union of their guardrails, using subject and target conditions to compose policies. This means a rule targeting team:finance-app with IBAN and credit card redaction can coexist alongside a rule targeting team:healthcare-app with PHI-specific hooks — and a request that matches multiple scopes inherits the combined policy set. The X-TFY-GUARDRAILS request header, documented on the headers page at truefoundry.com/docs/ai-gateway/headers, allows per-request dynamic guardrail selection for advanced use cases.

# TrueFoundry — per-team payload masking with four-hook guardrails
# Source: truefoundry.com/docs/ai-gateway/guardrails-configuration
#         truefoundry.com/docs/ai-gateway/guardrails-overview
name: data-residency-guardrails
type: gateway-guardrails-config
rules:
  # Finance team: IBAN, PII, secrets scrubbing on all four hooks
  - id: finance-pii-full-coverage
    when:
      subjects:
        operator: and
        conditions:
          in:
            - team:finance-app
    llm_input_guardrails:
      - my-guardrail-group/tfy-pii          # built-in PII/PHI — no external dep
      - my-guardrail-group/secrets-detection # built-in secrets — mutate mode
    llm_output_guardrails:
      - my-guardrail-group/tfy-pii
      - my-guardrail-group/secrets-detection
    mcp_tool_pre_invoke_guardrails:
      - my-guardrail-group/tfy-pii
    mcp_tool_post_invoke_guardrails:
      - my-guardrail-group/secrets-detection

  # Healthcare team: PHI detection via Azure PII (healthcare domain enabled)

    when:
      subjects:
        operator: and

          in:
            - team:healthcare-app
    llm_input_guardrails:

ℹ Note — Global Services Are a Narrow-Scope Caveat

TrueFoundry's architecture documentation explicitly acknowledges two globally hosted services outside customer infrastructure: a licensing/authentication endpoint and an analytics endpoint. The docs characterize these as "narrower in scope" compared to the control plane. The hot path — prompt content, routing, rate limits, guardrails — operates entirely in-memory within the cluster. For compliance reviews, these global endpoints should be documented as sub-processors with defined data scope (aggregate analytics, license validation), not as prompt-processing endpoints. Enterprise deployments can additionally self-host the control plane, documented at truefoundry.com/docs/platform/deployment-overview, to eliminate this dependency entirely.

Wins on deployment simplicity and zero cloud dependency. turn_off_message_logging: true plus operator-configured callbacks gives any team a clean residency baseline within hours, not days. PII masking requires Presidio integration — an operational dependency that must be scoped to the same residency zone.
Ideal for: Teams that need full stack sovereignty quickly, without Kubernetes, on Docker or ECS. Strong for internal tooling, regulated batch workloads, and organizations who can run Presidio in-region.
Kong AI Gateway
The Plugin-Composable Network Workhorse
Strongest data governance plugin catalog when enterprise features are available: bidirectional AI PII Sanitizer with restoration, Gen AI OTel attributes, standard Kong logging plugins routing to internal SIEMs, and token-level attribution. Version and license gating on the PII surface requires careful planning.
Ideal for: Existing Kong enterprise operators, high-throughput regulated workloads, and teams needing bidirectional prompt/response PII sanitization with model-transparent anonymization.
Portkey
The Composable Hybrid — Partial Sovereignty
Hybrid VPC mode genuinely keeps prompt payloads in the customer network and is well-documented. The hard constraint in hybrid mode is that control-plane sovereignty is not achievable: the dashboard, guardrail config UI, and analytics aggregation remain in Portkey's cloud by design. This is a defined, well-articulated architectural tradeoff — not a hidden gap.
Ideal for: Product and SaaS teams that can accept cloud-resident control plane, need fast deployment, and want GDPR data-localization (payload residency) without full infrastructure ownership.
TrueFoundry
The K8s-Native Sovereignty-First Platform
The strongest hot-path residency posture of the four: in-memory request handling, four-hook guardrails with composable per-team masking policies, built-in in-process PII/PHI and secrets detection, OTEL trace export to customer-controlled observability backends, and customer-managed storage options for request data in supported deployment modes. The K8s prerequisite is a real adoption barrier, and the exact storage boundary plus the two documented global services warrant explicit DPA review.
Ideal for: K8s-native platform teams in regulated industries (financial services, healthcare, government), enterprises requiring full stack sovereignty, and organizations building multi-tenant AI platforms with per-application data governance.

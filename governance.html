§0 — TL;DR & Quick-Picks

👻 Real-World Scenario: The Shadow AI Incident

It is a Thursday afternoon. Your security team discovers that three engineering squads have been routing production traffic through personal API keys to claude-opus-4-6 and gpt-5.4-pro — both legitimate frontier models, but neither approved by your organization for this regulated environment. They bypassed the enterprise gateway entirely, embedding raw provider keys in application environment variables. The prompts contain customer PII, internal database schemas, and proprietary business logic. None of this traffic was logged, rate-limited, or inspected by guardrails.

The damage inventory: 11 weeks of unmonitored production prompts containing regulated data sent to model endpoints outside organizational approval. No audit trail. No budget controls. No PII redaction. Your compliance officer is already drafting the incident report.

This is Shadow AI — the AI equivalent of shadow IT — and it is the inevitable result of deploying an AI gateway without governance enforcement. The question is not whether your gateway can proxy model calls. The question is: can it enforce which models are organizationally approved, who may call them, under what conditions, with what payload constraints, and with what metadata tagging — before the request ever reaches a provider?

§1 — Why Governance Is Specifically Hard for AI Gateways

Governance in the context of AI gateways is not simply access control. It spans both pre-deployment controls that validate workloads before they start, and runtime enforcement that fires synchronously on every request in the data path. Together, these cover five concerns:

  • Model endpoint ACL: Which caller identities may invoke which model endpoints? A contractor team should never reach gpt-5.4-pro in production. This must be enforced at the gateway, not trusted to application code.
  • Deep payload inspection: LLM prompts are unbounded JSON payloads. Unlike REST APIs with predictable schemas, a single /v1/chat/completions request can contain megabytes of conversation history, tool call schemas, and embedded documents. Governance must inspect this content — for PII, injection patterns, secrets, and policy violations — without degrading latency on the hot path.
  • Metadata-aware policy scoping: Different applications, teams, environments, and workload classes need different governance rules. A voice agent in production needs strict PII redaction; a batch summarization job in staging may tolerate relaxed guardrails. The gateway must route policy based on request metadata, not just caller identity.
  • Deployment and policy validation (pre-runtime): Before a service even starts sending requests, the deployment itself should be validated against organizational standards — required labels, auto-shutdown policies, environment restrictions, and approved model lists.
  • Audit and attribution: Every policy decision must be traceable — which rules fired, which guardrails were applied, and what the resolved model was — for compliance review.

LiteLLM §2 — Guardrail Pipeline with Budget-Aware Policy

Governance Architecture

LiteLLM's governance model is built around three control surfaces: virtual keys with model allowlists, a guardrail pipeline with named execution phases, and budget routing that makes economics a first-class policy signal. The architecture page at proxy/architecture documents that virtual-key validation and budget checks happen before the router — identity and economic controls precede routing decisions.

Model Endpoint ACL & Identity Scoping

Virtual keys are the primary identity and policy carrier. Each key can carry a models allowlist, a team_id for team budget consumption, and a max_budget for individual spend caps. Keys with a team assignment consume from the team budget rather than the personal budget, as documented at proxy/users. Router behavior is configurable per key or team at keys_teams_router_settings, making routing policy genuinely multi-tenant.

Guardrail Pipeline: pre_call / during_call / post_call

The guardrails quick-start page at proxy/guardrails/quick_start documents guardrails as YAML entries with a guardrail_name, provider parameters, and execution modes: pre_call, during_call, or post_call. This three-phase model lets operators place PII redaction before the model call, streaming-aware checks during generation, and output validation after completion. Provider integrations include Presidio, AIM, Aporia, and Lakera, plus a generic_guardrail_api entry for custom external services.

The pass-through guardrails page adds an important dimension: endpoint-specific guardrails compose with organization-, team-, and key-level guardrails. That means LiteLLM can govern not only LLM inference but also adjacent AI APIs (rerankers, OCR services) through the same policy framework.

Metadata Routing & Budget Policy

LiteLLM's metadata routing operates through the identity object populated by JWT claims (via team_alias_jwt_field) and virtual key attributes. Budget routing at provider_budget_routing attaches spend ceilings to providers, models, or tags — the router excludes exhausted targets from selection. Tag-based budgets (e.g., "product:chat-bot": "10/day") are the closest analog to metadata-scoped governance.

# LiteLLM — Guardrail pipeline with Presidio PII redaction

      guardrail: aim
      mode: [pre_call, post_call]

Kong §3 — Plugin-Composed Governance: Deterministic + Semantic + External

Governance Architecture

Kong's governance is plugin-composed: each security, inspection, and policy behavior is a distinct plugin attached to Services, Routes, Consumers, or Consumer Groups. This model means governance is not a single feature — it is the combination of AI Prompt Guard, AI Semantic Prompt Guard, AI PII Sanitizer, AI Azure Content Safety, AI AWS Guardrails, AI Rate Limiting Advanced, and AI Prompt Decorator, all composable in the same request path.

Deterministic vs. Semantic Payload Inspection

Kong offers two distinct approaches to prompt inspection. The AI Prompt Guard plugin provides deterministic regex-based allow/deny filtering where deny rules take precedence. The best-practices section documents Unicode bidirectional injection patterns — this is classical gateway security applied to AI traffic. The AI Semantic Prompt Guard plugin extends this to meaning-based allow/deny lists using embedding similarity, catching paraphrased attacks that regex cannot detect.

The AI PII Sanitizer plugin operates through an external PII service and supports request sanitization, response sanitization, or both, with optional restoration of original data in responses. The AI AWS Guardrails plugin integrates with AWS Bedrock Guardrails and exposes a response_buffer_size setting to control how streaming output is chunked for evaluation — a critical detail for streaming-first production workloads.

Consumer Group-Scoped Policy

Consumer Groups are Kong's primary mechanism for applying different governance rules to different caller classes on the same Route. A contractor Consumer Group can receive strict prompt deny patterns and tight token quotas, while an internal production group receives higher limits — all on the same Service/Route, differentiated by Consumer identity.

Metadata Routing & Prompt Shaping

Kong's metadata routing operates through standard Route-level path and header matching combined with Consumer Group assignment. The AI Prompt Decorator plugin allows platform teams to preset system prompts, prompt history, or phrase additions at the gateway level — centralizing prompt shaping as governance rather than leaving it to each application. This is a distinct capability: the gateway participates in prompt construction, not just prompt inspection.

# Kong decK — Layered governance: Prompt Guard + PII Sanitizer + Rate Limiting
# Sources: ai-prompt-guard, ai-sanitizer, ai-rate-limiting-advanced
_format_version: "3.0"

consumer_groups:
  - name: external-contractors
    plugins:
      - name: ai-prompt-guard
        config:
          deny_patterns:
            - "(\xE2\x80[\x8B-\x8D]|\xEF\xBB\xBF)"  # Unicode injection
            - "\xE2\x80[\xAA-\xAE]"
            - "(?i)internal.schema"
            - "(?i)database.credentials"
      - name: ai-rate-limiting-advanced
        config:
          llm_providers:
            - name: openai
              limit: [5000]
              window_size: [86400]

  - name: internal-production
    plugins:
      - name: ai-rate-limiting-advanced
        config:
          llm_providers:
            - name: openai
              limit: [100, 1000]
              window_size: [60, 3600]

# AI PII Sanitizer with external service
# Source: developer.konghq.com/plugins/ai-sanitizer

  - name: ai-sanitizer
    service: llm-service
    config:
      sanitize_request: true
      sanitize_response: true

📌 Kong Governance Summary

Model ACL: Governance scoped through standard gateway entities — Routes, Consumers, Consumer Groups. Payload inspection: Deterministic regex (AI Prompt Guard), semantic embedding (AI Semantic Prompt Guard), external PII service (AI PII Sanitizer), AWS Bedrock Guardrails, Azure Content Safety. Portkey §4 — Config-First Guardrails with Conditional Metadata Routing

Governance Architecture

Portkey treats governance as config-first: guardrails, routing conditions, and policy controls are all embedded in or attached to the config object. This design means a single JSON document can express a complete governance posture — which models are reachable, what guardrails fire, what metadata conditions apply, and what fallback behavior activates — and that entire policy can be attached as a default on an API key or passed per-request.

Guardrail Hooks & Actions

The guardrails system uses before_request_hooks and after_request_hooks as attachment points. Each hook can contain checks (partner guardrails, Portkey-native deterministic and LLM-based checks), and each check supports configurable actions: denial, async execution, and feedback annotation. The list of guardrail checks at list-of-guardrail-checks includes security-focused checks such as JWT validation and request-parameter validation, plus routing-oriented checks such as model whitelist enforcement.

Conditional Metadata Routing — MongoDB-Style Operators

Portkey's conditional routing at conditional-routing is, based on this review's public-doc analysis, the most flexible metadata routing surface among the four vendors. Conditions may reference metadata, params (request parameters like temperature, top_p), or the request URL path, using MongoDB-style query operators ($gt, $lt, $or, $and, $eq). Evaluation order matters — the first matching condition wins.

This means you can implement genuinely different governance schemas per application: a high-temperature creative writing workload routes to a permissive target with relaxed guardrails, while a structured extraction workload with temperature: 0 routes to a strict target with aggressive PII redaction — all expressed in a single config.

BYO Guardrails Webhook

The Bring Your Own Guardrails page at bring-your-own-guardrails documents a webhook contract: Portkey posts the full request body, extracted text, provider slug, request type, metadata, and event type to your endpoint. Your webhook returns a verdict and may optionally transform the request. This is an in-band extension point for teams with existing policy engines — the gateway delegates the policy decision to your infrastructure.

// Portkey — Conditional routing on request params + metadata
// Source: portkey.ai/docs/product/ai-gateway/conditional-routing
{
  "strategy": {
    "mode": "conditional",
    "conditions": [
      {
        "query": { "metadata.environment": { "$eq": "production" } },
        "then": "prod-strict-target"
      },
      {
        "query": {
          "$or": [
            { "params.temperature": { "$gt": 0.7 } },
            { "metadata.workload": { "$eq": "creative" } }
          ]
        },
        "then": "creative-permissive-target"
      }
    ],
    "default": "general-target"
  },
  "targets": [
    { "name": "prod-strict-target", "provider": "@openai-prod" },
    { "name": "creative-permissive-target", "provider": "@anthropic-prod" },
    { "name": "general-target", "provider": "@openai-sandbox" }
  ]
}

// Guardrail with before_request_hooks — regex deny + model whitelist
// Source: portkey.ai/docs/product/guardrails/creating-raw-guardrails-in-json
{
  "before_request_hooks": [{
    "type": "guardrail",
    "id": "production-policy",
    "checks": [{
      "id": "default.regexMatch",
      "parameters": { "regex": "(?i)(ssn|social.security|credit.card)" }
    }],
    "deny": true,
    "async": false,
    "on_fail": { "feedback": { "value": -1, "weight": 1 } }
  }]
}

Payload inspection: Regex guardrails, PII redaction (Pro tier, provider-backed + regex custom rules), BYO webhook for external policy engines. TrueFoundry §5 — Typed YAML Policy with Subject Conditions, Four-Hook Guardrails & Deployment Validation

Governance Architecture

TrueFoundry's governance is the most layered of the four. It operates at two distinct levels: deployment-time validation policies that enforce standards before a workload starts, and runtime gateway policies that enforce guardrails, rate limits, and budgets on every request. The gateway plane architecture page documents that the core enforcement path (auth, rate limits, load balancing, authorization) is evaluated in-memory on the hot path — with no external calls for those core checks — and async queueing for logs and metrics. Note that optional external guardrail integrations (e.g., Azure PII, Azure Content Safety) do introduce external service calls when configured.

Deployment-Time Validation Policies

The deployment-guardrails-and-policies page at applying-custom-policies documents executable validation code that runs during deployment. The ValidationInput includes the manifest and context — environment, creator, and active deployment — meaning policies can be sensitive to the resource, the target environment, the actor, and previous state simultaneously. The example enforces auto-shutdown for services:

// TrueFoundry — Deployment validation policy (TypeScript)
// Source: www.truefoundry.com/docs/applying-custom-policies
// Verified: ValidationInput includes manifest and context (environment, creator, active deployment)
import { ValidationInput, ValidationError } from '@src/types';

export function validate(validationInput: ValidationInput): void {
  const { manifest } = validationInput;
  if(manifest.type !== 'service') return;
  if (!manifest.auto_shutdown) {
    throw new ValidationError(
      'Auto shutdown is required for services'
    );
  }
}
Source: applying-custom-policies

This is a powerful governance primitive. Platform teams can encode environment-specific restrictions (e.g., no GPU-heavy deployments in dev, required labels for compliance, blocked model identifiers in staging) as executable code that runs before a workload ever starts. Combined with tfy apply --dry-run --show-diff for previewable changes, this creates a deployment governance workflow that is reviewable, testable, and version-controllable.

Four-Hook Guardrails with Subject Conditions

TrueFoundry's runtime guardrails are documented at guardrails-overview with four hooks: LLM input, LLM output, MCP pre-tool, and MCP post-tool. The MCP hooks are explicitly described as critical for agentic workflows where a model autonomously invokes tools. This four-hook model is unique among the four vendors evaluated.

The guardrails configuration page at guardrails-configuration uses typed YAML with first-match rule evaluation — only the first matching rule applies. Rules support subjects conditions (who is calling) and target conditions (what is being called), using operator: and with conditions.in arrays. Both subject and target conditions must match when both are present. This means governance policies can simultaneously depend on caller identity and request destination.

Metadata-Scoped Rate Limits & Budgets

The rate-limiting page at ratelimiting documents the rate_limit_applies_per field supporting ['model'] and ['metadata.project_id'] as explicit examples. The x-tfy-metadata request header (documented as stringified JSON with string key-value pairs) carries caller context into the gateway policy engine. This means rate limits and budgets can be segmented by any application-defined dimension — project, team, environment, feature flag, or customer tier — without requiring separate gateway configurations per segment.

Budget limiting at budgetlimiting uses the same pattern: budget_applies_per: ['metadata.project_id'] creates dynamically scoped budget instances from a single static rule.

Workspace-Environment Isolation

Workspaces and environments form the organizational scoping primitives. Deployments, secrets, and model access are workspace-scoped. The combination of workspace membership and environment defines what a workload may access at runtime. Secret groups with FQNs ensure provider credentials are consumed without raw values appearing in application manifests — the platform keeps credential management and workload configuration cleanly separated.

§7 — Editorial Verdict

The following assessments represent editorial judgment based on documented capabilities. "Well-suited for" is the authors' characterization, not a vendor claim.

LiteLLM
The Pragmatic Budget Governor
Three-phase guardrail pipeline with strong provider integrations (Presidio, AIM, Aporia, Lakera). Per-key model allowlists and team budget enforcement. Tag-based budget routing makes economics a governance signal. Custom Python auth hooks extend the policy surface. Gaps: no deployment-time validation policies, no MCP-specific guardrail hooks, and metadata routing is less granular than Portkey or TrueFoundry.
Well-suited for: Teams needing fast multi-provider governance with budget-aware enforcement and flexible guardrail provider choice.
Kong AI Gateway
The Plugin-Composed Security Stack
In this review, the widest content firewall surface among the four: deterministic regex, semantic embedding guards, external PII service with restoration, AWS Bedrock and Azure Content Safety integrations. Consumer Group-scoped policies compose cleanly. AI Prompt Decorator adds centralized prompt shaping. AI MCP Proxy adapts MCP traffic into standard HTTP so all Kong governance plugins can govern tool calls — extending the security surface to agentic workflows. Gaps: metadata-based policy scoping is less flexible than TrueFoundry or Portkey — routing conditions are Route/Consumer-level, not metadata-key-level.
Well-suited for: Enterprises with existing Kong infrastructure needing layered, plugin-composed governance with both deterministic and semantic payload inspection.
Portkey
The Metadata Routing Specialist
In this review, the most flexible conditional routing surface documented: MongoDB-style operators on request params, metadata, and URL path allow genuinely different governance schemas per application class.

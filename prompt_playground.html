§0 — TL;DR & Quick-Picks

🔐 Enterprise Scenario: The Air-Gapped Model Testing Problem

A quantitative risk team at a global asset manager has spent six months fine-tuning an internal credit-exposure model. The model runs inside a Tier-3 data center operating under strict regulatory controls: no inbound internet, strict L4 egress firewall rules, inference accessible only via mTLS from pre-approved internal service accounts. Compliance requires every system prompt template to pass a formal change-review board before it touches live position data.

The prompt engineering team's workflow is entirely manual: open a change ticket, wait 48 hours for a controlled test window, SSH through a bastion host, execute curl commands against an internal endpoint, and paste JSON output into a spreadsheet for comparison. One prompt iteration consumes three working days.

The team evaluates four AI gateways. The core question is not "which gateway supports the most providers" — that problem is already solved. The question is: which gateway gives prompt engineers a safe, governed, interactive testing surface that respects the fact that their model must never be publicly addressable? The answers vary dramatically — and two vendors now offer visual playgrounds while only one natively couples the playground to an in-VPC gateway data plane.

§1 — Why a Governed Prompt Playground Is an Infrastructure Problem

The distinction between a prompt development playground and a gateway-native testing surface matters operationally. A prompt development playground — like an IDE with multi-model comparison — lets prompt engineers craft, test, and version prompts efficiently against a wide range of cloud providers. It is an excellent product for prompt development velocity. A gateway-native playground goes a step further: it routes test traffic through the same policy stack that production traffic traverses, so the test result includes not just the model's response but proof of which guardrails fired, which routing decision was made, which model actually handled the request, and how long each stage took.

These are different tools solving adjacent problems. The choice between them depends on where your safety and compliance burden sits:

  • Guardrail validation in-loop. A prompt engineer testing PII detection against a production guardrail policy needs to know whether the guardrail is firing on their prompt, not just whether the model's output looks clean. That requires routing through the gateway stack during the test.
  • Silent fallback visibility. When a routing rule redirects a request from gpt-5.4 to claude-sonnet-4-6, output quality can change materially. The testing surface must expose which model actually handled the call, not just the logical alias.
  • Air-gapped model reachability. The most sensitive models sit behind the strictest network controls. A playground that requires cloud-accessible model endpoints is architecturally incompatible with the highest-value regulated use cases.
  • Prompt versioning as a production artifact. Prompts that pass review must be versioned and deployable, not ephemeral. Both the prompt development and gateway-native paths converge here — but via different mechanisms.
Testing topology for all four gateways. Kong has a Konnect cloud control plane but no prompt playground. Portkey and TrueFoundry both provide prompt playgrounds, with a key architectural difference: Portkey's studio connects to cloud providers, while TrueFoundry's playground routes through the in-VPC data plane, enabling air-gapped model testing. LiteLLM §2 — Admin UI and Management CLI as Testing Surfaces

What the Docs Say

LiteLLM's testing story begins and ends with its proxy. The Admin UI page documents a self-hosted web interface at /ui that can create virtual keys, track spend, and add models. This is an operations surface, not a prompt engineering surface. The public docs describe the UI as being for "creating keys, tracking spend, and adding models" — distinct from testing prompt behavior against a governed policy stack.

The closest analog to interactive prompt testing in LiteLLM is the management CLI, which can make chat and HTTP requests directly to the proxy. An engineer with CLI access can send test completions through the running gateway and observe responses. This is not a visual surface, but it routes through the same router, budget, and guardrail logic that production traffic traverses. The logging page notes that x-litellm-call-id is the most useful troubleshooting header, providing a stable identifier to correlate a test call with its callback and log events across components.

For air-gapped model testing specifically, LiteLLM's self-hosted model is both its strength and its constraint. Because the proxy, Admin UI, and all state (Redis, Postgres) are in the same environment as the model, there is no internet exposure of any inference path. However, the public docs reviewed center on a self-hosted proxy/Admin UI rather than a Konnect-style cloud control plane, so prompt engineers typically need private network access such as VPN or bastion access to reach the Admin UI in private deployments, eliminating much of the ergonomic benefit a testing surface is supposed to provide.

⚠ Limitation — No Dedicated Prompt Testing Playground

LiteLLM's Admin UI is documented for key management, spend tracking, and model registration. The management CLI routes chat requests through the proxy. There is no interactive visual playground for prompt iteration, guardrail testing, or prompt versioning in the public docs reviewed, and those docs center on a self-hosted proxy/Admin UI rather than a Konnect-style cloud control plane.

Kong §3 — Konnect Cloud Control Plane, Developer Tool Governance, No Playground

Architecture

Kong's deployment posture differs meaningfully from LiteLLM. Konnect provides a unified cloud control plane that manages configuration, plugins, and policies, with data plane nodes deployable self-hosted, on Kubernetes, or as fully managed cloud gateways. The quickstart provisions a Konnect control plane and exports both a control-plane URL (KONNECT_CONTROL_PLANE_URL) and a proxy URL (KONNECT_PROXY_URL), making the separation between management surface and inference path explicit from the first five minutes of use. Kong also supports fully self-managed deployments — the cloud control plane is an option, not a requirement.

However, despite having a cloud control plane, Kong has no dedicated visual prompt playground. Kong's approach to developer testing is through AI CLI proxying: routing requests from AI command-line tools — including Claude Code, Codex CLI, Qwen Code CLI, and Gemini CLI — through the gateway so that auth, rate limiting, audit logging, and semantic routing all apply to developer test traffic. For teams that already use these AI coding assistants, Kong becomes the governed conduit without requiring any change to the engineer's existing workflow.

Konnect also provides visualized LLM metrics and a visual traffic map that shows how requests flow between clients and models in real time. This is a monitoring and governance surface, not a prompt iteration surface — but it gives Kong users more operational visibility into test traffic than any purely self-hosted solution.

⚠ Limitation — No Visual Playground, No Prompt Version Management

Kong has a cloud control plane (Konnect) with analytics and LLM metrics, but no interactive prompt testing, no per-request guardrail experimentation UI, and no prompt versioning. Kong is a network control plane first. For organisations that govern AI CLI developer tools, this is a strong fit; for teams that need structured prompt iteration UX, external tooling is needed.

Portkey §4 — Full Prompt Engineering Studio: Playground, Versioning, Prompt API

Architecture

Portkey has invested significantly in prompt tooling. The Prompt Engineering Studio (accessible at prompt.new) is a full-featured prompt development environment offering: an interactive Prompt Playground, prompt versioning with labeled deployments and rollbacks, a shared prompt library with team access controls, prompt partials for reusable components, prompt observability, and a Prompt API for integrating saved prompts into application code. This is production-grade tooling built for prompt development at scale.

The Prompt Playground is a complete Prompt Engineering IDE. It supports: running prompts on any LLM (1600+ cloud models), side-by-side model comparison in parallel, multimodal input (text and images), tool integration, Mustache-based prompt templates, and AI-assisted prompt improvements. Once a prompt passes review in the playground, it is saved to the prompt library and deployed through the Prompt API — which then routes calls through Portkey's gateway stack with the full config-first routing, fallback, retry, and guardrail machinery applied.

This creates an important architectural distinction: Portkey's Prompt Engineering Studio is a prompt development environment that connects to cloud providers directly for testing, then deploys via the Portkey gateway for production. The gateway's guardrail policies, routing rules, and fallback chains are applied when the deployed prompt is invoked via the Prompt API in production, not necessarily during the playground test session itself. This is a deliberate product choice — it prioritizes development velocity (test across 1600+ models instantly) over gateway-policy-in-the-loop testing.

Prompt Versioning and Prompt API

Prompt versioning is a first-class feature: every update to a saved prompt creates a new version, and the docs explicitly state that you can switch back to any older version. Labeled deployments allow stable versions to be promoted to production while experimental versions continue in development. The Prompt API provides Completions and Render endpoints, making it straightforward to integrate versioned, tested prompts into application code.

Hybrid VPC and Air-Gapped Models

For infrastructure deployment, Portkey's hybrid VPC deployment runs the data plane gateway pod inside the customer's network via Helm, with prompt and response payloads staying in the customer's VPC while only metrics leave. The docs' Self-Hosting section also includes a separate Air-Gapped Deployments track. The key nuance is architectural: Prompt Engineering Studio at prompt.new is a SaaS development surface for testing against cloud providers, while testing private or air-gapped model endpoints requires routing through a self-hosted gateway path that the client can actually reach.

ℹ Key Architectural Nuance — Studio vs. Gateway Testing

Portkey's Prompt Engineering Studio is a powerful prompt development environment. The crucial distinction for governed use cases: when testing in the playground, calls go directly to cloud providers (testing for model output quality, prompt format, versioning). When prompts are deployed via the Prompt API, the Portkey gateway applies routing, guardrails, retries, and caching. These are two separate phases of the prompt lifecycle, which is a deliberate product choice optimizing for development velocity.

TrueFoundry §5 — Gateway-Native Playground: Guardrails, Routing, and Air-Gapped Models in One Loop

Architecture

TrueFoundry's Playground UI is architecturally distinct from Portkey's Prompt Engineering Studio in one critical dimension: test traffic routes through the same in-VPC gateway data plane that handles production traffic. The Playground overview positions the playground as the third step in the production onboarding flow — add providers, validate behavior in the playground, copy generated code snippets — rather than as a separate pre-gateway development environment.

The split-plane design enables this. The cloud control plane hosts the playground UI and distributes configuration. The stateless Hono gateway plane runs inside the customer's Kubernetes cluster, subscribing to configuration updates via queue with HTTP fallback. The gateway plane performs all hot-path evaluations in-memory — authentication, rate limits, routing, guardrails — with async queueing for logs and metrics. This means that when a prompt engineer submits a test from the cloud playground UI, the request flows through the in-VPC gateway plane before reaching the model. Guardrails fire. Routing rules apply. Fallbacks can trigger. The model receives no request from outside the network boundary.

It is worth being precise about data flows: the gateway plane maintains outbound connectivity to the control plane for configuration sync, and in hosted-control-plane deployments request-response data can also be exported to a control-plane ingestion service and backing bucket for logs and analytics. What does not require public exposure is the model endpoint itself — the hot path from gateway plane to model stays inside the VPC, so the model receives no inbound request from the public internet.

Per-Request Guardrail Testing Without Global Policy Change

The X-TFY-GUARDRAILS header accepts a four-array JSON object covering all hook points: llm_input_guardrails, llm_output_guardrails, mcp_tool_pre_invoke_guardrails, and mcp_tool_post_invoke_guardrails. A prompt engineer can attach or remove guardrails on a per-request basis from the playground without touching global gateway configuration — exactly the isolation needed for safe A/B testing of safety policies.

The response headers close the feedback loop directly in the playground: x-tfy-resolved-model exposes which physical model actually handled the call (exposing silent fallbacks), x-tfy-applied-configurations lists every control surface that fired, and the server-timing header breaks total latency into per-stage milliseconds across authentication, input guardrails, model call, output guardrails, and logging.

# TrueFoundry — per-request guardrail override via X-TFY-GUARDRAILS header
# Source: truefoundry.com/docs/ai-gateway/guardrails-getting-started
#         truefoundry.com/docs/ai-gateway/headers
import json, requests

guardrail_config = {
    "llm_input_guardrails":  ["testing-group/pii-detector"],
    "llm_output_guardrails": ["testing-group/secrets-scan"],
    "mcp_tool_pre_invoke_guardrails":  [],
    "mcp_tool_post_invoke_guardrails": []
}

resp = requests.post(
    "https://gateway.your-org.internal/openai/chat/completions",
    # ↑ internal cluster DNS — model unreachable from public internet
    headers={
        "Authorization":  "Bearer your-api-key",
        "x-tfy-metadata": json.dumps({"environment": "staging", "test_run": "pr-1234"}),
        "x-tfy-guardrails": json.dumps(guardrail_config)
    },
    json={
        "model": "internal-risk-scorer",  # logical virtual model alias
        "messages": [{"role": "user", "content": "SSN: 123-45-6789. Summarise."}]
    }
)

# Response headers reveal exactly what fired in the policy stack:
print(resp.headers.get("x-tfy-resolved-model"))
# e.g. → "anthropic/claude-sonnet-4-6"  (fallback rule fired silently)

print(resp.headers.get("x-tfy-applied-configurations"))
# e.g. → "guardrails-control, ratelimiting-config, air-gapped-routing"

print(resp.headers.get("server-timing"))
# e.g. → auth;dur=0.9, llm-in-guard;dur=0.7, model-call;dur=1350, llm-out-guard;dur=722, log;dur=1.1
Source: truefoundry.com/docs/ai-gateway/guardrails-getting-started · truefoundry.com/docs/ai-gateway/headers
# TrueFoundry routing config sketch — valid doc schema for an air-gapped internal target
# Source: truefoundry.com/docs/ai-gateway/load-balancing-configuration
#         truefoundry.com/docs/ai-gateway/virtual-model
name: air-gapped-routing
type: gateway-load-balancing-config
rules:
  - id: "risk-model-staging"
    type: "weight-based-routing"
    when:
      subjects:
        - "team:quant-research"
      models:
        - "internal-risk-scorer"
      metadata:
        environment: "staging"
    load_balance_targets:
      - target: "internal/risk-model-v2"
        weight: 100
        retry_config:
          attempts: 1
          # Gateway plane resolves this via internal cluster DNS
          # Model endpoint has no public port
Source: truefoundry.com/docs/ai-gateway/load-balancing-configuration · truefoundry.com/docs/ai-gateway/virtual-model

⚠ Limitation — Kubernetes Operational Prerequisite

The deploy-gateway-plane guide requires a running Kubernetes cluster, a domain, a TLS certificate, image-pull credentials, and a control-plane API key before any test traffic can flow through the in-VPC gateway plane. For teams without a dedicated Kubernetes platform function, this is a significant adoption barrier. The air-gapped testing capability is architecturally elegant; the operational prerequisite is substantial.

§7 — Editorial Verdict

Editorial assessments as of March 10, 2026.

LiteLLM
The Operations Console
LiteLLM's Admin UI excels at key management, spend visibility, and model registration. Testing happens through the management CLI or direct API calls — routed through the real proxy policy stack, but without a visual surface. The public docs reviewed center on a self-hosted proxy/Admin UI, so access in private deployments typically requires private network connectivity such as VPN or bastion. For teams comfortable in a terminal, this is workable; for structured prompt engineering workflows, external tooling is required.
Ideal for: Platform ops teams that manage the gateway programmatically. Not suited as a standalone prompt engineering workspace.
Kong AI Gateway
The Developer Tool Enforcer
Kong's Konnect cloud control plane provides LLM metrics, visual traffic maps, and decK-based configuration management. Its distinct contribution to the testing problem is AI CLI proxying: governing Claude Code, Codex CLI, Qwen Code CLI, and Gemini CLI traffic through the same plugin pipeline as production. For organisations where AI coding assistants are the primary test surface, this is the cleanest fit. For teams needing structured prompt iteration UI, Kong is deliberately out of scope.
Ideal for: Organisations governing AI developer tool usage at scale, already invested in the Kong/Konnect ecosystem.
Portkey
The Prompt Development Powerhouse
Portkey's Prompt Engineering Studio is the most complete prompt development environment in this comparison. Side-by-side model comparison across 1600+ models, versioning with labeled deployments, collaborative prompt libraries, Mustache templates, and the Prompt API make it the strongest choice for teams whose primary workflow is prompt iteration and cross-model evaluation. The key nuance: studio testing goes to cloud providers directly; gateway guardrails and routing apply on Prompt API production calls. Excellent for prompt development; the hybrid VPC path addresses data residency for inference; air-gapped model testing via the studio requires a reachable self-hosted gateway path.
Ideal for: Product engineering teams needing rapid cross-model prompt iteration, versioning, and a clean path from playground to production via Prompt API.
TrueFoundry
The Gateway-Native Testing Platform
TrueFoundry's playground is architecturally distinct: test traffic routes through the same in-VPC gateway plane as production. Guardrails fire during the test. The x-tfy-resolved-model header exposes silent fallbacks. server-timing shows per-stage latency. Per-request X-TFY-GUARDRAILS overrides enable safe A/B testing of safety policies. Code snippets generated from the playground reflect the exact gateway configuration tested. The cloud UI + stateless in-VPC data plane architecture makes TrueFoundry the most direct fit in this comparison for testing models whose endpoints are not publicly exposed.
Ideal for: K8s-native platform teams building governed, interactive prompt engineering workflows for air-gapped models, regulated industries, or agentic/MCP workloads requiring policy-in-the-loop testing.

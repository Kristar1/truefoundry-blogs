§0 — TL;DR & Quick-Picks

🔍 Real-World Scenario: The Invisible 15-Second Bottleneck

Your production RAG pipeline takes roughly 15 seconds end-to-end. On a Thursday afternoon, a subset of enterprise users starts reporting incomplete answers — not wrong, not errored, just suspiciously shallow. Your monitoring dashboard shows no 5xx spikes, latency p99 looks flat, and the LLM provider's status page is green.

The pipeline has five distinct stages: query rewriting, dense retrieval against a vector index, a re-ranking call, a synthesis call to claude-sonnet-4-6, and a post-processing step. Without traces that span the full chain, you're left grepping through flat log files trying to reconstruct causality from fragments.

This is the standard operating condition for any team running non-trivial LLM workloads in production. An AI gateway that doubles as a first-class observability layer fundamentally changes how quickly your team can isolate and fix these failures. But the four gateways take structurally different approaches to how telemetry is collected, where it goes, and whether it ever leaves your network.

Quick-Pick

§1 — Why the Gateway Is the Right Instrumentation Point

LLM providers expose minimal telemetry. An /v1/chat/completions response contains token counts and a finish reason — nothing you'd want in a trace. The gateway sits at the single choke-point where every LLM call passes. That positional advantage means it can:

  • Capture request and response payloads before they reach application code, without modifying application code.
  • Inject and propagate W3C trace context headers so gateway spans nest correctly inside upstream application traces.
  • Compute derived metrics — cost per token, tokens-per-second, cache hit rate — that neither the provider nor the application naturally surfaces.
  • Route telemetry independently of application telemetry, so a noisy dev environment doesn't contaminate your production SIEM.
⚠ Data Residency Implication

Gateway-layer logging captures full prompt and completion text. For regulated industries (HIPAA, PCI-DSS, GDPR), the export destination of that telemetry is a primary architectural constraint — one where the four gateways diverge sharply, as we will see.

The AI gateway sits at the single choke-point where every LLM call passes — the natural instrumentation layer for prompt content, token accounting, and distributed trace context.
LiteLLM §2 — Callback-Driven Polyglot Logging

LiteLLM's observability model is built around callbacks: hooks that can run synchronously or asynchronously on success, failure, and other lifecycle events. Each callback receives request kwargs, response details, timing, and computed metadata such as token usage and cost. That makes LiteLLM less of a bundled observability product and more of a flexible telemetry router for teams that already know where they want their traces, logs, and metrics to land.

OpenTelemetry Export

LiteLLM ships a first-class OpenTelemetry callback that wraps LLM calls in OTLP spans carrying GenAI semantic-convention fields. Current docs support both OTLP/gRPC and OTLP/HTTP exporters, and the same request can fan out to other backends such as Langfuse or LangSmith through callback configuration. The key design point is flexibility: LiteLLM gives you spans, but you decide whether they go to a collector, a SaaS tool, or multiple places at once.

# litellm_config.yaml — OpenTelemetry + Langfuse fan-out
model_list:
  - model_name: gpt-5
    litellm_params:
      model: openai/gpt-5
      api_key: os.environ/OPENAI_API_KEY
  - model_name: claude-sonnet-4-6
    litellm_params:
      model: anthropic/claude-sonnet-4-6
      api_key: os.environ/ANTHROPIC_API_KEY

litellm_settings:
  callbacks: ["otel"]
  success_callback: ["langfuse"]
  failure_callback: ["langfuse"]
  service_callbacks: []
  turn_off_message_logging: false

environment_variables:
  OTEL_EXPORTER_OTLP_ENDPOINT: "http://otel-collector.monitoring.svc:4317"
  LANGFUSE_HOST: "https://cloud.langfuse.com"
Source: docs.litellm.ai/docs/proxy/logging · opentelemetry_integration

Prometheus / Grafana Metrics

Adding "prometheus" to the callback list enables a /metrics endpoint on the proxy port. LiteLLM's current docs focus on Prometheus / Grafana as another callback destination and document example series such as failed-logging counters; they are less stable about the exact metric names than the original draft implied. The safe claim is that Prometheus / Grafana scraping is supported, while dashboards should be built from the metric series exposed by the version you are running.

# Multi-backend fan-out: OTel + Prometheus / Grafana + S3 archival
litellm_settings:
  callbacks: ["otel", "prometheus"]
  success_callback: ["langfuse", "s3"]
  failure_callback: ["langfuse", "sentry"]

environment_variables:
  AWS_BUCKET_NAME: "my-llm-logs-prod"

Custom Async Logger (Bring-Your-Own SIEM)

import aiohttp
from litellm.integrations.custom_logger import CustomLogger

class SIEMLogger(CustomLogger):
    def __init__(self, endpoint: str, timeout_seconds: float = 5.0):
        self.endpoint = endpoint
        self.timeout = aiohttp.ClientTimeout(total=timeout_seconds)

    async def async_log_success_event(self, kwargs, response_obj, start_time, end_time):
        payload = kwargs.get("standard_logging_object", {})
        event = {
            "request_id": payload.get("id"),
            "trace_id": payload.get("trace_id"),
            "call_type": payload.get("call_type"),
            "model": payload.get("model"),
            "response_cost": payload.get("response_cost"),
            "prompt_tokens": payload.get("prompt_tokens"),
            "completion_tokens": payload.get("completion_tokens"),
            "status": payload.get("status"),
        }
        async with aiohttp.ClientSession(timeout=self.timeout) as session:
            async with session.post(self.endpoint, json=event) as resp:
                resp.raise_for_status()
Source: docs.litellm.ai/docs/observability/custom_callback

LiteLLM's documented logging integrations include Langfuse, LangSmith, Helicone, Traceloop, Arize Phoenix, Datadog, Prometheus / Grafana, OpenTelemetry, S3, GCS, Sentry, and a generic custom_logger hook. The practical implication is that LiteLLM is usually strongest when it plugs into an observability system you already operate, rather than when you expect it to be the observability backend itself.

⚠ Operational Caveat

The proxy's built-in UI (:4000/ui) provides basic spend dashboards but is not a full trace viewer. For rich trace analysis, an external backend (Langfuse, Jaeger, etc.) is required. Misconfiguring async failure callbacks can silently drop events — validate against a known failure path in staging.

Kong §3 — Plugin-Composable Observability

Kong's observability model is inherited from its API gateway lineage: every cross-cutting concern is a plugin. For LLM observability, you compose a telemetry pipeline by stacking opentelemetry, prometheus, and one or more logging plugins on a route that is already handled by ai-proxy or ai-proxy-advanced. The upside is consistency with the rest of a Kong-based platform; the caveat is that some AI metrics depend on more than one knob being enabled at the same time.

OpenTelemetry Plugin

Kong's opentelemetry plugin exports OTLP spans for proxied requests. When combined with ai-proxy, those spans can carry GenAI-specific attributes. The code sample below stays deliberately conservative: it uses the currently documented trace endpoint field and leaves version-specific sampler or propagation knobs out of the snippet unless you have checked them against the exact Kong version you deploy.

# kong.yaml — AI metrics + tracing on an LLM route
_format_version: "3.0"

services:
  - name: llm-service
    url: https://api.openai.com
    routes:
      - name: llm-route
        paths:
          - /v1/chat/completions

plugins:
  - name: ai-proxy
    service: llm-service
    config:
      route_type: llm/v1/chat
      auth:
        header_name: Authorization
        header_value: Bearer ${OPENAI_API_KEY}
      model:
        provider: openai
        name: gpt-5
      logging:
        log_statistics: true

  - name: prometheus
    service: llm-service
    config:
      ai_metrics: true

  - name: opentelemetry
    service: llm-service
    config:
      traces_endpoint: http://otel-collector.monitoring.svc:4318/v1/traces
Source: plugins/opentelemetry · monitor-ai-llm-metrics · llm-open-telemetry

AI Token & Cost Metrics via Prometheus / Grafana

Kong's AI Prometheus metrics require two pieces of configuration together: config.ai_metrics: true on the Prometheus plugin and config.logging.log_statistics: true on ai-proxy or ai-proxy-advanced. With both enabled, Kong documents AI-specific series including request counts, token counts, provider latency histograms, and total AI cost. That makes Kong stronger on native cost visibility than the earlier draft suggested.

# PromQL examples for Kong AI dashboards

ℹ Kong Prompt Body Logging

Prompt and response body logging in Kong follows the standard logging-plugin path (http-log, file-log, kafka-log) rather than a dedicated AI-specific plugin. Careful payload redaction strategy is required to avoid logging sensitive user data verbatim — this is not automatic. Consult your Kong version's logging plugin documentation for body-capture configuration options.

Kong's AI observability is a plugin composition. AI Prometheus metrics require both ai_metrics: true on the Prometheus plugin and logging.log_statistics: true on the AI Proxy plugin; cost metrics additionally depend on pricing being configured for the routed model. Portkey §4 — Analytics-First SaaS Observability

Portkey bakes the observability dashboard into the gateway product itself. Every request is automatically logged, traced, and surfaced in a web UI with near-zero configuration. For teams that need data locality, Portkey documents a hybrid deployment where the data plane runs in your VPC, LLM payloads stay on that data plane, and Portkey-managed dashboards plus control-plane metadata remain SaaS-side. Enterprise docs also expose Prometheus metrics, direct Langfuse/LangSmith integrations, and OTEL-style export paths.

Headers, Metadata, and Automatic Tracing

Portkey's observability story starts at the request layer: API-key plus provider/config headers determine routing, while metadata and trace headers can be attached directly to requests. Current docs support W3C trace context, Portkey trace headers, metadata tagging, multi-step trace trees, and ingestion of broader application OpenTelemetry signals into the Portkey observability surface. For attribution, the safer framing is metadata and request-level identifiers rather than the older Virtual Keys terminology, which Portkey now documents as deprecated in favor of Model Catalog / provider-based configuration.

# Python — multi-span RAG pipeline: all steps under one trace_id
import openai
from portkey_ai import PORTKEY_GATEWAY_URL, createHeaders

TRACE_ID = "rag-session-abc-001"

def get_portkey_client(stage: str):
    return openai.OpenAI(
        api_key="dummy",
        base_url=PORTKEY_GATEWAY_URL,
        default_headers=createHeaders(
            api_key="pk-...",
            virtual_key="openai-prod-vk",
            trace_id=TRACE_ID,
            metadata={
                "pipeline_stage": stage,
                "_user":          "user-789",
                "environment":    "production",
                "app":            "rag-pipeline-v2",
            }
        )
    )

# Step 1: Query rewriting — logged as span 1 under TRACE_ID
rewrite_client = get_portkey_client("query-rewrite")
rewritten = rewrite_client.chat.completions.create(
    model="gpt-5-mini",
    messages=[{"role":"user","content":f"Rewrite for retrieval: {user_query}"}],
)

# Step 2: Synthesis — logged as span 2, same trace
synth_client = get_portkey_client("synthesis")
response = synth_client.chat.completions.create(
    model="claude-sonnet-4-6",
    messages=[
        {"role":"system", "content":"Answer using only the provided context."},
        {"role":"user",   "content":f"Context: {retrieved_docs}\n\nQuery: {user_query}"},
    ],
)
Source: portkey.ai/docs/product/observability/traces · observability/logs

⚠ Data Residency — Standard vs Hybrid Plans

On standard SaaS tiers, Portkey stores observability data in Portkey-managed infrastructure. The documented hybrid deployment keeps the LLM data plane — including prompt and response payloads — in your VPC. Portkey's own enterprise architecture docs still describe a Portkey-managed control plane that hosts the dashboard and stores non-sensitive metadata plus aggregated metrics, so avoid over-claiming this as a fully self-hosted observability stack.

Portkey auto-logs every request. In hybrid deployment, AI traffic stays on the VPC data plane while Portkey-managed dashboards, non-sensitive metadata, and aggregated metrics remain control-plane managed. TrueFoundry §5 — Built-In Monitor, Trace Export & Deployment-Dependent Data Paths

TrueFoundry's AI Gateway supports multiple deployment topologies — including gateway-only SaaS, gateway-plane-only, managed combinations, and fully self-hosted. In current docs, observability is not just a bring-your-own-backend story: TrueFoundry documents a built-in Metrics Dashboard, built-in trace viewing in Monitor, configurable request-log / trace retention, custom metadata tagging, native feedback attachment on traces, optional OpenTelemetry trace export, and Prometheus/Grafana integration for supported customer-hosted deployments. Which pieces you can use, and where the resulting telemetry lands, depends on deployment mode and data-routing configuration.

🔒 Telemetry Data Path — Deployment-Mode Dependent

TrueFoundry's broader platform architecture includes globally hosted services such as licensing/authentication and analytics outside customer infrastructure. Separately, the AI Gateway docs now let you configure where request logs (traces) and metrics are stored. The default destination is always enabled; custom destinations are Enterprise-only and currently apply to traces, while metrics continue to route to the default destination. Treat telemetry residency as a deployment-and-routing decision, not as a universal product guarantee.

OTEL Export Configuration

TrueFoundry documents OpenTelemetry-compliant tracing export for the AI Gateway. Two details matter: the export is specifically for traces, and the exported traces can coexist with the traces already stored and viewable inside TrueFoundry Monitor. Prometheus / Grafana-style metrics are documented separately, and current data-routing docs make clear that metrics and traces do not follow identical routing rules.

# Optional customer-managed trace export path for TrueFoundry
# Use OTLP for traces; handle metrics separately via Prometheus / Grafana-compatible scraping.

receivers:
  otlp:
    protocols:
      grpc:
      http:

processors:
  batch: {}

exporters:
  otlp/tempo:
    endpoint: http://tempo.monitoring.svc:4317
    tls:
      insecure: true

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/tempo]
Source: Standard OpenTelemetry Collector config — exporters are vendor-neutral and documented at opentelemetry.io/docs/collector · TrueFoundry OTEL export doc

The collector example above is intentionally narrow: it shows one optional trace-export topology, not the whole TrueFoundry observability story. Current docs separately describe built-in trace viewing, a built-in metrics dashboard, Prometheus-compatible metrics for supported customer-hosted gateways, and routing rules where traces and metrics can follow different destinations.

Logs, Retention, Metadata, Feedback, and Alerting

TrueFoundry treats request logs as traces. Current docs let operators control request tracing, query those traces in Monitor or via API, and configure where traces and metrics are stored. The important nuance is that traces and metrics do not follow the same routing rules: trace retention is configurable, custom trace destinations are an enterprise feature, and metrics continue to route to the default destination.

Two higher-level observability primitives are especially relevant for platform teams. First, X-TFY-METADATA lets you attach custom key/value metadata that can be used for filtering, custom metrics pages, and conditional routing rules. Second, every gateway response can return x-tfy-feedback-target-id, which lets product or QA workflows attach human feedback directly to the root trace. For customer-hosted gateways, TrueFoundry separately documents Prometheus/Grafana-based alerting plus export guides for monitoring platforms such as Elastic, New Relic, Dynatrace, and Last9.

ℹ Why In-Cluster OTEL Matters for Regulated Industries

In HIPAA-covered environments or financial services under SOC2/ISO 27001, keeping prompt and completion text within audited network boundaries is a compliance requirement. For teams on self-hosted gateway-plane deployments, TrueFoundry's OTLP export to in-cluster backends can satisfy this — but confirm your specific deployment topology against TrueFoundry's deployment documentation before treating this as a guarantee.

TrueFoundry combines built-in Monitor views with configurable trace routing, retention, metadata, feedback-on-trace workflows, and customer-managed metrics / alerting paths. Request logs are exposed as traces, and routing remains deployment- and destination-dependent.

§6 — Trace Dashboard Flexibility Across Platforms

All four gateways can ultimately feed Prometheus metrics, but the setup effort and out-of-box experience varies significantly. A compliance team needs audit-oriented log tables. A product team needs per-user cost heatmaps. An ML team needs prompt version performance comparisons.

Grafana + Prometheus / Grafana

All four gateways can feed metrics into a Prometheus / Grafana-style workflow, but the operator experience differs. LiteLLM and Kong are classic bring-your-own-dashboard products. Portkey has a built-in dashboard and also documents Prometheus metrics for enterprise/self-hosted deployments. TrueFoundry now sits in the middle: it documents a built-in Metrics Dashboard in Monitor, while also exposing Prometheus-compatible metrics and Grafana integration for supported customer-hosted gateway deployments.

Langfuse / Observability Ecosystems

LiteLLM documents native Langfuse and LangSmith callbacks. Portkey now documents direct Langfuse and LangSmith integrations, plus enterprise OTEL / complete-log export paths for broader observability ecosystems. TrueFoundry is better described as OTEL-friendly than as a native Langfuse product: you can forward traces to external tools through standard export pipelines. Kong remains the most custom here — you bridge to external tools through OTEL or logging plugins rather than a dedicated Langfuse/LangSmith feature.

Jaeger / Tempo / Jaeger (OTel-native)

LiteLLM, Kong, and TrueFoundry all document OTLP-friendly tracing paths that can be visualized in Jaeger or Tempo / Jaeger. Portkey still centers its own trace UI, but enterprise docs now also describe OTEL-style export and complete-log export options for self-hosted / enterprise use cases. So the accurate distinction is no longer “Portkey cannot”; it is “Portkey's built-in UI is primary, while self-managed backends are an enterprise/export-path story.”

§8 — Editorial Verdict

Editorial assessments as of March 25, 2026.

LiteLLM
The Polyglot Logger
LiteLLM's callback architecture is the most flexible. Twenty-plus documented integrations, support for synchronous or asynchronous callback patterns, and a clean base class for bring-your-own destinations mean almost any observability stack can be wired in. The cost is operational discipline: each integration needs validation, and the built-in UI remains deliberately minimal.
Ideal for: Teams that need maximum integration flexibility and are comfortable managing observability backends. Strong choice when Langfuse or LangSmith is already in the stack.
Kong AI Gateway
The Enterprise Composer
Kong's plugin composability is genuinely powerful for organisations already running Kong as their API gateway — LLM observability joins the same OTel/Prometheus / Grafana infrastructure your REST services already use. But the details matter: native AI cost and token metrics only appear when ai_metrics: true is enabled on Prometheus / Grafana and statistics logging is enabled on the AI Proxy plugin. Prompt body capture still follows the standard logging-plugin path and needs deliberate redaction strategy.
Ideal for: Organisations with an existing Kong deployment where LLM routes need to join an established API observability program.
Portkey
The Analytics-First Gateway
Portkey still delivers the lowest time-to-insight: a production-grade LLM observability dashboard requiring near-zero configuration. Hybrid deployment keeps LLM payloads on the VPC data plane while the Portkey-managed dashboard and control-plane metadata remain SaaS-side. Compared with the earlier draft, the key correction is that Portkey's enterprise story is broader than just the built-in UI: current docs also include direct Langfuse/LangSmith integrations and enterprise export paths.
Ideal for: Product teams that prioritise observability UX and time-to-insight; Enterprise teams using the hybrid data plane for payload residency.
TrueFoundry
The Built-In Monitor + Export Hybrid
TrueFoundry is no longer best described as only a bring-your-own observability stack. Current docs show built-in Monitor trace views, a built-in Metrics Dashboard, optional OTEL trace export, and Prometheus/Grafana integration for supported customer-hosted gateways. The caveat remains important: deployment mode and data-routing choices determine where telemetry lands, and traces and metrics do not share identical routing semantics.
Ideal for: Teams on self-hosted gateway-plane deployments with existing Prometheus/Grafana stacks, especially in regulated industries where payload residency requirements are strict.

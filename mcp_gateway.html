§0 — TL;DR & Quick-Picks

🔐 Enterprise Scenario: The HR Assistant Credential Leak

A large financial institution deploys an AI HR assistant powered by claude-sonnet-4-6. The assistant handles benefit queries from 6,000 employees and is wired to an internal PostgreSQL HR database via a Model Context Protocol tool: query_employee_record(employee_id, fields). The connection string — postgresql://hrapp:V@ult#2026@pg-prod.internal:5432/hr_db — lives in the system prompt so the LLM can construct the tool call.

Six weeks after launch, the security team's routine audit surfaces three problems simultaneously. First, a log aggregation pipeline is capturing raw LLM request payloads, which means the database password is now replicated across four SIEM systems. Second, a red-team exercise demonstrates that prompt injection through a malicious PDF attachment can elicit the connection string in a response. Third, the model's own output traces show it briefly echoing configuration fragments when asked to "explain your available tools." None of this required an attacker — just a naive integration.

The root failure is architectural: the gateway was not acting as a credential broker. In a correctly-designed MCP architecture, the LLM should only ever see a logical tool name and its safe parameters. Credentials should be managed at the infrastructure layer — vaulted at the platform or MCP-server level — so that the database password has no path into the LLM context window at any point in the chain.

This is the MCP gateway problem — and it is distinct from standard API gateway work. Model Context Protocol tool calls are stateful execution requests with trust, permission, and data-sensitivity semantics that standard HTTP proxies were not designed to reason about.

§1 — Why an MCP Gateway Is a Distinct Problem

Model Context Protocol gives LLMs the ability to invoke external tools: query databases, call APIs, read files, trigger workflows. Every major frontier model — gpt-5.4, claude-sonnet-4-6, gemini-3.1-pro-preview — supports tool calling natively in 2026. The enterprise adoption of agentic patterns has made MCP a critical infrastructure surface, and consequently a critical security surface.

What makes MCP governance different from LLM routing or rate limiting?

  • Credentials must not reach the LLM context window. If a system prompt or tool definition carries a database connection string, that string is now inside a model known to be susceptible to prompt injection, instruction following, and context extraction. Any log of the request replicates the credential. The correct approach is to keep credentials at the infrastructure layer, consumed only by the MCP server.
  • Tool call parameters are unsanitized user-derived data. An employee_id parameter that looks like 1 OR 1=1; DROP TABLE employees;-- is a SQL injection attempt that a naive MCP server will execute. The gateway must validate parameters before they reach any execution layer.
  • Tool results can contain sensitive data that must not return to the LLM. A query_employee_record call that returns a Social Security Number, salary, or health-related fields should have those fields redacted before the model sees the response. Otherwise the LLM can echo them in any subsequent turn.
  • Permissions are workload-scoped, not just user-scoped. An HR assistant should be allowed to call query_employee_record but not delete_employee. A customer service bot should not call any HR tools at all. This is a fine-grained access control requirement at the tool-call level.
  • MCP servers are not identical to upstream HTTP services. They are stateful execution brokers that can spawn side effects — write operations, file mutations, external API calls. Governing them requires awareness of the tool call lifecycle, not just request/response shapes.
Left: without platform credential management, credentials live in the LLM context — exposed to logs, injection, and model echoing. Right: credentials are managed at the infrastructure layer and consumed only by the MCP server; the LLM context carries only logical tool names and safe arguments.

Every AI gateway covered in this series has some answer to the MCP problem. But the depth of that answer varies dramatically — from basic access control over which keys can reach which MCP servers, to purpose-built pre/post-tool guardrail hooks with documented secret management primitives. The sections below examine each platform's specific implementation posture, with particular attention to what exactly happens between the model issuing a tool call and the tool executing against real infrastructure.

LiteLLM §2 — MCP Gateway with Permission Management

Architecture

LiteLLM's current MCP story is broader than the older Access Groups + AI Hub framing alone. The current public docs describe an MCP Gateway with a fixed endpoint for MCP tools plus permission management by key, team, or organization. The permission-management docs go further and document tool-level filtering with allowed_tools / disallowed_tools and parameter allowlists via allowed_params. That makes LiteLLM a real MCP-aware control plane, not just a catalog of published servers.

Access Groups and AI Hub still matter as administrative surfaces, but the more important current capability is that LiteLLM can decide which MCP servers, tools, and even parameters are available when a caller attempts to list or invoke tools. In practice, that gives platform teams a concrete answer to “which identities may call which MCP tools, with which arguments?”

On safety, the current docs now explicitly document MCP guardrails with pre_mcp_call and during_mcp_call modes. That is materially stronger than the earlier generic pre_call/post_call framing. The remaining gap in the public docs I checked is different: I did not find a documented post_mcp_call hook for tool-result redaction before the model sees the response, and I did not find a gateway-side mechanism that injects downstream MCP server credentials at proxy time. LiteLLM clearly has MCP-aware permissions and pre/during-call controls; it still does not document a downstream secret-broker model.

⚠ Remaining Gap — No Documented Post-MCP Hook or Gateway Secret Broker

LiteLLM's current public docs do describe MCP-aware permissions and pre_mcp_call / during_mcp_call guardrails. What they do not clearly describe is a separate post_mcp_call hook for redacting tool results before they re-enter model context, or a gateway-side mechanism that injects downstream MCP server credentials at proxy time. Teams that need those guarantees should plan to implement result redaction and secret isolation inside the MCP server or adjacent infrastructure.

Kong §3 — Direct MCP Routing with Native Auth & Tool ACLs

Architecture

Kong's current MCP documentation is stronger and more specific than a generic “HTTP adaptation” story. The AI MCP Proxy plugin sits directly in the MCP request path and converts MCP requests into standard HTTP calls so Kong can apply authentication, rate limiting, logging/tracing, request/response transformation, and ACL policies to MCP endpoints. Current examples also document per-tool access control using consumer groups and tool-level ACL rules.

The most important nuance is architectural: the docs say AI MCP Proxy is part of an MCP request flow, not an LLM request flow, and explicitly warn not to configure AI MCP Proxy together with other AI plugins on the same Service or Route. That means teams should keep the direct MCP proxy pattern separate from the AI Proxy Advanced + Prompt Guard examples Kong shows for LLM traffic governance. Both are valid Kong patterns, but they are not the same pipeline.

Kong's MCP surface also now includes AI MCP OAuth2, which the docs describe as a security layer for protecting MCP servers in listener or passthrough-listener modes. For direct MCP traffic, the better framing is therefore: Kong gives you inbound auth, per-tool ACLs, rate limiting, logging, and HTTP-level traffic control around MCP endpoints. For credential management, Kong's vault references still operate at configuration level — useful for keeping secrets out of declarative config, but not the same thing as runtime injection of downstream database credentials into tool calls.

For content and PII controls, Kong does document AI Prompt Guard and AI PII Sanitizer. But these belong to Kong's AI-plugin ecosystem more broadly, and the AI MCP Proxy docs require care when mixing plugin topologies. A safe technical reading is: Kong has strong governance building blocks, including native MCP auth and tool ACLs, but you must choose the right route topology instead of assuming every AI plugin stacks directly on the same MCP route.

# Kong declarative config — AI MCP Proxy + Key Auth + per-tool ACLs
# This follows the passthrough-listener ACL examples instead of mixing AI MCP Proxy with other AI plugins
_format_version: "3.0"

services:
  - name: mcp-acl-service
    url: http://host.docker.internal:3001/mcp

routes:
  - name: mcp-acl-route
    paths:
      - /mcp
    service:
      name: mcp-acl-service

plugins:
  - name: key-auth
    config:
      key_names:
        - apikey

  - name: ai-mcp-proxy
    service: mcp-acl-service
    config:
      mode: passthrough-listener
      tools:
        - description: List users
          name: list_users
          acl:
            allow:
              - admin
            deny:
              - developer
        - description: Get user
          name: get_user
          acl:
            allow:
              - admin
              - developer

consumer_groups:
  - name: admin
  - name: developer

consumers:
  - username: alice
    groups:
      - name: admin
    keyauth_credentials:
      - key: alice-key
  - username: bob
    groups:
      - name: developer
    keyauth_credentials:
      - key: bob-key

Kong's current MCP docs support a direct MCP route with AI MCP Proxy, inbound auth, per-tool ACLs, and standard gateway controls like rate limiting and logging. Keep that topology separate from Kong's LLM-side AI-plugin flows; the direct AI MCP Proxy route should not be stacked with other AI plugins on the same Service or Route.
⚠ Important Constraint — Separate Direct MCP Routes from LLM-Side AI Plugin Routes

Kong now documents native MCP tool ACLs and AI MCP OAuth2, so the platform is stronger here than older summaries suggested. The key implementation constraint is route topology: the AI MCP Proxy docs explicitly say not to configure AI MCP Proxy together with other AI plugins on the same Service or Route. Use direct MCP routes for MCP auth / ACL / traffic control, and separate AI-plugin routes when you want Prompt Guard or other LLM-flow controls. Kong still does not document downstream credential injection into tool parameters; vault references remain a configuration-level secret-management pattern.

Portkey §4 — Dedicated MCP Registry with Provisioning, Auth, and Observability

Architecture

Portkey now treats MCP as a first-class control plane through its dedicated MCP Gateway and MCP Registry. Rather than describing MCP mainly as standard LLM traffic flowing through generic configs, the current docs position the Registry as the central place to add internal or external MCP servers, configure authentication, monitor connection health, and control which teams can access which servers.

That control plane is deeper than the earlier config-first framing. Team Provisioning controls which workspaces and users can access a server; Tool Provisioning lets admins enable or disable individual tools, resources, and prompts at organization or workspace scope; and the authentication docs cover API keys, built-in OAuth 2.1, external OAuth, JWT validation, and identity forwarding for upstream servers.

Guardrails need a careful distinction. Portkey now documents a dedicated Guardrails surface under MCP Gateway, but that page currently says MCP guardrails are coming soon and offers early access. The broader Portkey AI Gateway still documents generic guardrails and webhook-based custom checks, so webhook validation is best described as an adjacent extensibility path today — not as a fully shipped, dedicated MCP guardrail layer in the current public MCP docs.

Credential handling is also stronger than the prior draft implied. The MCP Gateway docs say Portkey authenticates the client once, checks server and tool access, then injects the appropriate upstream credentials — OAuth tokens for services like GitHub or Slack, API keys for public servers, or identity headers for internal servers. The same docs and quickstarts also emphasize full request/response logging and MCP observability, including tool-call monitoring with full context and logs.

TrueFoundry §5 — K8s-Native MCP Control Plane with Four-Hook Security, Virtual MCPs, and Platform Secret Management

Architecture: Gateway as the Policy-Bearing MCP Proxy

TrueFoundry's AI Gateway overview makes an unusually direct statement: the gateway is the proxy layer between applications and both LLM providers and MCP servers. The gateway is positioned as the coordination layer for model and tool interactions — not just inference. Because all MCP traffic flows through the gateway, the gateway can reason about it at the tool-call-semantic level, not just at the HTTP level.

The gateway plane architecture describes a stateless, Kubernetes-native data plane built on the Hono framework. The hot path has no external calls except cache — rate limits, load balancing, authentication, authorization, and guardrails are all evaluated in-memory. Logs and metrics are queued asynchronously. This architectural design, combined with the platform's K8s deployment model, makes it feasible for MCP servers to run as in-cluster workloads network-adjacent to the gateway, though the specific MCP-server-as-K8s-pod deployment pattern represents an architectural possibility enabled by the documented platform model rather than a prescribed single configuration.

Coverage Beyond Basic MCP Proxying

ℹ What TrueFoundry Explicitly Documents for MCPs
  • Coverage of MCPs, including custom MCPs: the docs cover public MCP servers, self-hosted MCP servers, custom remote MCP servers, and deploying your own MCP servers from code or from package-based runtimes such as npx / uvx.
  • Virtual MCP Servers: TrueFoundry documents a Virtual MCP Server abstraction that lets teams curate a safer logical server by combining selected tools from multiple underlying MCP servers without standing up another custom broker.
  • Authentication / authorization: the MCP docs separate inbound gateway auth from outbound auth to MCP servers and document PAT / VAT, OAuth2, header-based auth, token passthrough, and tool-level access control with RBAC / ABAC-style policy surfaces.
  • Observability: the docs explicitly cover MCP request logs, per-tool metrics, latency and failure metrics, method-call breakdowns, and OpenTelemetry export — which is closer to an MCP operations surface than a generic LLM request log.

The Four-Hook Guardrail System

TrueFoundry's guardrails overview defines a four-hook execution model that is the most MCP-specific guardrail architecture documented across the four platforms in this comparison. The hooks are: LLM input, LLM output, MCP pre-tool, and MCP post-tool. The distinction between LLM-level hooks and MCP-tool-level hooks matters enormously in practice.

llm_input_guardrails
Fires before the model call. Use for prompt injection detection, PII masking in the user message, and content moderation.
Hook: LLM Input
llm_output_guardrails
Fires on the model response. Use for hallucination checks, secrets detection in model output, output validation.
Hook: LLM Output
mcp_tool_pre_invoke
Fires when the model issues a tool call, before the tool executes. Use for SQL injection checks, parameter validation, permission checks, sanitization.
Hook: MCP Pre-Tool — critical for agentic workflows per TrueFoundry docs
mcp_tool_post_invoke
Fires on the tool result before it returns to the model. Use for code safety checks, secrets detection, PII redaction from tool results.
Hook: MCP Post-Tool — data residency / leak prevention

The docs are explicit that the MCP hooks are critical for agentic workflows where a model autonomously invokes tools. The guardrails-getting-started page documents the X-TFY-GUARDRAILS request header, which includes all four hook arrays — giving callers the ability to specify guardrails per-request dynamically, not only through gateway-wide configuration.

Platform Secret Management: Keeping DB Passwords Out of LLM Context

The secret management documentation describes Secret Groups as a first-class platform primitive for credential management. Secret Groups are not merely encrypted key-value stores — they are governed credential namespaces that can be attached to workloads and referenced by gateway configurations via the platform's identity system. The architectural implication for MCP security: MCP server workloads deployed on the TrueFoundry platform can consume database credentials from Secret Groups via the platform's secret reference mechanism. In this pattern, the LLM sees only the logical tool name and schema; the connection string is consumed by the MCP server pod at runtime through the platform layer, rather than being passed through the LLM context. This represents a well-documented architectural pattern enabled by the combination of the platform's Secret Groups and Kubernetes deployment model — teams should evaluate their specific deployment topology against the deployment overview docs.

The changelog documents continued active investment in MCP governance: Cedar-based guardrails for MCP tools and OAuth inbound authentication for the MCP Gateway are both noted in the v0.121.x changelog window (early 2026). Cedar-based policy is particularly significant for MCP because Cedar is a structured policy language that can reason about named principals calling named actions on named resources — the semantic model needed for fine-grained MCP tool permission governance.

⚠ Limitation — Kubernetes Operational Prerequisite

TrueFoundry's gateway-plane deployment requires a running Kubernetes cluster, domain, TLS certificate, and image-pull credentials as documented in the deploy-gateway-plane guide. §7 — Editorial Verdict

Editorial assessments as of March 25, 2026.

LiteLLM
The MCP Gateway Generalist
LiteLLM now has a materially stronger MCP story than earlier drafts credited: an MCP Gateway, permission management by key/team/org, tool and parameter filtering, and MCP-aware guardrails via pre_mcp_call / during_mcp_call. That makes it a credible default choice for teams already standardized on LiteLLM. The main documented gaps I found are the absence of a clearly documented post_mcp_call hook and no documented gateway-side broker for downstream tool secrets.
Ideal for: Teams already on LiteLLM that want a single MCP gateway with permissions, tool filtering, and pre-call safety controls. Less suitable when post-tool-result redaction or gateway-brokered downstream secrets are hard requirements.
Kong AI Gateway
The Plugin-Stack Adaptor
Kong's current MCP offering is stronger than a simple adaptor story: AI MCP Proxy, per-tool ACLs, and AI MCP OAuth2 give it a real native MCP control surface. The tradeoff is architectural discipline. Kong is excellent when you want MCP traffic inside the same auth / ACL / rate-limit / observability envelope as the rest of your gateway estate, but you must keep direct MCP routes separate from incompatible LLM-side AI-plugin flows.
Ideal for: Existing Kong Gateway users that want native MCP auth, per-tool ACLs, and strong ops controls on direct MCP routes, and are comfortable managing route topology carefully.
Portkey
The MCP Registry Control Plane
Portkey's current MCP story is stronger than a config-first wrapper. Its dedicated MCP Gateway + Registry give teams central server onboarding, Team Provisioning, Tool Provisioning, built-in OAuth 2.1, JWT/identity-forwarding options, runtime credential brokering, and MCP observability. The main caveat is guardrails: native MCP guardrails are still coming soon / early access, so custom validation is better understood as an adjacent webhook path than as a fully mature first-class MCP policy engine.
Ideal for: Product and platform teams that want low-ops MCP onboarding, provisioning, auth, and observability without running a Kubernetes gateway, and can tolerate MCP-native guardrails still maturing.
TrueFoundry
The Purpose-Built MCP Security Platform
TrueFoundry remains the most purpose-built MCP governance surface in this comparison. The public docs cover dedicated MCP pre/post-tool guardrail hooks, support for public/self-hosted/custom MCPs, Virtual MCP Servers, explicit MCP auth/authz patterns, and MCP-specific observability. Secret Groups and the K8s deployment model support a strong credential-isolation architecture, while Cedar-based policy and inbound OAuth for the MCP Gateway show continued investment in enterprise MCP control.
Ideal for: Platform teams that want the deepest documented MCP control surface — especially when custom MCP onboarding, curated virtual tool exposure, enterprise auth/authz, and MCP auditability matter as much as raw model routing.

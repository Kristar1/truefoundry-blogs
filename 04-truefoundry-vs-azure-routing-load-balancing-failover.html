What this blog argues

Routing comparisons usually walk feature checklists — weighted, priority, retry, circuit breaker — and find parity. The checklist comparison is correct but misses the operationally consequential point. The question that survives the production test is not which strategies are supported; it is what does the application code bind to, and what does the platform team operate? The two products answer this differently. Azure binds the application to APIM backends with routing logic expressed in policy XML and ARM-managed pool resources. TrueFoundry binds the application to virtual models with routing strategy hidden beneath. Both can be operated to similar latency and resilience outcomes. The operational cost of a routing change differs structurally — and for AI workloads where routing strategy evolves weekly, that cost compounds.

§0 — TL;DR & Quick-Picks

A routing change that becomes a contract question

At 11:47 on a Wednesday in early March, a fintech AI team rolls out what they describe in the change ticket as "a small experiment" — they want to send 80% of credit-decisioning traffic to Azure OpenAI gpt-4o in East US and 20% to a self-hosted Llama-3.1-70B vLLM deployment running in their Sweden Central K8s cluster. The economics are compelling: their finance lead has calculated that the experiment, if successful, would shave nearly 28% off their monthly AI bill. The platform team adds a new backend pool to APIM with weighted routing, edits the policy XML on the credit-decisioning API to reference the new pool, deploys the change, and watches the dashboards.

Within forty minutes, customer-success is fielding complaints about latency. The Llama deployment is healthy. CPU is well under target, GPU utilization is in the green, the error rate is zero. But the time-per-output-token on the self-hosted deployment is consistently 2.3× the Azure OpenAI median — not because Llama is slow per-token in absolute terms, but because gpt-4o on Azure's optimized inference infrastructure is exceptionally fast, and the streaming UX of the credit-decisioning UI is calibrated against the faster backend. The 20% of users routed to the self-hosted backend are seeing visibly slower responses; some are complaining; one large customer has paged their account manager.

The platform team's instinct is correct: this is a routing problem, not a model problem. The right strategy is latency-aware routing — measure the streaming time-per-output-token on each backend, and shift traffic toward the lower-latency backend automatically. But here the architecture decision made four months ago lands directly on the team's hour-by-hour: latency-aware routing is not a first-class APIM strategy. The team can implement it, but the implementation is a custom policy expression that reads cached metrics and selects a backend in set-backend-service; the cached metrics need to be populated by some out-of-band aggregator; the policy XML on three different APIs has to be updated; the deploy goes through a PR, a CI pipeline, and a controlled rollout. Six engineers, three hours, two pull requests. The fix lands at 15:08; latency normalizes by 15:30.

The architectural lesson the team takes away from the postmortem is what this blog is about: the unit the application binds to should not be the unit the platform team optimizes against. If the application calls a stable virtual model and the platform routes underneath it, routing changes are config edits — apply, NATS sync, done. If the application calls a backend by name, and the backend's routing strategy is expressed as policy XML on the API, every routing change is a policy redeploy with the change visible to anything that consumes the policy. Both architectures can produce the same outcome; the cost of getting there is what differs.

TL;DR
  • Azure routes through APIM backends and load-balanced pools. Backends are first-class ARM resources. The load-balanced pool supports round-robin, weighted, priority, and session-aware strategies. Circuit breakers are first-class properties of the backend resource (failureCondition, tripDuration, acceptRetryAfter). The retry policy with exponential backoff is institutional. The contracting unit is the backend (or the API that references it).
  • TrueFoundry routes through virtual models. The application binds to a stable virtual model name; routing happens beneath it with weighted, priority, latency-aware (TPOT-based), retry, circuit-breaker, and metadata-driven targeting. Provider/model swap is a config edit. The contracting unit is the virtual model — one indirection from the application's view.
  • Latency-aware routing is the AI-specific strategy gap. APIM's pool strategies are excellent for traditional API traffic but not built around streaming-LLM latency metrics like time-per-output-token. TrueFoundry treats latency-aware as first-class.
  • Honest framing. APIM's policy DSL is genuinely more mature than TrueFoundry's routing config; for traditional API governance, the comparison is reversed. For AI-specific routing patterns, the abstraction TrueFoundry offers reduces operational cost in ways that compound.
Quick-Pick
Azure
The right call when the AI workload sits inside an existing APIM estate, the routing patterns are similar to the API patterns the team already operates, the policy-expression DSL is institutional muscle, and the team has discipline around policy XML edits as part of routing changes.
Quick-Pick
TrueFoundry
The right call when AI engineering teams run many routing experiments — provider mix tuning, latency-aware fallback, cost-aware steering — and the friction of policy redeploys per change is the dominant operational cost. Especially decisive when streaming LLM latency (time-per-output-token) is the metric that drives routing decisions.

Why this opens Movement II

§1 — The contracting-unit question

The routing comparison that finds parity walks features. Both products do weighted load balancing. Both do priority groups for failover. Both have retry with backoff. Both have circuit breakers. Both can drive routing decisions from request metadata. If routing comparison were a checklist, this blog would already be over.

The question that decides the architecture is one level up. What is the unit the application code binds to, and what is the unit the platform team operates? When those two units are the same — application calls a backend, platform configures the backend — every routing change is a change visible to the application. When they are different — application calls a stable abstraction, platform configures providers underneath — routing changes are platform-team-only operations. The first model is easier to reason about for a single-team API where the application owners and the platform owners are the same people. The second is easier to operate for an AI program where the application code is owned by feature teams, the routing strategy is owned by an AI platform team, and routing strategy needs to evolve weekly without a coordination cost on every change.

This is not a theoretical distinction. The fintech scenario at the top of this blog is one concrete instance; the same pattern shows up everywhere routing strategy is part of how AI cost and latency get tuned. A few examples that the comparison framework needs to account for:

  • Provider mix tuning. An AI program runs production on Azure OpenAI but adds Anthropic Claude as a 10% canary for a specific workload to evaluate quality differences. The 10% needs to ramp to 30% over two weeks, then to 50% if quality holds. Each step is a routing-weight change.
  • Latency-aware fallback. A high-traffic chat endpoint routes primarily to AOAI East but falls back to AOAI West when East's observed time-per-output-token exceeds a threshold. The threshold needs to be tunable based on observed traffic patterns; the metric is streaming-LLM-specific (TPOT, not request latency).
  • Cost-aware steering. A tier-2 internal tool routes to a self-hosted Llama backend during business hours and to AOAI on-demand outside business hours. The schedule changes after a quota negotiation with the provider.
  • Tenant-aware routing. An enterprise SaaS sends the top-10 highest-paying customers to a dedicated provisioned-throughput AOAI deployment and the rest to a shared standard deployment. The list of top-10 customers changes monthly.

For each of these patterns, the routing logic is the same on both products — weighted, priority, latency-aware, metadata-driven. The structural difference is what changes when the routing logic itself changes. On Azure, every change is a policy XML edit (or a backend pool reconfiguration through ARM), reviewed and deployed through whatever pipeline governs APIM changes. On TrueFoundry, the change is a YAML edit to the virtual model definition, applied through tfy apply, propagated to gateway pods through NATS in seconds.

The honest boundary on this argument

Azure's policy DSL is more mature and expressive than TrueFoundry's routing config. For complex transformation logic — rewriting requests, manipulating headers, chaining policies — APIM is genuinely better engineered, and the fluency that years of Azure API Management practice produces in a team is real institutional muscle. The argument is not that TrueFoundry's routing layer is more powerful; it is that for AI-specific routing patterns, the abstraction TrueFoundry chose reduces the operational cost of routing changes. For traditional API governance, the comparison reverses; for AI workloads, the abstraction wins.

The rest of the blog walks both architectures in working detail. §2 walks the Azure routing surface — backends, load-balanced pools, the circuit-breaker property, the retry policy, and the AOAI-specific policies that make APIM's AI-routing story competitive — with a working Bicep + policy composition. §3 walks TrueFoundry's virtual model surface — the routing strategies, the application contract, latency-aware as a first-class strategy — with a working YAML + Python composition. §4 walks four concrete routing scenarios side by side. §5 ends with the editorial verdict.

Microsoft§2 — Azure: backends, pools, circuit breakers, and policy expressions

APIM's routing surface is, frankly, one of the most mature in the industry. Backends are first-class ARM objects representing upstream services. Backend pools aggregate multiple backends with a chosen strategy. Circuit breakers are first-class properties of the backend resource. The retry policy with exponential backoff is institutional. set-backend-service with policy-expression-based selection covers any custom routing logic. The combination, written carefully, produces a credible AI-routing implementation.

The architectural choice that shapes the operating model is that routing logic lives in policy XML. Backend pools are ARM resources with declarative configuration; policies that reference those pools are XML documents on APIs, products, or operations. Both surfaces are first-class — but both surfaces are visible to anything that consumes the policy or the API definition.

Layer 1 — Backends and load-balanced pools

A backend in APIM is an ARM resource of type Microsoft.ApiManagement/service/backends. A "single" backend points at one URL with credentials and circuit-breaker config. A "Pool" backend references multiple single backends with priority groups and weights.

Microsoft's backends documentation describes the pool semantics directly: backends in lower priority groups will only be used when all backends in higher priority groups are unavailable because circuit breaker rules are tripped. Within a priority group, traffic is distributed across backends according to their weight values. This gives the operator a clean two-dimensional model: priority for failover, weight for distribution within a tier.

Layer 2 — Circuit breakers as backend properties

The circuit breaker is a property of the backend itself, not a separate policy. Microsoft's documentation specifies the shape:

  • failureCondition — the conditions under which a request counts as a failure. Specified as some combination of count, percentage, interval, statusCodeRanges, and errorReasons (the symbolic reasons APIM emits like 'Server errors', 'Timeout', 'BackendConnectionFailure').
  • tripDuration — how long the circuit stays open after tripping. ISO 8601 duration format (PT10S, PT1M, PT1H).
  • acceptRetryAfter — when set to true, the circuit breaker honors the Retry-After header from the backend. AOAI returns 429 with a Retry-After header on rate-limit; acceptRetryAfter: true uses that value as the trip duration. This is genuinely well-engineered for AOAI traffic; it makes circuit recovery synchronized with the upstream service's own recovery signal.

When the circuit breaker trips, APIM stops sending requests to that backend and returns 503 Service Unavailable to the caller — unless the request is handled by a load-balanced pool that still has another backend available, in which case the pool routes to that backend. This composition (pool with priority + per-backend circuit breaker) is the canonical AOAI failover pattern Microsoft's own architecture guidance documents.

Layer 3 — AOAI-specific policies

For AI traffic specifically, APIM exposes a set of AOAI-aware policies that complement the routing layer:

  • azure-openai-token-limit — TPM and quota enforcement keyed by subscription, IP, or any policy expression. Semantics covered in Blog 6.
  • azure-openai-emit-token-metric — emits per-request token attribution to Azure Monitor with up to 5 custom dimensions.
  • azure-openai-semantic-cache-lookup / azure-openai-semantic-cache-store — semantic caching, covered in Blog 5.

The architectural point is that these AOAI policies are composable with the routing layer, not part of it. A team that wants to combine token-limit enforcement, semantic caching, and weighted routing across two AOAI deployments writes one policy XML document that includes all of them in the correct order.

What the operator writes — Bicep + policy composition

The Bicep below defines two AOAI single backends (East + West) each with a circuit breaker, a self-hosted Llama backend at lower priority, and a load-balanced pool that composes them.   name: 'contoso-apim'
}

resource aoaiEast 'Microsoft.ApiManagement/service/backends@2024-05-01' = {
  parent: apim
  name: 'aoai-east'
  properties: {
    url: 'https://contoso-aoai-east.openai.azure.com/openai'
    protocol: 'http'
    type: 'Single'
    circuitBreaker: {
      rules: [{
        name: 'aoai-east-rule'
        failureCondition: {
          count: 1
          interval: 'PT10S'
          statusCodeRanges: [
            { min: 429, max: 429 }
            { min: 500, max: 503 }
          ]
        }
        tripDuration: 'PT10S'
        acceptRetryAfter: true   // honors AOAI's Retry-After header
      }]
    }
  }
}

The companion policy XML on the credit-decision API:

XML
<policies>
  <inbound>
    <base />
    <set-backend-service backend-id="ai-credit-decision-pool" />

    <!-- AOAI-specific token enforcement, keyed by subscription -->
    <azure-openai-token-limit
        tokens-per-minute="200000"
        counter-key="@(context.Subscription.Id)"
        estimate-prompt-tokens="true"
        retry-after-header-name="Retry-After" />

    <!-- Tenant-tier override: route tier-A explicitly to East -->
    <choose>
      <when condition="@(context.Request.Headers.GetValueOrDefault(&quot;X-Tenant-Tier&quot;,&quot;&quot;) == &quot;A&quot;)">
        <set-backend-service backend-id="aoai-east" />
      </when>
    </choose>

    <authentication-managed-identity resource="https://cognitiveservices.azure.com/" />
  </inbound>
  <backend>
    <retry condition="@(context.Response.StatusCode == 429
                       || context.Response.StatusCode >= 500)"
           count="3" interval="1" max-interval="10" delta="2"
           first-fast-retry="true">
      <forward-request />
    </retry>
  </backend>
  <outbound>
    <azure-openai-emit-token-metric namespace="Contoso/AI">
      <dimension name="tier"
        value="@(context.Request.Headers.GetValueOrDefault(&quot;X-Tenant-Tier&quot;,&quot;default&quot;))" />
    </azure-openai-emit-token-metric>
    <base />
  </outbound>
</policies>
Composition based on Microsoft Learn for set-backend-service, retry, azure-openai-token-limit, and azure-openai-emit-token-metric. The choose/when construct is the standard idiom for tenant-aware routing override.

What APIM is genuinely strong at — and the AI-specific gap

Reading the Bicep + policy together, several things stand out as genuinely well-engineered:

  • Circuit breaker semantics for AOAI traffic. acceptRetryAfter: true is a thoughtful piece of design. It synchronizes circuit recovery with AOAI's own quota recovery signal, which means the pattern handles AOAI's rate-limit behavior correctly without the operator having to tune trip durations against AOAI's quota refresh cadence.
  • Pool semantics with priority groups. The two-dimensional model (priority for failover, weight for distribution within priority) is exactly the right shape for production AI routing. The fintech scenario at the top of this blog could be expressed cleanly in this model.
  • Retry policy with first-fast-retry. The first-fast-retry="true" attribute is genuinely well-tuned for AI traffic — the first retry happens immediately, subsequent retries back off exponentially. This handles transient failures fast without amplifying load on degraded backends.
  • Composition with token-limit and emit-token-metric. One policy XML document handles routing, enforcement, and attribution. For teams that already write APIM policy fluently, this is excellent leverage.
The honest caveat — latency-aware routing

Latency-aware routing — choosing a backend based on observed time-per-output-token, the metric that actually matters for streaming LLMs — is not a first-class APIM strategy. It can be implemented through policy expressions reading custom variables populated from cached metrics, but the implementation is the customer's, not the platform's. For an AI workload where TPOT-aware routing is the right strategy (which it often is — streaming UX is what users perceive), this is a meaningful gap. The fintech scenario at the top of this blog is exactly this gap landing on a production team.

The architectural fact that produces both the strengths and the gap is the same: Azure's routing surface is a composition of strong primitives, all expressed through policy XML and ARM resources. For traditional API governance and for routing strategies that map cleanly to the existing primitives, the composition is excellent. For AI-specific routing patterns that require a different abstraction — like TPOT-aware routing — the customer ends up implementing the abstraction inside policy expressions. The result works, but the work is custom rather than platform-provided.

TrueFoundry§3 — TrueFoundry: virtual models as the application contract

TrueFoundry's virtual model is one concept, and that one concept produces the operational difference. A virtual model is a stable, application-facing endpoint with a name. The application calls it by name; the platform decides which provider/model fulfills the call based on the routing policy attached to that virtual model.

The architectural choice that shapes the operating model is that routing logic lives beneath the virtual model abstraction. The application code references the virtual model name; everything underneath — providers, weights, fallback behavior, latency-aware shifts, retry semantics, circuit breakers — is platform-team config that does not appear in application code or in any policy the application consumes.

Layer 1 — Virtual models as the contract

A virtual model in TrueFoundry has a name (e.g., credit-decision-prod), a mode (chat-completions, embeddings, completions), and a routing configuration. Applications call the virtual model the way they would call a model from any OpenAI-compatible provider — same SDK, same API shape, same parameter names. The only thing that's different is the model parameter is the virtual model name rather than a provider model name like gpt-4o.

This is the contract: the application binds to a name, not to a backend. The platform team can swap the model underneath, change the provider, ramp a canary, shift traffic based on observed latency, fall back on rate-limit — all without the application changing at all.

Layer 2 — Routing strategies as first-class primitives

The routing configuration documents the strategies the platform exposes:

  • Weighted. Same as APIM's pool weight semantics. Distributes traffic across backends in proportion to weight values.
  • Priority. Same as APIM's priority groups. Higher-priority backends are used first; lower-priority backends are fallback.
  • Latency-aware. First-class strategy. The gateway tracks observed time-per-output-token per backend over a configurable window; when one backend's p99 TPOT exceeds the configured threshold, the strategy shifts traffic toward the lower-latency backend automatically. This is the strategy that, in the fintech scenario at the top, would have prevented the production incident.
  • Metadata-driven. Routing rules can match on request metadata (tenant, environment, region, custom fields). The when: { metadata.tier: "A" } shape replaces APIM's choose/when policy expressions with declarative routing rules.

Layer 3 — Retry and circuit breaker per virtual model

Both retry-with-backoff and circuit-breaker semantics are configured per virtual model rather than per backend. The retry config specifies status codes, max attempts, and backoff parameters. The circuit-breaker config specifies failure threshold and recovery duration. When the breaker opens, the gateway routes around the failing backend (using the priority/weighted/latency-aware strategy on the remaining backends) until the recovery duration elapses.

The semantics are similar to APIM's; the configuration locus is different. On TrueFoundry, the retry and circuit-breaker config sits on the virtual model alongside the routing strategy — one declarative document. On Azure, retry is a policy on the API and circuit breaker is a property of the backend resource — two configuration locations that have to be coordinated.

What the operator writes — virtual model definition + application contract

The YAML below defines the same routing scenario the Bicep above expressed: 70/30 weighted split between AOAI East and West with latency-aware shifting, fallback to self-hosted Llama, tenant-tier override, retry, and circuit breaker. Note that this is one document, applied through one CLI command:

YAML
# Illustrative virtual model with all routing concerns in one document
# Source: TrueFoundry virtual-model + load-balancing-configuration docs
type: virtual-llm-model
name: credit-decision-prod
mode: chat-completions

# Routing rules — evaluated top-to-bottom
routing:
  # Tenant-tier override — explicit routing for tier-A
  - when: { metadata.tenant_tier: "A" }
    target: { provider: azure-openai, model: gpt-4o-east, priority: 1 }

  # Default — latency-aware over the primary tier, fallback to self-hosted
  - default:
      strategy: latency-aware
      latency_window_seconds: 60
      shift_threshold_p99_ms: 800
      backends:
        - { provider: azure-openai, model: gpt-4o-east, weight: 70, priority: 1 }
        - { provider: azure-openai, model: gpt-4o-west, weight: 30, priority: 1 }
        - { provider: self-hosted, model: vllm/llama-3-1-70b, priority: 2 }

# Retry — same status codes as APIM, similar backoff semantics
retry:
  on_status: [429, 500, 502, 503, 504]
  max_attempts: 3
  backoff: { initial_ms: 1000, max_ms: 10000, multiplier: 2.0 }
  honor_retry_after: true   # honor provider Retry-After header

# Per-backend circuit breaker (default applies to all; per-backend overrides supported)
circuit_breaker:
  failure_threshold: 5
  failure_window_seconds: 30
  recovery_after_seconds: 30

# Token enforcement (covered in Blog 6)
budget:
  scope: workspace
  monthly_usd: 50000

rate_limits:
  - { scope: virtual_account, tokens_per_minute: 200000 }
Illustrative virtual model definition consistent with virtual-model docs and routing configuration. The latency-aware strategy tracks per-backend TPOT and shifts traffic when observed p99 exceeds the threshold. Apply through tfy apply; NATS propagates the change to gateway pods in seconds.

The application's view of the world is unchanged regardless of routing strategy. The same client code calls the same virtual model name; the metadata header drives the tenant-tier override; everything else happens beneath the contract:

Python
from openai import OpenAI

# Application binds to the virtual model name. Provider/strategy choice is platform-side.
client = OpenAI(
    base_url="https://gateway.contoso.com/api/llm/api/inference/openai",
    api_key=os.environ["TFY_GATEWAY_TOKEN"],
)

response = client.chat.completions.create(
    model="credit-decision-prod",      # virtual model — stable contract
    messages=[
        {"role": "system", "content": "You are a credit-decisioning assistant..."},
        {"role": "user", "content": user_prompt},
    ],
    # Optional metadata steers the routing rule
    extra_headers={
        "x-tfy-metadata": json.dumps({
            "tenant_tier": tenant.tier,    # drives the tenant-tier routing rule
            "request_id": req_id,
        }),
    },
)
Application-side call remains constant across routing changes. Standard OpenAI SDK shape; the only difference is the base_url and the x-tfy-metadata header that the gateway consumes for metadata-driven routing.
What this earns operationally

(1) Routing experiments don't require application redeploys. The fintech scenario at the top of this blog — switching from weighted to latency-aware — is a YAML edit on one virtual model definition, applied through tfy apply, propagated through NATS to gateway pods. No PR on the application code, no APIM redeploy, no policy XML diff to review. (2) Provider/model swaps are config-only. The same virtual model can have its provider changed from AOAI to Anthropic or to a self-hosted model without the application contract changing. (3) Latency-aware routing is a platform-provided strategy, not a custom policy expression. The TPOT measurement, the window, and the shift logic are platform code; the operator picks parameters. (4) Hot-path routing decisions are in-memory. The gateway pod holds the routing config and the per-backend latency observations in memory; routing decisions are made without external lookups (Blog 1's gateway plane architecture).

§4 — Four routing scenarios compared

The most useful way to compare these routing models is to walk concrete operational scenarios and identify the time-to-deploy for each routing change. The four scenarios below are realistic patterns drawn from production AI engineering practice. Each is shown side by side under both products.

SCENARIO 1Switch from weighted to latency-aware routing under production pressure

The fintech scenario from the opening. The team has a 70/30 weighted split running in production; observed TPOT differences are causing customer-visible latency degradation; they need to switch to latency-aware routing to shift traffic away from the slower backend automatically.

Azure path

Latency-aware is not a first-class strategy. The team must (1) implement TPOT measurement — typically by emitting a custom metric from the policy on response, populating Application Insights, and aggregating per-backend; (2) build a cache lookup that reads the per-backend TPOT from a backing store; (3) write a policy expression in set-backend-service that picks a backend based on the cached TPOT values; (4) review and deploy the policy XML changes through the standard pipeline. Achievable but custom. Time-to-deploy for the scenario: hours to days, depending on how much TPOT measurement infrastructure already exists.

TrueFoundry path

Latency-aware is a first-class strategy. The team edits one field in the virtual model YAML — change strategy: weighted to strategy: latency-aware, set latency_window_seconds and shift_threshold_p99_ms — runs tfy apply, NATS propagates the change to gateway pods, and the gateway begins making latency-aware decisions immediately. Time-to-deploy: minutes.

SCENARIO 2Multi-region failover — primary region's AOAI deployment trips circuit

Production AI traffic flows primarily to AOAI in the team's home region. The home-region AOAI instance hits an extended quota issue (or a regional incident); circuit breaker trips; traffic should automatically fail over to a secondary-region AOAI deployment without any manual intervention.

Azure path

✅ This scenario is exactly what APIM's pool semantics are designed for. Both regions are backends in a pool with priority 1 (home) and priority 2 (secondary); each has a circuit breaker; acceptRetryAfter: true synchronizes circuit recovery with AOAI's quota-recovery signal. When the primary trips, the pool routes to priority 2 backends. When the trip duration elapses (or the Retry-After from AOAI passes), traffic automatically returns to priority 1. Battle-tested pattern. Time-to-deploy: zero (already covered by initial setup).

TrueFoundry path

✅ Same outcome through priority-based routing in the virtual model config. Both regions defined as backends with priority: 1 (home) and priority: 2 (secondary); circuit breaker per backend; honor_retry_after: true on retry config. Architecturally identical to Azure's pattern. Time-to-deploy: zero.

SCENARIO 3Add a 10% canary to a new provider for quality A/B

An AI program runs production on AOAI and wants to add Anthropic Claude as a 10% canary on one workload to evaluate quality differences. The 10% needs to ramp to 30% over two weeks if quality holds, then to 50% if approved.

Azure path

Add Anthropic as an APIM backend (with bearer-token auth instead of managed identity). Add it to the existing pool with weight 10. Each ramp step is a Bicep change to the pool composition + ARM apply. The application doesn't change — it still calls the APIM API endpoint — but every weight change is a pool edit. Time-to-deploy per ramp step: a config-change PR, Bicep diff review, ARM apply. Not slow, but a coordination cost per step.

TrueFoundry path

Add Anthropic as a backend in the virtual model with weight 10. Each ramp step is a YAML edit on the virtual model definition + tfy apply. Same outcome with the same coordination but the change surface is one document rather than ARM + policy. For an AI team running multiple canary experiments simultaneously, the cumulative coordination cost is meaningfully lower. Time-to-deploy per ramp step: one YAML edit, one apply.

SCENARIO 4Tenant-aware routing — top-10 customers to provisioned-throughput

A SaaS company sends its top-10 highest-paying customers to a dedicated AOAI provisioned-throughput deployment, and the rest of the tenant base to a shared standard-tier deployment. The list of top-10 customers changes monthly based on revenue.

Azure path

The conditional routing is a choose/when policy expression that reads a tenant ID from a header and maps it to a backend. Implementing the actual top-10 list as a maintainable data structure is the operational question — APIM named values can hold the list but updating them is a separate workflow; storing the list externally (Cosmos DB, App Configuration) requires an API call from the policy. Achievable, but the moving-list-of-tenants pattern has multiple workable shapes and the team has to pick one. Time-to-deploy per monthly update: depends on the chosen pattern.

TrueFoundry path

The conditional routing is a metadata-driven rule on the virtual model. The list of top-10 tenants can live in the virtual model's metadata config (updated through tfy apply), or the application can pass a tier header that the rule matches on. Both shapes are first-class in the routing config; either is a single-document edit per monthly update. Time-to-deploy per monthly update: one YAML edit.

§5 — Editorial verdict

Best fit
Azure
APIM's routing surface is genuinely one of the most mature in the industry. Backend pools with priority and weight, circuit breakers as first-class backend properties with the well-engineered acceptRetryAfter semantics for AOAI, the retry policy with first-fast-retry, and the AOAI-specific policies (azure-openai-token-limit, azure-openai-emit-token-metric) compose into a credible AI-routing implementation. For traditional API governance and for routing strategies that map cleanly to the existing primitives, APIM is best-in-class. The institutional muscle a team builds operating APIM is real; the documented patterns from Microsoft's architecture guidance are battle-tested.
Choose Azure if: the team already operates APIM, AI workloads sit inside an existing API Management estate, the routing patterns the workload needs map to weighted/priority/metadata-driven, and policy-XML edits as part of routing changes are acceptable cost.
Best fit
TrueFoundry
Virtual models as the application contract make routing experiments cheap. Latency-aware routing as a first-class strategy matches the metric that actually matters for streaming LLMs — and the fintech scenario at the top of this blog is exactly the kind of incident that virtual-model abstraction structurally prevents. NATS-driven config propagation makes routing changes safe to make often. For AI programs where routing strategy evolves weekly — provider mix tuning, latency-aware fallback, cost-aware steering, tenant-aware routing — the cumulative reduction in coordination cost is the value the abstraction delivers.
Choose TrueFoundry if: AI engineering teams need to do many routing experiments, latency-aware routing is the right strategy for the workload, app-stable routing changes are an explicit operational requirement, or the team prefers one declarative document for routing strategy rather than two coordinated surfaces.

The most honest framing is that the architectures optimize for different sides of the same trade-off. Azure's routing surface composes strong primitives expressed through mature configuration languages — policy XML and ARM Bicep. The composition is excellent and well-engineered for traditional API governance, including AI workloads that fit the established primitives. TrueFoundry's routing surface compresses the same capabilities into a virtual-model abstraction tuned specifically for AI workloads. The compression buys an application-stable contract and a single edit surface for routing changes; it pays the cost of a less expressive policy DSL for non-routing concerns.

For most AI workloads, the right answer depends on whether the workload's routing strategy is set-once-and-forget or evolves with the program. Set-once routing — pool to two AOAI regions with circuit breaker, retry, and a tier-aware override — both products handle equally well; the architectures look similar at the matrix level. Evolving routing — weekly experiments, latency-aware tuning, canary rollouts, tenant-list updates — is where the abstraction TrueFoundry chose pays off, and where the architecture's editorial argument lands.

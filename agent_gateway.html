§0 — TL;DR & Quick-Picks

🔥 Hypothetical Composite Scenario: One Automation, Many Governed Calls

This is a composite hypothetical illustrating a class of failure seen in agentic deployments. It is meant to explain the architectural pattern, not describe one specific incident.

An enterprise platform team deploys an autonomous remediation agent. To the business owner, it looks like one automation: read the ticket, inspect the codebase, propose a fix, run tests, and open a pull request. Under the hood, it is a chain of many LLM calls and tool invocations.

That is where agent gateway design matters. The gateway cannot just authenticate and route the first request; it has to carry identity, budgets, policy checks, and trace context across every call in the loop. Otherwise the riskiest and most expensive part of the workflow happens after the front door — leading to harder debugging, weaker spend control, and less trustworthy automation.

Agent loops are governed call chains, not single requests. That is why this post evaluates gateways on four dimensions: connection lifecycle, tool-call governance, cross-call state, and traceable observability.

§1 — Why Agent Loops Break the Gateway Contract

Every AI gateway in this series was evaluated first against its LLM routing and load-balancing characteristics. Those capabilities matter for agents too — but they don't address the structural problem agents introduce. A synchronous chat completion is a request-response pair: one HTTP connection, one LLM call, one response. An agent loop is an orchestration graph: potentially dozens of LLM calls, tool invocations via MCP or REST, accumulated context state, and feedback cycles — all of which must be governed, observed, and protected while lasting far longer than any reasonable HTTP timeout.

The gateway problems agents expose cluster into four categories:

  • Connection lifecycle mismatch. Standard NGINX, ALB, and Kubernetes ingress timeouts sit between 30 and 300 seconds. Agent loops for complex tasks routinely run 5–20 minutes. Accumulated intermediate results and token context may be lost when the connection is severed, unless the application checkpoints state or externalizes orchestration progress. The fix is not simply "raise the timeout" — it requires either streaming intermediate progress via SSE, switching to a persistent WebSocket, or moving execution behind a queue-backed async service that decouples the client from the execution lifecycle.
  • Tool-call governance gap. When a model calls a tool — whether via MCP, REST, or a function-calling schema — that tool call is a lateral API request that conventional gateways do not inspect. The model can be manipulated into invoking dangerous tools, passing malformed parameters, or leaking secrets through tool results. A gateway that only governs the LLM call but ignores the tool call has an unguarded injection surface.
  • Cross-call budget and identity drift. A single agent run may cost orders of magnitude more than a single chat turn. An agent that burns through a team's daily token budget in one long loop is a real operational risk. Budget state, rate limit windows, and virtual key associations must persist across the full loop — not reset per HTTP request.
  • Trace-level observability, not request-level. When an agent fails, the root cause is rarely the last LLM call. It may be a tool call three steps earlier that returned poisoned context. You need a trace tree — with spans for every LLM call, every tool invocation, every guardrail evaluation — not a flat request log.
Agent loop timeline showing where each gateway concern fires. The real question is not whether a platform can accept the first request, but whether it can govern every repeated LLM and tool call across the full loop.

Each gateway in this review addresses these concerns to varying degrees. The critical architectural distinction is that the gateway layer governs individual calls — each LLM request, each tool invocation. It does not (and should not) maintain the agent's reasoning state or conversation history. That state belongs to the application or the execution substrate. What differs between platforms is: how many of these call-level concerns they govern natively, whether their parent platform provides an async execution substrate for the full loop, and how those two layers are integrated.

LiteLLM §2 — Redis-Backed Identity State + MCP Access Groups

Architecture for Agents

LiteLLM's most important contribution to agent governance is architectural: the proxy server validates virtual key identity, checks budget state, and applies rate limits before the router sees a request. As documented on the proxy architecture page, this pipeline — virtual key validation → Redis/in-memory cache lookup → database fallback → per-key, per-user, per-team, and global rate-limit checks → router → provider — fires on every single call in an agent loop. The Redis cache ensures that key lookup and budget state are consistent across calls without a database round trip for each LLM request, which matters when an agent makes 15+ calls in rapid succession.

For agent resource bundling, the access groups page documents a capability that is particularly well-suited to multi-tool agents: a single access group can bundle models, MCP servers, and agents, then be attached to a key or team. That means a virtual key issued to an agent service automatically scopes the entire set of resources that service is allowed to invoke — not just the LLM endpoints. Budget and rate limits then apply to all calls made under that key, regardless of whether they are LLM calls or pass-through calls to external agent APIs.

LiteLLM also exposes a pass-through endpoint capability for routing requests to any external API with centralized authentication, spend tracking, and budgeting applied. This is the mechanism for pulling orchestrator APIs, rerankers, or custom tool endpoints under the same budget envelope as the LLM calls, giving a unified cost picture across a long agent run. At the guardrail layer, LiteLLM supports pre_call, during_call, and post_call execution modes, documented in the guardrails quick start, plus in-memory prompt injection detection using similarity checks and LLM evaluation.

⚠ Limitation — No Native Async Execution Substrate

LiteLLM is a proxy, not an execution runtime. It governs each call within an agent loop excellently, but it has no native mechanism for handling the full loop as a long-running async job. Your application or orchestration layer must handle the connection lifecycle problem — either by implementing SSE streaming, polling, or a queue-backed async worker. At 1000+ RPS, the deployment guide explicitly states Redis is required to prevent database connection exhaustion — meaning production agent deployments need Redis as a first-class infrastructure dependency, not an optimization.

Kong §3 — MCP Protocol Adapter with Shared Plugin Governance

Architecture for Agents

Kong's answer to agent governance is protocol-level: the AI MCP Proxy plugin converts MCP requests into standard HTTP calls and returns responses in MCP format. This is architecturally significant because MCP tool traffic can then traverse Kong's existing plugin ecosystem, letting many of the same governance controls — auth, rate limiting, prompt guards, and audit logs — apply to tool calls without requiring a separate governance surface. The MCP Traffic Gateway documentation explicitly frames this as applying security, governance, and observability controls to MCP servers using available Kong plugins.

For the connection lifecycle problem, Kong documents a response_streaming configuration parameter in the streaming guide with allow, deny, and always modes. This governs how streaming output is handled through the gateway data plane — including the include_usage flag that lets the gateway surface token usage statistics in the final SSE frame of a streaming response. AI Proxy Advanced extends streaming support for multi-target scenarios, including a response_buffer_size setting (documented in the AI AWS Guardrails plugin) to control how streaming output is chunked for guardrail inspection — a critical capability when running content-safety checks on streamed agent outputs.

Kong's observability for agent workloads is also strong: the metrics page documents that AI metrics can include a request_mode label (from Kong v3.12) with values oneshot, stream, and realtime, making it possible to separately track and alarm on streaming vs. non-streaming agent traffic in Prometheus. The Gen AI OTel attributes reference documents rich span metadata including operation name, provider name, model name, and request parameters.

⚠ Limitation — Enterprise Plugin Dependencies & No Async Compute Plane

The MCP Traffic Gateway and AI MCP Proxy plugin are documented for Kong Enterprise. AI Proxy Advanced (required for multi-target balancing with streaming) requires a minimum Kong 3.8 and an enterprise license. As with LiteLLM, Kong has no native async execution substrate — the gateway governs each call in the loop, but you need separate infrastructure to handle the long-running loop itself. Always verify plugin version and license requirements against the plugin compatibility reference.

Portkey §4 — Trace-First Agent Resilience via Composable Configs

Architecture for Agents

Portkey's primary contribution to agent governance is observability depth combined with composable resilience. The tracing page documents OTel-compliant tracing that groups related LLM calls into spans under a trace ID using a tree structure, with explicit support for W3C trace context. The docs describe this as suitable not only for single prompt-response flows but specifically for multi-step agentic and tool-augmented workflows. This trace-tree model makes Portkey one of the strongest options for diagnosing where inside an agent loop a failure occurred — something flat request logs cannot provide.

Resilience for agent loops is handled through Portkey's config-first composable strategy graph. The fallback, retry, and circuit breaker pages show that all three compose into the same JSON config object. The circuit breaker tracks per-strategy-path health with configurable failure thresholds, failure-rate percentages, cooldown intervals, and minimum request counts before statistical evaluation — making it a path-aware reliability control rather than a provider-global kill switch. For an agent making 14+ LLM calls, a circuit breaker that opens on a specific routing path after 3 failures prevents cost from accumulating on a broken execution branch.

The analytics page surfaces a metric that directly validates agent resilience investment: the number of requests rescued by gateway strategies. That single metric tells you whether the retry/fallback/circuit-breaker investment in your agent config is actually firing and saving executions.

⚠ Limitation — MCP Governance Depth & No Native Async Execution

Portkey features a dedicated MCP Gateway (GA in January 2026) that provides centralized authentication, RBAC, and observability for MCP servers. However, it governs traffic primarily at the gateway policy layer, and public docs do not currently detail per-tool-invocation mutation hooks equivalent to TrueFoundry. Furthermore, like LiteLLM, Portkey has no native async execution substrate — long agent loop management must be handled in application or orchestration code. The hybrid VPC data plane (Helm-deployed; documented at enterprise/hybrid2) keeps payload data in-network but the connection lifecycle problem remains an application concern.

TrueFoundry §5 — K8s-Native Async Execution + Four-Hook MCP Guardrails

Architecture for Agents

In the public docs reviewed here, TrueFoundry is the only platform explicitly documenting both the gateway governance problem and the execution substrate problem within one architecture. The gateway problem — governing each LLM and tool call — is addressed by the stateless Hono gateway plane with its four-hook guardrail system. The execution problem — running the full agent loop without connection timeouts — is addressed by the async service runtime in the compute plane.

That separation matters operationally: gateway support for agent workloads is kept distinct from orchestration and runtime. The gateway plane proxies and governs LLM and MCP traffic, while orchestration state and long-running execution live in the control and compute planes rather than being fused into MCP itself.

The gateway plane's four-hook model, documented in the guardrails overview, is one of the most complete tool-call governance surfaces in this review. The four hooks are: LLM input (prompt injection, moderation, validation before any model sees the prompt), LLM output (hallucination checks, secrets detection, output filtering), MCP pre-tool (SQL injection checks, parameter validation, permission verification before tool execution), and MCP post-tool (secrets detection, PII redaction, result sanitization after tool execution). The docs explicitly call MCP hooks "critical for agentic workflows where a model autonomously invokes tools." This symmetric pre/post framing around tool execution is what differentiates TrueFoundry from gateways that do not explicitly document both MCP hook positions.

The gateway plane architecture is intentionally stateless: documented at the gateway plane architecture page as performing in-memory evaluation for rate limits, load balancing, authentication, and authorization on the hot path with no external calls unless cache is involved, and async queueing for logs and metrics. Multiple gateway plane pods can be deployed cross-region while configuration is managed centrally via a queue with HTTP fallback — making it horizontally scalable without coordination overhead.

The response headers x-tfy-resolved-model and x-tfy-applied-configurations, documented on the headers page, expose exactly which routing decisions and guardrail surfaces fired for each individual call in the loop. Combined with the server-timing header that breaks latency into authentication, guardrails, model call, and logging phases, this gives per-call observability depth that complements trace-level observability.

For the connection lifecycle problem, the platform's async service runtime — documented in the async service introduction and async service deployment pages — provides a first-class K8s-native runtime for queue-backed long-running work. Async services are designed precisely for application behavior that is fundamentally queue-backed and should not be forced through synchronous request-response semantics. Deploying an agent loop as a TrueFoundry async service decouples client connection lifecycle from execution lifecycle: the agent run proceeds behind a queue while the platform's Kubernetes substrate manages the worker pods. This is a structural solution to the timeout problem, not a timeout-limit increase.

⚠ Limitation — K8s Prerequisite & Adoption Barrier

The async service runtime and the self-hosted gateway plane require a running Kubernetes cluster, domain, TLS certificate, image-pull credentials, and a control-plane API key. For teams without a dedicated K8s platform engineering function, this is a meaningful adoption barrier compared to LiteLLM's single Python container or Portkey's SaaS edge. The deployment guide at deploy-gateway-plane makes these prerequisites explicit. The platform offers a hosted control plane and gateway-SaaS deployment mode that reduces this overhead, but full self-hosting is where the async service runtime advantage is most pronounced.

§7 — Editorial Verdict

Editorial assessments as of March 2026.

LiteLLM
The Budget-State Workhorse
LiteLLM's Redis-backed budget and identity state is highly practical for long agent loops: every LLM call inherits the same key, team, and budget context without a database round trip. Access groups for bundling models + MCP servers + agents into one governed resource envelope is the right abstraction. The gap is execution: there's no native mechanism for handling the loop's connection lifecycle.
Ideal for: Teams that already have an async job infrastructure and need per-call budget governance across multi-provider agent routing. Fast adoption path with Python + Redis.
Kong AI Gateway
The Protocol Normalizer
Kong's AI MCP Proxy is architecturally significant: it brings tool traffic into Kong's existing plugin ecosystem, so many of the same controls can apply without building a separate tool-governance plane. Streaming governance with response_streaming modes and response_buffer_size for guardrail inspection of chunks is well documented. Especially strong for orgs already operating Kong that want LLM-call and tool-call policy to live in one gateway stack.
Ideal for: Existing Kong users; high-throughput orgs that need MCP protocol normalization, semantic routing, and RAG injection alongside agent governance — all in one plugin layer.
Portkey
The Trace-First Observer
Portkey's W3C trace-context–aware trace trees for multi-step agent flows are the cleanest in this review for diagnosing loop failures. The composable fallback + circuit breaker + retry config handles per-call resilience well. The honest gap is the absence of a native async execution substrate — which requires application-side orchestration for long-running loops.
Ideal for: Product teams needing trace-first agent observability without K8s overhead. Best fit for LLM-heavy agents where observability and per-call resilience matter more than deep tool-call governance.
TrueFoundry
The Full-Stack Agent Platform
In the public docs reviewed here, TrueFoundry is the only platform explicitly documenting both gateway governance and execution lifecycle from one architecture. The four-hook guardrail system (LLM input, LLM output, MCP pre-tool, MCP post-tool) is especially complete for tool-calling agents. Just as importantly, the split-plane design keeps agent gateway support separate from orchestration: the gateway governs traffic, while async services handle the long-running loop. The gap is K8s operational prerequisite — a real barrier for teams without platform infra.

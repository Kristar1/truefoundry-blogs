What this blog argues

Deployment topology is not a feature dimension. It is the structural property that determines what regulatory regimes the AI platform can be operated under and how the platform behaves when external network paths fail. The two products optimize for different deployment shapes. Azure optimizes for breadth of regulated regions through sovereign clouds — Government, China-21Vianet, and the EU Data Boundary commitment are first-class. TrueFoundry optimizes for depth of customer-controlled installation — fully self-hosted with a documented air-gapped install path that requires no external runtime dependency. Both are legitimate architectures. The question for any specific workload is which regulatory shape and which network-availability assumption fits.

§0 — TL;DR & Quick-Picks

A regulated workload that becomes a deployment question

At a regional German hospital network, a radiology AI program is six weeks from go-live. The workload reads de-identified DICOM imaging, runs inference with a fine-tuned vision-language model, and produces structured findings that flow back into the radiology information system. Under DSGVO and the institution's interpretation of the BSI's KRITIS guidelines for healthcare-critical infrastructure, the data must remain in Germany, must not transit any cloud region outside the EU, and — per a stricter reading the institution's compliance team adopted after the 2024 KRITIS amendments — must operate without outbound internet egress on the inference path. The hospital's existing Kubernetes estate runs in two on-premises datacenters. The compliance team's verdict is unambiguous: the AI gateway must run inside our existing clusters, with no runtime dependency on any vendor-operated cloud.

The platform team brings two candidates to the architecture review. The first is Azure: APIM with a self-hosted gateway deployed into the hospital's K8s clusters, paired with Azure OpenAI in Sweden Central (within the EU Data Boundary), plus Azure AI Content Safety and Application Insights. The architects walk the team through the design carefully. The self-hosted gateway, they explain, is a Linux-based Docker container that runs inside the customer's K8s; it pulls configuration from the cloud APIM service every ten seconds and reports a heartbeat every minute; it is designed to fail static, continuing to serve traffic with the last-known-good configuration when connectivity to Azure is interrupted. The compliance team's question lands sharp: "so the gateway is in our cluster, but it still needs to call Azure?" The answer is yes — every ten seconds for config, every minute for health, plus telemetry to Azure Monitor if enabled. The architects offer the configuration-backup option, which lets a self-hosted gateway resume after a restart from a persistent volume containing the last-downloaded config — but the steady-state design assumes connectivity. For a workload that must operate without outbound internet on the data path, the answer is qualified at best.

The second candidate is TrueFoundry, fully self-hosted. The architects walk the same review: the control plane installs on the hospital's K8s, the gateway plane installs on the same clusters, NATS handles config sync inside the cluster, all four planes run inside the hospital's network boundary. The compliance team asks the analogous question: "and what does it need to call out to?" The answer is: in the air-gapped install path, nothing on the data path. The platform documents a forward-proxy pattern (a Squid configuration is included as the canonical example) for the package-pull and update channels, but the inference request path itself — gateway pod receiving a request, validating identity, applying guardrails, forwarding to the locally-deployed inference service, returning a response — has no outbound dependency. The compliance team's eyes narrow only briefly. The Azure architects, to their credit, do not push back. The deployment-topology axis decided the architecture review.

The hospital chooses TrueFoundry. The Azure proposal was technically credible; a self-hosted gateway with configuration backup is a real and supported deployment shape. But "fail static" is a degraded-mode property, not a steady-state operating posture. For a workload whose compliance interpretation requires no outbound traffic as the steady state, the architectures simply optimize for different things.

TL;DR
  • Azure deployment is region-anchored and cloud-federated. Sovereign clouds (Government, China-21Vianet) are first-class but distinct Azure clouds with separate Entra tenants and partial service catalogs. The EU Data Boundary commits to keeping certain customer data inside EU/EFTA but is a residency commitment, not a network-topology guarantee. The self-hosted gateway runs in customer K8s with documented "fail static" behavior — but pulls config from the cloud APIM service every 10 seconds and heartbeats every minute as the steady-state design.
  • TrueFoundry deployment is topology-flexible. Four documented paths: SaaS, customer cloud (split: control plane SaaS, gateway+compute on customer K8s), fully self-hosted (all four planes on customer infrastructure), and air-gapped install with documented forward-proxy patterns.
  • The decisive question is rarely "which regions are supported." It is "what is the smallest set of network egress and external dependencies required for the platform to run?" Different answers for different regulatory shapes.
  • Honest framing. The Azure self-hosted gateway is a credible solution for hybrid scenarios. The architectural fact that matters is its federation model with the cloud APIM service. For workloads that accept that federation, it works well; for workloads whose compliance interpretation rejects steady-state outbound traffic, it does not.
Quick-Pick
Azure
The right call when sovereign-cloud reach is a hard requirement (US Federal under FedRAMP High / DoD IL5, China onshore under 21Vianet), when the rest of the workload already runs on Azure, or when the EU Data Boundary commitment is a sufficient residency posture for the regulatory regime.
Quick-Pick
TrueFoundry
The right call when the deployment must be inside customer infrastructure with a documented dependency surface — VPC, on-prem K8s, or air-gapped — or when the org's compliance posture prefers physical control over the runtime to vendor-issued residency commitments. Especially decisive when forward-proxy egress patterns are part of the security architecture.

§1 — Why deployment is Foundations' third axis

Deployment topology determines what compliance regimes the platform can be operated under, what failure modes are routine versus exceptional, and how the platform behaves when external network paths are degraded. It is the foundations dimension that is hardest to refactor after the fact: changing region in Azure can require redeploying every dependent service; moving from SaaS to self-hosted in either platform requires re-establishing identity, secrets, and network paths. The right time to choose is before the first model is live.

The most common framing mistake is to compare deployment by counting regions. Azure has dozens; TrueFoundry runs on customer K8s anywhere. That checkmark comparison is correct but misses the operational point. The relevant question is what each platform requires from the network on the steady-state request path, what external services it depends on at runtime, and what the residency commitment actually means under realistic regulatory scrutiny. Those answers determine whether the platform clears compliance review for a given workload — not the count of available regions.

There are three sub-questions that compose the deployment axis, and most evaluations conflate them:

  • Where can the gateway and runtime physically execute? A region in a managed cloud, a customer-operated Kubernetes cluster, or both?
  • What external services does the runtime depend on at request time? A vendor-operated control plane reachable over the internet, a managed identity service in the same cloud, or only services inside the customer's own boundary?
  • What is the residency commitment, and is it a vendor commitment or a structural property? A vendor commitment ("we will not store this data outside region X") is a contract; a structural property ("the runtime is physically inside region X with no path out") is enforced by the architecture itself. Different regulatory regimes accept these differently.

The most consequential of the three is the second. A customer-K8s deployment that depends on vendor-cloud reachability for steady-state operation is structurally different from a customer-K8s deployment that runs without that dependency, even if both are described as "self-hosted" in marketing material. The Azure self-hosted gateway is the former; it is well-engineered and supports a documented "fail static" mode, but the steady-state design assumes connectivity to the cloud APIM service every 10 seconds for configuration updates and every minute for heartbeat reporting. TrueFoundry's fully self-hosted install is the latter; the control plane runs inside the customer's cluster, NATS handles config distribution inside the cluster, and the inference path has no external dependency.

The honest boundary on this argument

"Fail static" is a real and useful property. A self-hosted gateway that survives a 30-minute Azure connectivity blip without dropping requests is a credible engineering choice — and it is what most hybrid workloads actually need. The argument is not that Azure's self-hosted gateway is poorly designed; it is that federation with a cloud control plane is its steady-state design. For a workload whose compliance interpretation accepts that federation, the design is appropriate. For a workload whose interpretation requires no outbound traffic on the data path as the routine state, federation is the wrong shape regardless of how well it degrades.

The rest of the blog walks both architectures in working detail. §2 walks the Azure deployment surface — regions, sovereign clouds, the EU Data Boundary, and the self-hosted gateway's federation model — with the documented connectivity cadence and a working Helm composition for restricted-network deployment. §3 walks the TrueFoundry deployment surface — SaaS, customer-cloud-split, fully self-hosted, air-gapped — with a working install pattern that includes the forward-proxy configuration. §4 is the heart of the blog: five concrete deployment scenarios shown side by side, including the radiology workload from the opening, a US Federal classified workload, a multi-region SaaS firm, a Chinese onshore deployment, and a multi-cloud financial-services posture. §5 ends with the editorial verdict.

Microsoft§2 — Azure: regions, sovereign clouds, and the federated self-hosted gateway

Azure's deployment surface for AI is a layered set of options. Each layer has documented strengths and documented limitations. Understanding the composition requires walking each layer in turn — region selection, sovereign cloud reach, the EU Data Boundary commitment, the gateway flavors, and the federation model that ties self-hosted gateways back to the cloud APIM service.

Layer 1 — Azure regions and AI service availability

Azure is available in dozens of regions globally, with the standard option of choosing a region per resource. For AI workloads, the practical concern is that not every region has Azure OpenAI capacity, not every region has the same model availability, and quota allocation is region-specific. A team that wants gpt-4o at production capacity in Switzerland North needs to verify both the model availability for that region and the deployment-type tier (Standard, Provisioned Throughput, Global Standard, or Data Zone Standard) available there. AI Foundry's catalog is increasingly available in more regions but the specific GA status varies by feature. None of this is unusual for a managed cloud; the discipline it requires is part of the operating model.

For multi-region production workloads, APIM's multi-region deployment in the Premium tier replicates the gateway across regions with a shared configuration. Front Door or Application Gateway in front of multiple APIM instances handles geo-routing. The pattern is mature and widely deployed.

Layer 2 — The EU Data Boundary commitment

The EU Data Boundary is Microsoft's commitment that customer data and pseudonymized personal data for in-scope services remain within EU/EFTA. For GDPR programs that rely on Microsoft as a sub-processor, this is a meaningful contractual commitment with documented service scope. The boundary applies to a specific list of services (Azure core, Microsoft 365, Dynamics 365, Power Platform) with periodic updates as new services are added.

The shape of the commitment

The EU Data Boundary is a residency commitment, not a network-topology guarantee. Microsoft commits that the relevant data does not leave the boundary; the customer cannot, by inspecting the architecture from the outside, prove that no data ever crosses. For most regulatory regimes this is sufficient — the contractual commitment plus Microsoft's audit posture satisfies the auditor. For regimes that require structural proof rather than commitment, this is the layer where the difference shows up.

Layer 3 — Sovereign clouds: Azure Government, Azure China-21Vianet

For US federal workloads, Azure Government is a separate, isolated Azure cloud with FedRAMP High and DoD IL5 attestations. For China onshore workloads, Azure China-21Vianet is a distinct Azure cloud operated by 21Vianet under Chinese regulation. Both are mature, but both are separate Azure clouds with the consequences this implies:

  • Separate Entra tenants. A workload that spans a sovereign cloud and the commercial cloud cannot share an Entra tenant; identity must be federated explicitly.
  • Separate ARM endpoints. The management plane URLs differ; existing tooling that assumes management.azure.com needs configuration to point at the sovereign endpoint.
  • Partial service catalogs. Some Azure services are not yet in Azure Government; AI service availability lags the commercial cloud, sometimes by several quarters. The team must verify the specific AI services they need are available before committing.
  • Separate operational tooling. Azure portals, CLI configurations, and many third-party tools require sovereign-cloud awareness; a workload that spans both has two operational surfaces to maintain.

Layer 4 — APIM gateway flavors and the federation model

The APIM gateways comparison documents five flavors. For deployment-topology purposes, the two that matter are the workspace gateway (region-anchored, Azure-managed) and the self-hosted gateway (customer-managed, federated). Architecturally, the self-hosted gateway is the most consequential because it is the path teams reach for when they need to run gateway logic inside their own infrastructure.

The self-hosted gateway is a Linux-based Docker container that the customer deploys to Kubernetes (AKS, Azure Arc-enabled K8s, or any K8s). Each self-hosted gateway is associated with a Gateway resource in cloud APIM and federated with the cloud control plane. The connectivity cadence is documented:

  • Heartbeat every minute from the gateway to the cloud APIM management plane.
  • Configuration check every 10 seconds, applying any updates that have been published.
  • Telemetry to Azure Monitor / Application Insights if configured, with delivery semantics documented as best-effort (Application Insights uses an in-memory buffer without delivery guarantees per Microsoft's gateway comparison page).

When connectivity to Azure is lost, the gateway is designed to fail static: it cannot receive new configuration updates, report its status, or upload telemetry, but it continues serving traffic with the last known-good configuration. With the configuration-backup option enabled, the gateway also persists the last-downloaded configuration to a volume, so it can resume after a restart even without connectivity. This is genuinely useful — a 30-minute Azure connectivity blip does not cause a customer outage.

A specific architectural caveat — rate-limit synchronization

Per Microsoft's gateway comparison documentation: rate-limit counts in a self-hosted gateway can be synchronized locally among gateway instances inside one cluster (via Helm chart configuration), but they do not synchronize with other gateway resources configured in the same APIM instance, including the managed gateway in the cloud. For a workload that splits traffic across the cloud managed gateway and a self-hosted gateway in customer K8s, the rate-limit counters live in two separate worlds. This is not a defect — it is a documented architectural property — but it shapes how rate-limit policies must be designed in hybrid topologies.

What the operator writes — Helm install with restricted-network configuration

The shape below shows a self-hosted gateway deployed to a K8s cluster behind a corporate forward proxy, with configuration backup enabled and Azure Monitor disabled (the customer can choose to enable telemetry or to suppress it for residency reasons):

YAML
# Illustrative Helm values for self-hosted APIM gateway in restricted-network K8s
# Source: Microsoft Learn — self-hosted gateway settings reference + Helm chart docs
config:
  service:
    endpoint: "contoso-apim.configuration.azure-api.net"
  auth:
    # Token issued from cloud APIM, rotated periodically
    key: "GatewayKey contoso-prod-gw&xxxxxxxxxxxxxxxxxxxxxxxx"

# Forward proxy config — required when cluster has no direct internet egress
env:
  - name: HTTPS_PROXY
    value: "http://proxy.internal.contoso-hospital.de:3128"
  - name: NO_PROXY
    value: ".internal.contoso-hospital.de,localhost,.svc.cluster.local"

# Configuration backup — gateway persists last-downloaded config to PVC
configurationBackup:
  enabled: true
  storageClass: "hospital-rbd"
  size: 50Mi

# Telemetry disabled for residency reasons (no outbound to Azure Monitor)
telemetry:
  azureMonitor:
    enabled: false
  applicationInsights:
    enabled: false

# Local Syslog instead — telemetry stays inside the hospital network
extraEnv:
  - name: telemetry.logs.std
    value: "text"
  - name: telemetry.logs.local.localsyslog.endpoint
    value: "/dev/log"

resources:
  requests: { cpu: 500m, memory: 512Mi }
  limits:   { cpu: 2000m, memory: 2Gi }

replicaCount: 3
Illustrative composition based on the self-hosted gateway overview, settings reference, and Helm deployment guide. The cloud APIM endpoint must remain reachable for steady-state config sync; the configuration-backup volume preserves last-known-good config for restart after disconnection. Local syslog routing keeps telemetry inside the cluster.

Read this Helm composition as a federation contract. The gateway pod itself is in the hospital's cluster, but its operating model is "subscriber to cloud APIM." The forward-proxy entry, the configuration-backup volume, and the telemetry suppression are all operator choices to contain the federation — to limit what the federation costs in egress and what it leaks in telemetry — but the federation itself is structural. For the radiology workload from the opening scenario, this is the architectural fact that ended the Azure proposal: even a well-configured self-hosted gateway is a federated component, not a self-contained one.

When this is the right shape

For most hybrid AI workloads — a multi-cloud financial-services firm, a retail company that wants gateway logic close to backends in multiple regions, a SaaS company that needs to operate gateways inside customer VPCs — the self-hosted gateway's federation model is exactly right. The customer gets gateway execution close to the workload, with one cloud APIM service as the management surface. "Fail static" handles the routine connectivity blips. Configuration backup handles restart-after-disconnection. The architecture is genuinely well-engineered for the scenarios it targets. The compliance-shape question is what determines whether this is one of those scenarios.

TrueFoundry§3 — TrueFoundry: SaaS, customer cloud, fully self-hosted, air-gapped

TrueFoundry's deployment overview documents four installation paths arranged along an axis of customer control. Each path makes different trade-offs between time-to-value and the dependency surface on TrueFoundry's cloud. The differentiator that matters most for regulated workloads is the rightmost path on that axis: a fully self-hosted control plane with documented air-gapped operation.

The four paths

The paths share the four-plane architecture from Blog 1 — control, gateway, compute, data — and differ in which planes execute where:

  1. SaaS. All four planes operated by TrueFoundry. The customer creates an account, points clients at the SaaS endpoint, and starts deploying. This is the fastest path to first model call. Customer data flows through TrueFoundry's cloud; the platform is appropriate for non-regulated workloads, internal tools, and proof-of-concept work.
  2. Customer cloud (split). Control plane in TrueFoundry SaaS; gateway plane and compute plane installed on customer Kubernetes clusters. Application traffic — user prompts, model responses, tool calls — stays in customer infrastructure; only configuration metadata and aggregated telemetry flow back to the SaaS control plane. This is the most common enterprise install.
  3. Fully self-hosted. All four planes installed on customer infrastructure. The control plane runs in the customer's K8s, the gateway and compute planes alongside it (or on adjacent clusters), the data plane points at customer-operated databases. TrueFoundry provides install bundles, Helm charts, and operational runbooks; the customer operates the platform end-to-end, with TrueFoundry providing support and updates through documented channels.
  4. Air-gapped. Fully self-hosted with explicit forward-proxy configuration and a documented dependency surface for package pulls, image registry access, and update channels. The platform supports operation behind a corporate proxy with documented Squid configuration as a canonical example. The inference request path itself has no outbound dependency.

The architectural property that distinguishes "fully self-hosted" from "self-hosted gateway"

This is worth being precise about, because the marketing terminology overlaps and the architectural fact is what matters for compliance review. Both products advertise customer-Kubernetes deployment. The structural difference is which planes live inside the customer boundary:

  • In Azure's self-hosted gateway model, the gateway plane (one of the four planes from Blog 1's framing) runs in the customer cluster. The control plane — the management API that holds configuration, the workspace metadata, the policy definitions — remains in cloud APIM. The gateway federates with the cloud control plane via the documented connectivity cadence.
  • In TrueFoundry's fully self-hosted install, all four planes run inside the customer's infrastructure. The control plane is part of the install bundle. Configuration is held in a customer-operated database, distributed via NATS inside the cluster, and consumed by the gateway pods on the same cluster. There is no federation contract with TrueFoundry's cloud; the platform runs as a self-contained system.

The implication for the radiology scenario is direct: in TrueFoundry's air-gapped install, the inference path enters the gateway pod, validates JWT against cached SSO public keys (Blog 2), runs guardrails, forwards to a self-hosted vLLM service in the same cluster, and returns. No external service is called. The compliance team's "what does it need to call out to" question has a clean answer: nothing on the data path.

What the operator writes — air-gapped install with forward proxy

The shape below shows a fully self-hosted control-plane install configured for an air-gapped environment. The customer has pre-loaded TrueFoundry's container images into a private registry; the install uses an internal CNI; outbound HTTPS for the install itself goes through the corporate forward proxy, but the runtime data path does not:

Bash
# Illustrative TrueFoundry self-hosted install with forward proxy
# Source: TrueFoundry deployment-overview + forward-proxy documentation

# Step 1 — pre-stage images in private registry
# (This step is the air-gap-friendly pattern; in connected environments,
# the install pulls from TrueFoundry's public registry.)
for img in $(cat tfy-image-manifest.txt); do
  docker pull "$img"
  docker tag "$img" "registry.contoso-hospital.de/tfy/${img#*/}"
  docker push "registry.contoso-hospital.de/tfy/${img#*/}"
done

# Step 2 — install control plane
helm install truefoundry truefoundry/control-plane \
  --namespace truefoundry --create-namespace \
  --values values-airgapped.yaml \
  --set image.registry=registry.contoso-hospital.de/tfy \
  --set proxy.http=http://corp-proxy.contoso-hospital.de:3128 \
  --set proxy.https=http://corp-proxy.contoso-hospital.de:3128 \
  --set proxy.noProxy="cluster.local,.svc,127.0.0.1,10.0.0.0/8" \
  --set imagePullSecrets[0].name=tfy-private-registry \
  --set telemetry.outbound.enabled=false

# Step 3 — install gateway plane on the same or adjacent cluster
helm install truefoundry-gateway truefoundry/gateway \
  --namespace truefoundry-gateway --create-namespace \
  --values values-airgapped-gateway.yaml \
  --set controlPlane.endpoint=https://truefoundry.contoso-hospital.de \
  --set image.registry=registry.contoso-hospital.de/tfy

# Step 4 — verify no outbound from inference path
# (smoke test: request through gateway, observe network policies block
# any connection attempt to non-cluster-local destinations)
kubectl apply -f network-policies/inference-path-deny-egress.yaml
Illustrative install pattern based on the deployment overview, forward-proxy documentation, and install guide. The Squid forward-proxy pattern referenced in the docs is one tested example; any allowlisting proxy works. Network policies enforcing data-path egress denial are a customer responsibility; the platform documents which destinations the runtime needs (the answer is "intra-cluster only" for the air-gapped install).

The TrueFoundry deployment topology diagram

TrueFoundry deployment paths arranged from left to right by increasing customer control over the runtime. Path 4 (air-gapped) is the topology with no outbound dependency on the inference path.
What this earns operationally

(1) One install bundle, four paths. The same Helm chart can produce all four shapes with different values files; teams can start on path 1 (SaaS) and migrate to path 3 (fully self-hosted) as compliance requirements solidify. (2) Documented dependency surface. The air-gapped path has explicit documentation for what the install requires (forward proxy, private registry) versus what the runtime requires (intra-cluster only). (3) Same operating model across paths. Workspaces, virtual models, identity, and observability work the same way regardless of where the planes execute. The team that learns the platform on path 2 can operate path 4 without retraining.

§4 — Five deployment scenarios compared

The most useful way to compare these deployment models is to walk concrete regulatory and operational scenarios and identify what each platform requires to clear the workload. The five scenarios below are realistic patterns drawn from common compliance and architecture review postmortems. Each is shown side by side under both products. None is biased toward either platform by construction; the goal is to show what each architecture's path looks like for the workload shape.

SCENARIO 1EU healthcare under DSGVO + KRITIS, on-prem K8s, no outbound on inference path

The radiology workload from the opening scenario. Data must stay in Germany. The institution's compliance interpretation requires no outbound internet egress on the inference path as the steady state, not just under failure.

Azure path

Self-hosted gateway in customer EU K8s, paired with cloud APIM in EU North or West Europe. Steady-state outbound to Azure is required for config sync (every 10s) and heartbeat (every 1m). EU Data Boundary applies but is a residency commitment, not an air-gap. Configuration backup mitigates short outages but the steady-state design is federated. Azure Stack HCI is a more aggressive option but introduces significant operational complexity. Verdict: qualified. The Azure architecture is hybrid-cloud-friendly but not air-gap-friendly.

TrueFoundry path

Fully self-hosted or air-gapped install on customer EU K8s. No vendor cloud control plane required. Forward-proxy pattern documented for install and update channels; the inference data path itself is intra-cluster. Verdict: clean fit. The compliance team's "what does it call out to" question has a one-word answer: nothing.

SCENARIO 2US federal classified workload, FedRAMP High required

A federal agency needs to deploy an AI assistant for analyst workflows on classified data. FedRAMP High is the floor; DoD IL5 is the ceiling. The workload runs in an environment with strict audit and supply-chain attestations.

Azure path

Azure Government is the natural fit. FedRAMP High and DoD IL5 attestations are first-party. Azure OpenAI is available in Azure Government with a documented (and sometimes lagging) AI service catalog. The agency's existing Entra Government tenant is the identity surface. APIM, AOAI, AI Foundry, and the supporting services have established patterns inside the Government cloud. Verdict: this is exactly what Azure Government is designed for. Best-in-class fit.

TrueFoundry path

Fully self-hosted install on customer K8s estate that is itself FedRAMP-attested. The compliance attestations belong to the customer's environment, not to TrueFoundry as a vendor. Workable when the agency operates K8s with the appropriate boundary controls; less institutional than Azure Government's first-party FedRAMP story. Verdict: viable for agencies that operate their own attested K8s estate. For agencies that prefer vendor-attested cloud, Azure is the cleaner answer.

SCENARIO 3Multi-region SaaS firm with consolidated platform team

A B2B SaaS company runs production AI in three regions — US East, EU West, APAC — with one platform team and a desire for a single AI gateway operating model across regions. Standard residency requirements per region; no air-gap requirements.

Azure path

APIM Premium with multi-region deployment. One APIM service replicated across regions with shared configuration; Azure Front Door in front for geo-routing. Mature and widely deployed. The platform team operates one APIM instance with region-specific gateway pods. AOAI deployments per region with capacity allocation tuned to regional demand. Verdict: this is exactly what APIM Premium multi-region is designed for. Best-in-class fit.

TrueFoundry path

Customer-cloud-split install with gateway pods deployed to a K8s cluster per region. All gateway pods subscribe to the same NATS config stream from the control plane. The application's virtual model abstraction (Blog 4) is region-aware — the same virtual model name resolves to the regional model deployment. Verdict: clean fit. Both architectures handle this well; the operational model is different but the outcome is similar.

SCENARIO 4China onshore production for a multinational

A consumer electronics multinational operates AI features for Chinese customers and must keep that workload onshore in China under PIPL and data-localization requirements.

Azure path

Azure China-21Vianet is the natural fit. A distinct Azure cloud operated by 21Vianet under Chinese regulation. Mature service catalog (with the documented partial-availability for newer AI services). The team operates a separate Azure tenant in China-21Vianet alongside their commercial Azure tenant for non-China workloads. Verdict: best-in-class for China onshore; no other AI gateway vendor has a comparable first-party offering.

TrueFoundry path

Self-hosted on customer K8s in a Chinese cloud (Aliyun, Tencent Cloud, or Azure China). The platform itself runs anywhere; the regulatory relationship is between the customer and the underlying cloud provider in China. Workable but the cloud-provider relationship is the customer's problem to maintain. Verdict: viable but not differentiated. Azure's first-party China-21Vianet relationship is a meaningful advantage here.

SCENARIO 5Multi-cloud financial-services posture, regulator prefers customer-controlled runtime

A global investment bank runs workloads on AWS (primary), Azure (secondary), and GCP (specific data analytics workloads). Their regulator requires that critical AI inference run on infrastructure they directly control, with no vendor-cloud dependency. The platform team wants one operating model across all three clouds.

Azure path

Self-hosted gateway federated with cloud APIM in Azure. Works for the Azure portion of the workload. For AWS and GCP workloads, the self-hosted gateway can run on customer K8s in those clouds — but it still federates with the Azure-resident APIM control plane. The bank ends up with a cross-cloud dependency on Azure regardless of which cloud the workload actually runs on. Verdict: workable but produces an Azure-anchored operating model that contradicts the multi-cloud posture.

TrueFoundry path

Fully self-hosted install on customer K8s in each cloud. Same Helm chart, same operating model. Each cloud's workload has its own self-contained TrueFoundry control plane (or one centralized control plane in the bank's primary K8s estate, with gateway and compute planes per cloud). No cross-cloud vendor dependency. Verdict: this is the architecture's strongest fit pattern. The cloud-agnostic install is the value proposition.

§5 — Editorial verdict

Best fit
Azure
Sovereign-cloud reach is genuinely best-in-class. Azure Government's FedRAMP High and DoD IL5 attestations are first-party Microsoft commitments with no equivalent from any other AI gateway vendor. Azure China-21Vianet is the only mature path to AI workloads onshore in China for non-Chinese vendors. The EU Data Boundary commitment is a meaningful contractual posture for GDPR programs that accept vendor commitment as their compliance basis. The mature multi-region story through APIM Premium is real and widely deployed. For workloads whose regulatory regime accepts federated architectures, Azure's deployment surface is comprehensive.
Choose Azure if: sovereign cloud is a hard requirement (US federal, China onshore), the existing Azure footprint is large enough that adding AI workloads is a small marginal cost, the regulatory regime accepts vendor-commitment-based residency, or the workload is multi-region commercial cloud rather than air-gapped.
Best fit
TrueFoundry
Air-gapped install with documented forward-proxy patterns is the topology Azure does not have a clean answer for. Fully self-hosted control plane removes the vendor-cloud dependency from the steady-state architecture, not just from the failure-mode architecture. The platform runs on customer K8s anywhere — which means it inherits whatever regulatory attestations the customer's K8s estate already has, and operates uniformly across clouds without anchoring to one. For defense, classified, KRITIS, multi-cloud financial-services, and strict-interpretation healthcare workloads, this structural property is decisive in a way that "fail static" connectivity behavior is not.
Choose TrueFoundry if: air-gapped operation is required as steady state (not just as a failure mode), the org's compliance posture prefers structural control over the runtime to vendor-issued residency commitments, the workload spans multiple clouds and the platform must run consistently across them, or the regulatory interpretation forbids steady-state outbound dependencies on the data path.

The most honest framing is that these architectures optimize for different regulatory shapes. Azure optimizes for the breadth of sovereign-cloud regulatory regimes — US federal, China onshore, EU residency commitments — where the customer accepts a vendor-managed control plane in exchange for first-party attestations and operational simplicity. TrueFoundry optimizes for "the platform runs inside our boundary, on our infrastructure, with the network paths we control" — where the customer prefers structural control even if it means the platform inherits regulatory attestations from the customer's own estate rather than from the vendor.

The question for any specific workload is not which architecture is universally better. It is whether the regulatory regime accepts a federated architecture (Azure's strength) or requires a self-contained one (TrueFoundry's strength). For most enterprise AI workloads, either answer works. For the workloads at the edges — sovereign clouds at one edge, air-gapped install at the other — the architectures are differentiated cleanly, and the editorial verdict is each platform wins the edge it's designed for.
